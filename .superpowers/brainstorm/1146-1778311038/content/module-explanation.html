模块化单体 — 模块划分解读

一个 Git 仓库，每个模块是一个独立编译单元（Maven/Gradle submodule），之间通过接口调用，不是 HTTP

项目根目录 / 子模块结构
chengjialiye/    ← 一个 Git 仓库
├── common/  ← 公共模块（被所有人依赖）
实体类(User, Order)、工具类、异常定义、
Redis配置、统一的 API 返回格式
├── admin-api/  ← 管理后台接口
管理员登录、用户管理、审核、财务对账、
量表配置、活动运营、推广管理...
├── miniapp-api/  ← 小程序接口
觅缘推荐、心印测评、精选/理想型、VIP 购买、
消息/通知、社区动态...
└── admin-web/  ← 管理后台前端（Vue 3）

调用关系

admin-api
管理后台后端
→
common
实体/工具/配置
←
miniapp-api
小程序后端

两个 API 模块都依赖 common，但 admin-api 和 miniapp-api 互相独立
同一套 User/Order 实体，不用写两遍

一句话概括

common = 大家都要用的东西放这里（实体类、工具类、公共配置）

admin-api = 运营人员用的后台系统的接口

miniapp-api = 用户在小程序上看到的所有功能的接口

admin-web = 前端页面（Vue 项目，调 admin-api 的接口）
