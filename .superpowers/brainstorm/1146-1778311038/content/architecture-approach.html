后端整体架构选型

第一步：先定后端架构范式，这决定了目录结构、部署方式、团队协作模式

A

单仓库单体应用

一个后端项目，按模块分 controller/service/mapper。管理后台和小程序共享同一套 API 服务层。

Pros

  • 开发最快，一个 IDE 搞定
  • 共享鉴权、ORM、工具类
  • 部署简单，一个 jar/容器
  • 小团队（3-5人）最高效

Cons

  • 后期模块耦合风险
  • 全部模块一起编译部署
  • 团队扩大后易冲突
B

单仓库模块化单体

一个 Git 仓库，Maven/Gradle 多模块。拆分为 common / admin-api / miniapp-api / service 等子模块。

Pros

  • 模块边界清晰，强制分层
  • 可独立编译子模块
  • Common 层复用，避免代码重复
  • 后期可拆分微服务

Cons

  • 初期搭建多一步模块配置
  • 模块间接口需提前约定
C

微服务（多仓库/多模块）

独立部署：用户服务、认证服务、测评服务、财务服务、运营服务。通过 Gateway + RPC/消息队列通信。

Pros

  • 独立部署、独立扩容
  • 团队可并行开发不同服务
  • 技术栈可异构

Cons

  • 运维成本高（DevOps/SRE）
  • 分布式事务、服务发现复杂
  • 初期 ROI 极低
  • 小团队根本忙不过来

推荐

B — 单仓库模块化单体，理由：

  • 模块边界预埋好（admin-api / miniapp-api / common），未来拆微服务零重构
  • 团队开发时各自改自己的模块，不会互相踩脚
  • Common 层抽取后，两个 API 模块体量都很轻
