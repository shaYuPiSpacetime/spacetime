# 成家立业管理后台细化 PRD-08

## 1. 文档信息

- 文档主题：推荐算法配置后台细化需求
- 对应移动端文档：`细化PRD-08_推荐算法（觅缘匹配与精选）`
- 对齐基准文档：`成家立业 · 管理后台产品需求文档 V1.0`
- 使用目标：在不突破现有报价文档模块边界的前提下，细化推荐算法模块对后台的落地要求

## 2. 对齐原则

本文件继续以总体后台需求文档为基准，不重新定义一级模块，不默认新增独立后台系统。仅当移动端 PRD-08 已确认需求无法自然落入既有后台模块时，才补充 `必要细化` 的二级能力。

### 2.1 本文档与总体后台文档的映射关系


| 本文档内容                  | 对齐总体后台文档模块                  | 说明                   |
| ---------------------- | --------------------------- | -------------------- |
| 推荐权重、阈值、规则配置           | `3.6.2 参数配置`                | 直接细化                 |
| 婚姻状态、感情状态、脱单目标枚举       | `3.6.3 字典与枚举管理`             | 直接细化（部分已在 PRD-01 定义） |
| 行为标签配置（使用时长/滑动模式/时段标签） | `3.6.3 字典与枚举管理`             | 直接细化                 |

| 推荐效果监控                 | 复用小程序的统计模块                  | 首版不做独立看板，保留影响评估与二期入口         |

### 2.2 本期不单独立项的能力

1. 独立 `推荐策略配置中心`（作为一级菜单）
2. 独立 `算法实验室 / AB 测试平台`

#### 3.1.3 配置生效方式

后台需在每个配置项旁标注"立即生效"或"待重训"，并在保存后给出对应提示。

| 生效方式 | 配置范围 | 说明 |
| --- | --- | --- |
| 立即生效 | 匹配分数、地理位置分、年龄分、感情状态配对分、活跃度权重、去重窗口、精排返回数量、精选资料完整度阈值、CB2/性格来源权重、双向分合成方式 | 保存后影响后续实时打分或推荐接口，不要求重新训练 CatBoost |
| 待重训 | 行为标签阈值、行为矩阵权重、ALS 隐因子维度、KMeans 聚类数、候选召回数、负采样比例、scale_pos_weight、正样本标签阈值、Optuna 主目标、Optuna trials 数 | 保存后标记"待重训"，下次离线训练时生效；线上模型切换前不改变当前模型 |

#### 3.3.2 SHAP 特征重要度查看

本期不做 SHAP 可视化入口，仅要求离线训练管道保留 SHAP / CatBoost 特征重要度数据。后续如启用，在用户详情页提供"推荐特征解释"入口（运营/管理员可见）：

- 展示该用户最近一次推荐中，各个特征的平均 SHAP 贡献值排名。
- 首版增强建议优先用表格展示（特征名 + SHAP 均值 + 排名），横向柱状图作为二期增强。
- 数据来源：离线 SHAP 计算结果，非实时。

本期不做的影响评估：

1. 不影响推荐排序、模型训练和线上推理。
2. 运营暂时无法在后台可视化查看推荐理由，只能通过离线训练报告或日志排查。
3. 若上线后需要面向运营解释推荐效果，应优先补表格版特征重要度查看。

### 3.4 推荐效果监控

本期不强制新增推荐效果监控卡片。若复用现有统计/看板模块，建议新增以下推荐相关指标：


| 指标     | 计算方式                  | 刷新频率 |
| ------ | --------------------- | ---- |
| 日均推荐次数 | 推荐接口调用次数 / 天数         | 每小时  |
| 右滑率    | 右滑次数 / 推荐展示次数         | 每小时  |
| 匹配转化率  | 相互匹配次数 / 右滑次数         | 每小时  |
| 消息转化率  | 发消息匹配数 / 相互匹配数        | 每小时  |
| 约会转化率  | 持续聊天（≥ 3 轮）数 / 发消息匹配数 | 每小时  |


告警规则：

1. 右滑率突降 50%（较前 7 日均值）→ 运营人员通知。
2. 匹配转化率持续 3 天低于 5% → 算法排查 + 模型重训练。

本期不做的影响评估：

1. 不影响推荐功能上线，但无法通过后台实时观察推荐质量。
2. 推荐质量异常需要依赖离线周报、接口日志或人工数据查询发现。
3. 二期建议在统计模块新增"推荐算法"Tab，并将告警接收人限定为管理员和运营人员。

### 3.5 操作日志

对齐 `3.1.5 审计与安全`：

1. 推荐参数修改（3.1 中各配置项）→ 记录修改前后的值、操作人、操作时间。
2. 权重修改 → 记录变更前后各权重值。
3. 字典枚举变更 → 记录新增/编辑/启停操作。
4. 模型版本切换 → 记录切换前后的模型文件 hash、版本号。
5. 需重新训练配置项变更 → 记录配置项、变更值，并标记"待重训"。
6. 操作日志保留 90 天，支持按操作人 + 时间段筛选，与现有审计日志模块保持一致。
