# 成家立业管理后台细化 PRD-03

## 1. 文档信息

- 文档主题：消息、私信与通知中心后台细化需求
- 对应移动端文档：`细化PRD-03_消息、私信与通知中心`
- 对齐基准文档：`成家立业 · 管理后台产品需求文档 V1.0`
- 使用目标：在不突破现有报价文档模块边界的前提下，细化移动端 PRD-03 对后台的落地要求

## 2. 对齐原则

本文件继续以总体后台需求文档为基准，不重新定义一级模块，不默认新增独立后台系统。仅当移动端 PRD-03 已确认需求无法自然落入既有后台模块时，才补充 `必要细化` 的二级能力；其余可增强但非本期必需的能力，统一归为 `后续增值服务建议`。

### 2.2 本期不单独立项的能力

以下能力不建议在本期作为独立模块写入报价交付范围：

1. 独立 `IM 会话运营工作台`
2. 独立 `消息风控中心`
3. 独立 `通知投放系统`
4. 独立 `聊天质检中心`
5. 独立 `消息 BI 看板`

说明：

1. 以上能力如确有需要，可先通过现有页面的详情 Tab、筛选项、模板配置、日志抽屉承接。
2. 若甲方后续希望做成独立菜单或专题系统，建议作为后续增值服务二期扩展。

## 3. 本期后台范围定义

PRD-03 对后台的本期范围，控制在以下 6 类能力：

1. 用户私信、悄悄话、通知记录的查询与人工排查
2. 与聊天相关的举报、拉黑、失效状态联动
3. 站内通知模板、通知详情文案、保护提示文案配置
4. 悄悄话、普通私信、女性保护机制、红点规则等核心参数配置
5. 悄悄话消耗、VIP 免费次数等商业化规则承接
6. 权限、日志、脱敏等基础安全能力对上述功能的支撑

## 4. 模块细化方案

## 4.1 用户管理中心

对应总体后台文档：`3.2 用户管理中心`

本期不新增一级菜单，仍挂靠在 `用户管理中心` 下完成。

### 4.1.1 用户列表

在总体文档既有字段基础上，补充 PRD-03 必需字段，方便运营、客服、审核人员快速定位消息与通知状态。

#### 搜索条件

1. 用户 ID

5. 注册时间范围
6. 用户状态
7. 三项认证状态
8. 关系反馈准入状态
9. 是否存在私信会话
10. 是否存在悄悄话待回复
11. 是否命中女性保护机制
12. VIP 状态

#### 列表展示

1. 头像
2. 昵称
3. 手机号脱敏
4. 学校
5. 三项认证状态
6. 当前有效私信会话数
7. 悄悄话待回复数

#### 操作项

### 4.1.2 用户详情

沿用总体文档 `用户详情` 能力，但为 PRD-03 补充以下展示区块与详情 Tab。

#### A. 消息互动摘要

1. 是否完成 `实名认证 + 头像认证 + 学历认证`
2. 是否具备真实聊天资格
3. 当前私信会话数
4. 当前悄悄话待回复数
5. 最近一次匹配成功时间
6. 是否命中女性保护机制
7. 当前未读通知数
8. 最近一次聊天异常/拉黑时间

#### B. 私信会话 Tab

1. 会话 ID
2. 对话双方用户 ID
3. 会话状态
4. 最后消息摘要
5. 最后消息时间
6. 是否可发送
7. 失效原因

#### C. 悄悄话记录 Tab

1. 发送方用户 ID
2. 接收方用户 ID
3. 发送时间
4. 当前状态：
   - 待回复
   - 已回复
   - 已忽略
   - 已失效
5. 是否命中 7 天冷却期
6. 消耗方式：
   - VIP 免费次数

#### D. 通知记录 Tab

1. 通知 ID
2. 通知类型
3. 业务类型
4. 标题
5. 已读状态
6. 生成时间

#### E. 黑名单与举报记录 Tab

1. 拉黑关系
2. 举报时间
3. 举报原因
4. 处理状态
5. 处罚结果
6. 对会话的影响结果

说明：

1. `私信会话 Tab`、`悄悄话记录 Tab`、`通知记录 Tab` 属于本期 `必要细化`，建议作为用户详情的二级 Tab 承接。
2. 本期默认以 `查询、排查、状态核对` 为主，不建议开放手工伪造消息内容。

### 4.1.3 用户生命周期管理

对齐总体后台文档：`3.2.3 用户生命周期管理`

本期重点细化与消息链路直接相关的状态联动：

1. 任一方冻结、停用、注销中、已注销后：
   相关会话状态改为 `invalid`，不可继续发送。
2. 任一方被拉黑后：
   保留历史聊天记录，但会话改为失效态，不再允许新消息发送。
3. 用户三项认证失效或被撤销后：
   不再展示用户私信会话入口，不可发送悄悄话与普通私信。
4. 通知信息一直保留，私信记录长期保留。

### 4.1.4 客服操作

对齐总体后台文档：`3.2.4 客服操作`

PRD-03 阶段建议客服/运营侧仅保留必要能力：

1. 清除异常登录态
2. 冻结/解冻账号
3. 查看私信会话与悄悄话状态
4. 查看女性保护机制命中状态
5. 查看通知发送与已读状态
6. 处理举报与申诉

说明：

1. 本期不建议开放 `后台替用户发消息`、`后台手工解除女性保护`、`后台手工补发悄悄话` 这类高风险动作。
2. 如确需人工修正，建议仅向超级管理员开放，并强制记录操作日志与原因。

### 4.1.5 必要新增说明

总体后台文档已经具备 `用户详情`、`行为摘要`、`客服操作` 等基础能力，但未把 `私信会话`、`悄悄话记录`、`通知记录` 的后台查看方式拆开。本文件将其细化为 `用户详情下的消息相关 Tab`，属于 `必要细化`，不建议包装成新的独立一级模块。

## 4.2 运营中心

对应总体后台文档：`3.5 运营中心`

### 4.2.1 内容审核与处罚联动

对齐总体后台文档：`3.5.3 内容审核`

本期不新增独立 `聊天举报中心`，仍挂靠在现有 `内容审核` 能力下处理：

1. 用户在聊天页发起举报后，运营可按既有处罚规则执行：
   - 警告
   - 禁言 24 小时
   - 封号 7 天
   - 永久封禁
2. 处罚生效后，需联动会话状态：
   - 会话进入失效态
   - 不可继续发送消息
   - 保留历史聊天记录供排查
3. 举报处理结果需生成对应站内通知。

说明：

1. 本期重点是承接举报结果与处罚结果，不展开完整消息内容审核系统。
2. 若后续需要做 AI 风控、敏感词实时拦截、会话质检，可作为增值服务。

### 4.2.2 文案与消息中心

对齐总体后台文档：`3.5.4 文案与消息中心`

本期补充以下模板与文案配置：

1. 官方消息模板
2. 匹配成功通知模板
3. 收到悄悄话通知模板
4. 三项认证结果通知模板
5. 举报处理结果通知模板
6. 违规警告通知模板
7. 充值成功 / 成家币变动 / VIP 到期提醒模板
8. 女性保护机制提示文案
9. 悄悄话忽略后轻提示文案
10. 未完成三项认证的消息页引导文案

明确要求：

1. 举报处理结果统一跳转 `通知详情页`，不单独新开发举报结果页面。
2. 关闭某类通知开关，只影响红点与提醒，不影响站内消息落库。

### 4.2.3 匹配运营策略

对齐总体后台文档：`3.5.5 匹配运营策略`

本期只承接与 `消息开口条件`、`女性保护机制`、`悄悄话匹配成功` 有关的轻量细化，不展开推荐算法：

1. 普通私信仅在 `匹配成功后` 开放
2. 悄悄话回复可触发匹配成功
3. 女性保护期从 `匹配成功时间` 开始连续计算 3 天

说明：

1. 本期不把 `互关私信`、`任意人私信` 作为并行开口规则保留。
2. 设置中的“谁可以给我发私信”旧口径，需在后台文案和配置说明中同步替换。

## 4.3 系统管理中心

对应总体后台文档：`3.6 系统管理中心`

### 4.3.1 参数配置

对齐总体后台文档：`3.6.2 参数配置`

本期需补充并固化以下消息与通知核心参数：

#### A. 消息开放规则

1. `消息 Tab` 对所有已登录用户可见
2. 未完成三项认证用户仅可查看官方消息与通知中心
3. 普通私信仅在匹配成功后开放
4. 悄悄话发送需三项认证全部通过

#### B. 悄悄话规则

1. 普通用户可直接使用成家币发送
2. VIP 每日免费悄悄话次数：
   首版默认为 `1`
3. 免费次数用完后，VIP 仍可继续使用成家币发送
4. 同一对象在上一条悄悄话未被回复或未失效前，不允许重复发送
5. 对方忽略后，发送方 `7 天内` 不可再次向同一对象发送

#### C. 女性保护机制

1. 保护期开关
2. 保护期时长：
   首版默认为 `3 天`
3. 保护期起算时间：
   从匹配成功时间开始连续计算
4. 解除男性禁发条件：
   仅限女性发送真实用户消息
5. 系统提示消息不计入解除条件

#### D. 通知与红点规则

1. 首版纳入的通知类型范围
2. 通知开关关闭后是否计入红点：
   首版确认为 `不计入`
3. 通知信息保留策略：
   一直保留
4. 私信记录保留策略：
   长期保留

### 4.3.2 字典与枚举管理

对齐总体后台文档：`3.6.3 字典与枚举管理`

本期需要承接以下字典与枚举：

1. 会话状态枚举
2. 悄悄话状态枚举
3. 通知类型与业务类型枚举
4. 举报原因字典
5. 女性保护机制状态枚举

说明：

1. 字典体系尽量复用总体后台文档已有结构，不建议另起一套独立消息枚举中心。

## 4.4 移动端配置管理

对应总体后台文档：`3.7 移动端配置管理`

本期仍以此模块作为消息、私信、通知规则的主要可配置承接区。

### 4.4.1 商业化配置

对齐总体后台文档：`3.7.2 商业化配置`

本期需明确以下规则：

1. 悄悄话可走成家币发送
2. VIP 每日赠送 `1` 次免费悄悄话
3. VIP 免费次数用完后，仍可按成家币发送
4. 成家币单次消耗价格继续在后台可配

说明：

1. 本期不额外新增 `消息专属套餐中心`。
2. 若后续需要按人群、活动期配置差异化悄悄话价格，建议作为增值服务。

### 4.4.2 社交权限与配额配置

本期需承接的核心配置包括：

1. 普通私信开口条件
2. 悄悄话发送开关

对齐总体后台文档：`3.7.5 其他移动端配置`

## 5. 权限、日志与脱敏要求

对应总体后台文档：`3.1 权限与账号体系`

本期继续沿用总体后台既有要求，并对消息链路补充如下约束：

1. 只有具备对应数据权限的角色，才能查看用户私信会话摘要、悄悄话记录、通知记录。
2. 私信内容查看属于高敏操作，应默认仅允许审核/超级管理员角色访问。
3. 手机号、实名信息等敏感字段继续按总体文档要求脱敏展示。
4. 查看会话详情、改动消息规则配置、处理举报结果，均需记录操作日志。
5. 高风险操作如手工失效会话、恢复会话，应仅向超级管理员开放，并要求二次确认。

## 6. 本期不单独立项但可承接的轻量能力

以下能力如甲方希望首版可见，建议只做轻量实现，不单独起系统：

1. 在用户详情页增加 `私信会话`、`悄悄话记录`、`通知记录` 二级 Tab
2. 在配置页增加 `女性保护机制` 配置组
3. 在举报处理结果中增加 `会话已失效` 标签
4. 在文案中心增加 `通知详情页` 模板

## 7. 后续增值服务建议

以下能力不建议纳入当前报价必交范围，如甲方后续需要可作为增值服务：

1. 独立 IM 会话运营工作台
2. 聊天内容 AI 风控与实时拦截
3. 消息质检与客服巡检专题后台
4. 独立通知投放与触达分析平台
5. 消息链路 BI 看板与漏斗分析

## 8. 结论

管理后台 PRD-03 仍以总体后台报价文档为基准，不新增一级模块。移动端 `消息、私信与通知中心` 对后台的要求，首版主要落在以下既有模块中：

1. `用户管理中心`：承接私信会话、悄悄话记录、通知记录和黑名单/举报排查
2. `运营中心`：承接举报处罚联动、通知模板、保护提示文案
3. `系统管理中心`：承接消息开放条件、悄悄话规则、女性保护机制、红点策略等核心参数
4. `移动端配置管理`：承接悄悄话消耗、VIP 免费次数、冷却期、页面轻配置

除上述必要细化外，其余能力统一收敛为后续增值服务，确保本期范围与甲方报价口径一致。
