# 成家立业管理后台细化 PRD-02

## 1. 文档信息

- 文档主题：关系反馈与互动链路后台细化需求
- 对应移动端文档：`细化PRD-02_关系反馈与互动链路`
- 对齐基准文档：`成家立业 · 管理后台产品需求文档 V1.0`
- 使用目标：在不突破现有报价文档模块边界的前提下，细化移动端 PRD-02 对后台的落地要求

## 2. 对齐原则

本文件继续以总体后台需求文档为基准，不重新定义一级模块，不默认新增独立后台系统。仅当移动端 PRD-02 已确认需求无法自然落入既有后台模块时，才补充 `必要细化` 的二级能力；其余可增强但非本期必需的能力，统一归为 `后续增值服务建议`。

### 2.2 本期不单独立项的能力

以下能力不建议在本期作为独立模块写入报价交付范围：

1. 独立 `关系反馈工作台`
2. 独立 `访客/匹配异常任务中心`
3. 独立 `女性保护策略中心`
4. 独立 `匹配链路 BI 看板`
5. 独立 `聊天风控中心`

说明：

1. 以上能力如确有需要，可先通过现有页面的筛选条件、详情 Tab、配置项、日志抽屉承接。
2. 若甲方后续希望做成独立菜单或专题系统，建议作为后续增值服务二期扩展。

## 3. 本期后台范围定义

PRD-02 对后台的本期范围，控制在以下 6 类能力：

1. 用户关系反馈记录的查询与人工排查
2. 喜欢记录、访客记录、匹配成功记录的后台承接
3. 与关系反馈有关的处罚、失效、申诉支撑
4. 解锁规则、VIP 权益、访客与喜欢我的展示规则配置
5. 匹配成功弹窗、保护提示、空态与引导文案配置
6. 权限、日志、脱敏等基础安全能力对上述功能的支撑

### 4.1.3 用户生命周期管理

对齐总体后台文档：`3.2.3 用户生命周期管理`

本期重点细化与关系反馈链路直接相关的状态联动：

1. 任一方冻结、停用、注销中、已注销后：
   与其相关的 `喜欢我的`、`最近看过我的`、`相互喜欢` 记录按业务规则失效或不再展示。
2. 任一方被拉黑或封禁后：
   相互喜欢记录立即失效，后续不可继续进入聊天链路。
3. 用户撤回喜欢后：
   `喜欢我的` 当前有效记录同步失效；若业务支持撤回后回收匹配关系，则 `相互喜欢` 也同步失效。
4. 最近看过我的：
   底层可保留长期记录，但移动端仅查询最近 7 天内的可展示访客。

### 4.1.4 客服操作

对齐总体后台文档：`3.2.4 客服操作`

PRD-02 阶段建议客服/运营侧仅保留必要能力：

1. 清除异常登录态
2. 冻结/解冻账号
3. 查看关系反馈记录明细
4. 查看匹配成功来源
5. 查看单条解锁与 VIP 解锁记录
6. 处理与骚扰、虚假资料相关的申诉

说明：

1. 本期不建议开放 `后台手工制造匹配成功`、`后台手工补喜欢记录` 这类高风险动作。
2. 如确需人工修正，建议仅向超级管理员开放，并强制记录操作日志与原因。

### 4.1.5 必要新增说明

总体后台文档已经具备 `用户详情`、`行为摘要`、`客服操作` 等基础能力，但未把 `喜欢记录`、`访客记录`、`匹配成功记录` 的后台查看方式拆开。本文件将其细化为 `用户详情下的关系记录 Tab`，属于 `必要细化`，不建议包装成新的独立一级模块。

## 4.2 运营中心

对应总体后台文档：`3.5 运营中心`

### 4.2.1 内容审核与处罚联动

对齐总体后台文档：`3.5.3 内容审核`

本期不新增独立 `关系举报中心`，仍挂靠在现有 `内容审核` 能力下处理：

1. 用户因 `骚扰`、`虚假资料`、`不当私信` 被举报后，运营可按既有处罚规则执行：
   - 警告
   - 禁言 24 小时
   - 封号 7 天
   - 永久封禁
2. 处罚生效后，需联动关系记录展示状态：
   - 不再继续向对方展示可互动入口
   - 已形成的相互喜欢可失效
3. 若举报核实成立，需保留处罚记录与处理人信息。

说明：

1. 本期重点是承接处罚结果，不展开完整消息内容审核系统。
2. 若后续需要把 `骚扰举报`、`女性保护会话拦截` 做成专题后台，可作为增值服务。

### 4.2.2 文案与消息中心

对齐总体后台文档：`3.5.4 文案与消息中心`

本期补充以下文案模板配置：

1. 匹配成功弹窗标题、副标题、按钮文案
2. 喜欢我的单条解锁弹窗文案
3. 最近看过我的单条解锁弹窗文案
4. VIP 引导弹窗文案
5. 成家币充值引导文案
6. 关系反馈链路空态文案
7. 女性保护机制提示文案
8. 处罚/失效结果的站内通知文案

明确要求：

1. 匹配成功弹窗按已确认规则：`当次喜欢操作触发双向喜欢时立即弹出，且同一对用户仅首次匹配成功时弹一次`。
2. 女性保护机制相关提示，需支持在聊天前置页面或提示弹层中动态配置。

### 4.2.3 匹配运营策略

对齐总体后台文档：`3.5.5 匹配运营策略`

本期只承接与 `匹配成功来源` 有关的轻量细化，不展开推荐算法：

1. 匹配成功来源枚举展示：
   - 双方互送爱心
   - 精选页送出 `对 ta 心动` 后对方回送爱心
   - 悄悄话后对方回复
2. 运营可在后台查看每条匹配成功记录的来源类型，用于客服排查和活动复盘。

说明：

1. 推荐优先级、加权逻辑、人群包策略仍以总体后台文档为准。
2. 本期不将 `推荐算法配置` 写入 PRD-02 范围。

## 4.3 系统管理中心

对应总体后台文档：`3.6 系统管理中心`

### 4.3.1 参数配置

对齐总体后台文档：`3.6.2 参数配置`

本期需补充并固化以下关系反馈核心参数：

#### A. 准入与展示规则

1. 关系反馈链路准入条件：
   `实名认证 + 头像认证 + 学历认证` 三项全部通过后才可进入。
2. 关系反馈链路展示对象：
   仅展示异性用户。
3. 普通用户 `喜欢我的` 模糊条数上限
4. 普通用户 `最近看过我的` 模糊展示策略
5. 是否允许开启 `隐藏我的访问记录`

#### B. 访客规则

1. 访客记录保留策略：
   底层记录可长期保留，但前台只查最近 7 天展示数据。
2. 访客去重窗口：
   同一访客 30 分钟内连续进入同一主页，仅记 1 次访客记录。
3. 统计口径：
   - 今日访客 = UV
   - 今日浏览量 = PV
   - 总浏览量 = 累计 PV
4. 来访生成规则：
   凡进入真实用户主页且未命中隐藏规则的行为，都计入来访。

#### C. 喜欢与匹配规则

1. 喜欢记录仅保留当前有效喜欢关系
2. 相互喜欢记录在未失效前长期保留
3. 撤回喜欢、拉黑、封禁后相关记录的失效规则
4. 匹配成功弹窗触发规则
5. 匹配成功后主页按钮从 `Yo` 变更为 `聊天 icon`
6. 匹配成功后底部主按钮变更为 `聊天 icon + 聊天`

#### D. 女性保护机制

1. 保护期天数：
   首版默认 3 天。
2. 生效范围：
   仅针对匹配成功后的私聊阶段。
3. 核心规则：
   保护期内，对女性采取保护机制，只有女性回复后，男性才可以继续回复。

说明：

1. 若后续消息 PRD 对发送顺序再做细化，应以消息模块细化 PRD 为准补充接口级规则。
2. 本期后台先确保参数可配置、可查看、可留痕。

### 4.3.2 字典与枚举管理

对齐总体后台文档：`3.6.3 字典与枚举管理`

本期需要承接以下字典与枚举：

1. 匹配成功来源枚举
2. 喜欢/访客来源场景枚举
3. 举报原因字典
4. 理想型标签、恋爱观标签、运动标签等既有标签字典

## 4.4 移动端配置管理

对应总体后台文档：`3.7 移动端配置管理`

本期仍以此模块作为关系反馈链路最主要的可配置承接区。

### 4.4.1 商业化配置

对齐总体后台文档：`3.7.2 商业化配置`

本期需明确以下规则：

1. `喜欢我的` 单条解锁走成家币
2. `最近看过我的` 单条解锁走成家币
3. `解锁全部` 首版统一走 VIP，不支持成家币解锁全部
4. VIP 生效期间可查看全部清晰记录
5. VIP 到期后回退普通态，但已单条购买的记录永久可见
6. `查看喜欢我的高清头像`、`查看所有访客` 继续作为 VIP 权益项保留
7. 成家币单次消耗价格继续在后台可配

说明：

1. 本期不额外新增 `关系反馈专属套餐中心`。
2. 若后续需要按人群、活动期配置差异化价格，建议作为增值服务。

### 4.4.2 社交权限与配额配置

对齐总体后台文档：`3.7.3 社交权限与配额配置`

本期需承接的核心配置包括：

1. 三项认证通过后开放关系反馈链路
2. `喜欢我的` 免费可见模糊条数上限
3. `最近看过我的` 的访客展示天数
4. 访客单条解锁所需成家币
5. 喜欢我的单条解锁所需成家币
6. 隐藏访问记录权益开关
7. 女性保护机制开关
8. 女性保护期天数
9. 悄悄话回复是否可触发匹配成功

说明：

1. `理想型搜索后默认展示 5 个高斯模糊用户` 不属于关系反馈链路，建议归入后续 `理想型/精准搜索` 或推荐相关 PRD，不纳入本文件本期范围。
2. `推荐算法`、`量表测评` 相关配置不在本期展开。

### 4.4.3 其他移动端配置

对齐总体后台文档：`3.7.5 其他移动端配置`

## 5. 权限、日志与脱敏要求

对应总体后台文档：`3.1 权限与账号体系`

本期继续沿用总体后台既有要求，并对关系反馈链路补充如下约束：

1. 只有具备对应数据权限的角色，才能查看用户关系记录详情。
2. 手机号、实名信息、身份证号等敏感字段继续按总体文档要求脱敏展示。
3. 查看关系记录、改动关系规则配置、处理处罚结果，均需记录操作日志。
4. 高风险操作如手工失效记录、恢复关系记录，应仅向超级管理员开放，并要求二次确认。

## 6. 本期不单独立项但可承接的轻量能力

以下能力如甲方希望首版可见，建议只做轻量实现，不单独起系统：

1. 在用户详情页增加 `关系记录` 二级 Tab
2. 在配置页增加 `女性保护机制` 配置组
3. 在处罚记录中增加 `影响关系链路` 标签
4. 在运营消息模板中增加 `匹配成功`、`解锁引导`、`保护提示` 模板

## 7. 后续增值服务建议

以下能力不建议纳入当前报价必交范围，如甲方后续需要可作为增值服务：

1. 独立 `关系反馈运营工作台`

## 8. 结论
