# 用户准入页面说明

> 对应 PRD：`管理后台细化PRD-01_用户准入与资料认证初始化.md`
> 对应设计稿：Figma `Page 1 / 后台-客户管理`、`Page 1 / 后台-客户管理-画像弹窗`
> 口径说明：本页在设计稿中以 `客户管理` 命名，实际承担用户准入阶段的客户池管理、画像查看与牵线入口；页面布局以 Figma 为准，字段与操作口径以 git 提交前版本为主，仅在设计稿明确体现新层级或新交互时补充说明。

## 1. 页面定位

- 左侧一级导航高亮为 `匹配`
- 顶部页签依次为 `客户管理`、`牵线台操作`、`画像标签`、`匹配规则`
- 当前页签为 `客户管理`
- 页面主体不是传统后台表格，而是 `客户卡片宫格页`

## 2. 整体布局

### 2.1 外层框架

1. 页面使用后台标准壳布局：`左侧导航 + 顶部页签栏 + 主内容区`。
2. 左侧导航为浅灰蓝底色窄侧栏，当前高亮项为 `匹配`。
3. 顶部为独立白色页签栏，左侧放二级页签，右侧放通知铃铛、头像与管理员名称。
4. 主内容区位于顶部页签栏下方，分为上下两块白色内容面板。

### 2.2 主内容分区

1. 第一块为 `数据概览区`，位于页面上半部。
2. 第二块为 `筛选 + 客户卡片列表区`，位于页面下半部。
3. 页面整体留白较多，内容采用大圆角白卡承载，不要做成满屏表格密集排布。

## 3. 数据概览区

### 3.1 布局形式

1. 数据概览区是一个横向大卡片容器。
2. 卡片内部仅在左侧排布 3 个统计小卡，右侧保留大面积留白。
3. 3 个统计卡为同一行排列，宽度一致，间距较小，视觉上偏轻。

### 3.2 统计卡内容

1. `实名人数`
2. `千寻币`
3. `VIP金额`

### 3.3 设计要求

1. 每个统计卡都带浅色渐变背景和圆形图标区。
2. 数值字号明显大于标题，作为该区主视觉焦点。
3. 本区只承载概览数据，不承载筛选、操作或表格字段。

## 4. 筛选与列表区

### 4.1 筛选条布局

1. 筛选条位于第二块内容面板顶部，单行横向排布。
2. 从左到右依次为：
   `搜索客户` 输入框、`会员等级` 下拉、`跟进状态` 下拉、`跟进状态` 下拉、`搜索` 按钮、`重置` 按钮。
3. 输入框内带搜索图标。
4. `搜索` 为主按钮，`重置` 为次按钮。

### 4.2 筛选条说明

1. 设计稿当前展示了两个同名 `跟进状态` 下拉，文档按视觉稿保留该结构。
2. 后续若产品确认两个下拉对应不同业务含义，应保持当前横向布局不变，只替换文案和数据来源。
3. 设计稿首屏未体现 `导出`、`批量操作`、`高级筛选折叠区`，不应默认放入本页首屏主操作区。

## 5. 客户卡片列表

### 5.1 列表形态

1. 列表采用 `5 列卡片宫格` 布局。
2. 首屏展示 `2 行` 卡片。
3. 卡片之间横向、纵向间距统一，整体更接近客户池/销售线索面板，而不是后台表格。

### 5.2 单张卡片结构

1. 卡片顶部为客户头像、昵称、基础标签行。
2. 顶部右侧放状态胶囊，例如 `头像审核`。
3. 部分卡片左上角带深色小角标，用于表达特殊状态或重点标记。
4. 中部为多行资料信息，按横线分隔。
5. 底部为两个等分操作入口：`画像`、`牵线`。

### 5.3 单张卡片展示信息

1. 顶部展示用户头像、昵称和一行基础身份摘要。
2. 顶部右侧展示状态胶囊，设计稿当前可见为 `头像审核`。
3. 中部展示若干行资料摘要，当前设计稿可见内容主要是：
   资产信息、职业/公司、学历/学校、红娘信息。
4. 摘要信息下方展示多枚标签胶囊。
5. 底部保留两个等分操作入口：
   `画像`、`牵线`，中间有竖向分割线。
6. 卡片首屏只承载字段的轻量摘要，完整字段仍以本文件后续字段口径章节为准。

### 5.4 布局约束

1. 本页首屏必须维持 `卡片式客户池`，不要改成带多列表头的表格。
2. 卡片首屏只展示字段摘要，不承载完整后台字段展开。
3. 更详细的列表字段、详情字段与操作口径，统一在下方章节定义，并通过右侧详情抽屉承载。

## 6. 分页区

1. 分页摘要位于列表左下角，展示口径类似 `共400条记录 第1 / 40页`。
2. 分页控件位于列表右下角。
3. 分页控件包含：
   上一页、页码、更多页省略、跳转页输入、每页条数选择。
4. 分页是独立底部区域，不与筛选条混排。

## 8. 详情页面弹窗标准

### 8.1 标准目标

1. 本页详情相关交互统一采用 `弹层式` 结构，不再拆成独立详情页。
2. 一级详情统一为 `右侧详情抽屉`。
3. 二级审核/图片查看统一为 `居中模态弹窗`。
4. 同类客户详情、资料查看、审核操作，后续均应复用该标准，不应为单个功能再发明新的详情容器形态。

### 8.2 弹层层级标准

1. `L1：右侧详情抽屉`
   适用于查看客户完整画像、资产明细、红娘备注、牵线入口等重信息场景。
2. `L2：居中审核弹窗`
   适用于查看单张图片、执行单步审核、确认通过/失败等轻决策场景。
3. 层级关系为：
   主列表页 -> L1 详情抽屉 -> L2 审核弹窗。
4. 打开 L2 时，L1 不关闭，只在其上增加新的遮罩与弹窗，关闭 L2 后回到原 L1 状态。

### 8.3 L1 右侧详情抽屉标准

1. 触发方式：
   点击客户卡片底部 `画像` 入口打开。
2. 呈现方式：
   页面左侧客户列表保留并加蒙层，右侧滑出白色抽屉面板。
3. 布局特征：
   抽屉为大宽度、接近半屏的固定侧边层，顶部和底部均为固定操作区。
4. 顶部栏标准：
   左侧放标题 `画像详情`，右侧放关闭按钮。
5. 内容区标准：
   内容按模块纵向分段，每段都采用 `浅灰蓝标题条 + 白色内容体` 的组合样式。
6. 底部栏标准：
   底部固定操作栏右侧放主按钮 `快速牵线`，底部操作不得随内容滚动丢失。

### 8.4 L1 抽屉内容模块标准

1. 模块顺序标准：
   基本信息 -> 资产/千寻币明细 -> 红娘备注。
2. 基本信息模块标准：
   顶部先展示头像、昵称、基础身份信息；
   右侧放黑金色 `VIP会员` 信息卡；
   下方为多列文字摘要和标签区域。
3. 该模块实际承载的字段，以 `7.6 基本资料字段`、`7.7 认证信息字段`、`7.8 准入信息字段` 为主。
4. 资产明细模块标准：
   采用表格式明细，设计稿当前可见列为 `时间`、`类型`、`金额`、`余额`、`来源/用途`。
5. 红娘备注模块标准：
   标题区左侧为模块名，右侧为 `添加备注`；
   内容区采用表格，设计稿当前可见列为 `红娘`、`备注时间`、`备注内容`。
6. 长文本标准：
   `择偶要求`、`备注内容` 这类文本优先保证内容展示宽度，不要压缩成过窄列。
7. 本节约定的是详情区的版式结构；具体字段口径仍以 `7.5` 至 `7.11` 为主。

### 8.5 L1 抽屉视觉标准

1. 模块标题条统一为浅灰蓝底，左侧配蓝色竖条强调。
2. 模块内容体统一为白底、浅边框卡片。
3. 表格头与表格行高度保持一致节奏，视觉上偏轻，不使用重表格线。
4. 抽屉内容区允许滚动，但顶部标题栏和底部操作栏需固定。

### 8.6 L2 居中审核弹窗标准

1. 标准名称示例：
   `头像审核`。
2. 触发场景：
   从详情抽屉内点击头像、审核图片或相关审核入口后打开。
3. 呈现方式：
   在当前页面与当前抽屉之上再叠加一层遮罩，并居中弹出白色模态框。
4. 结构标准：
   顶部标题栏 + 中部预览区 + 底部按钮区。
5. 顶部标题栏：
   左侧为弹窗标题，右侧为关闭按钮。
6. 中部预览区：
   以大尺寸图片预览为核心，图片放在浅灰底的居中预览容器内。
7. 底部按钮区：
   按钮右对齐，左侧为次按钮 `审核失败`，右侧为主按钮 `审核通过`。

### 8.7 L2 审核弹窗交互标准

1. L2 的目的应聚焦于 `单对象查看 + 单步审核决策`，不要塞入过多表单项。
2. 若只是查看图片，不应额外带复杂资料字段，重点保留大图预览。
3. `审核失败` 使用次级白色按钮样式，`审核通过` 使用蓝色主按钮样式。
4. 关闭 L2 后，应返回 L1 的原滚动位置与上下文，不应强制刷新或关闭整个详情层。

### 8.8 标准复用要求

1. 同类详情查看优先复用 `L1 右侧详情抽屉`。
2. 同类图片查看和审核动作优先复用 `L2 居中审核弹窗`。
3. 详情相关视觉语言统一沿用本页：白底大圆角、浅灰蓝模块头、蓝色主按钮、右侧抽屉 + 居中模态的双层结构。

## 9. 页面交互要求

1. 顶部页签切换保持同一页面框架，不应跳出当前后台主壳。
2. `画像` 是查看详情入口，优先打开 `L1 右侧详情抽屉`。
3. `牵线` 是业务操作入口，保持作为卡片底部一级动作，不要藏到更多菜单中。
4. `头像审核` 等图片审核动作，优先在 `L2 居中审核弹窗` 中完成。
5. 当前设计稿未体现 `冻结/解冻`、`启用/停用` 的首屏按钮，这类强后台动作不应破坏卡片布局，可下沉到详情抽屉或后续操作面板。
