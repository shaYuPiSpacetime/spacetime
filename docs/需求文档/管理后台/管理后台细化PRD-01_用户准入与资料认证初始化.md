# 成家立业管理后台细化 PRD-01

## 1. 文档信息

- 文档主题：用户准入与资料认证初始化后台细化需求
- 对应移动端文档：`细化PRD-01_用户准入与资料认证初始化`
- 对齐基准文档：`成家立业 · 管理后台产品需求文档 V1.0`
- 使用目标：在不突破现有报价文档模块边界的前提下，细化 PRD-01 对后台的落地要求

## 2. 对齐原则

本文件以总体后台需求文档为基准，不重新定义一级模块，不默认新增独立后台系统能力。仅当移动端 PRD-01 无法落入现有后台模块时，才补充 `必要新增` 的二级能力；其余可增强但非本期必需的内容，统一归为 `后续增值服务建议`。

### 2.1 本文档与总体后台文档的映射关系

| 本文档内容 | 对齐总体后台文档模块 | 说明 |
| --- | --- | --- |
| 用户列表、用户详情、账号状态处理 | `3.2 用户管理中心` | 直接细化 |
| 实名、学历、头像认证处理 | `3.2 用户管理中心` | 作为用户管理下的认证审核能力细化 |
| 资料附加照片、开放性文字审核 | `3.5.3 内容审核` | 直接细化，但范围仅限资料内容 |
| 首登规则、准入门槛、认证超时、上传限制 | `3.6.2 参数配置` | 直接细化 |
| 学校、标签、学历、地区等字典 | `3.6.3 字典与枚举管理` | 直接细化 |
| 认证模板、认证引导文案 | `3.7.5 其他移动端配置` | 直接细化 |
| 权限控制、操作日志、敏感字段脱敏 | `3.1 权限与账号体系`、`3.1.5 审计与安全` | 直接引用，不重复发明模块 |

### 2.2 本期不单独立项的能力

以下能力不建议在本期作为独立模块写入报价交付范围：

1. 独立 `工作台`
2. 独立 `异常任务中心`
3. 独立 `操作日志中心`
4. 独立 `准入策略中心`
5. 审核 SLA 看板、质检中心、专题 BI 报表

说明：

1. 以上能力如需要，可在现有页面中以列表筛选、状态标签、日志抽屉等轻量形式承接。
2. 若甲方后续希望做成独立一级/二级菜单，建议作为增值服务二期扩展。

## 3. 本期后台范围定义

PRD-01 对后台的本期范围，控制在以下 5 类能力：

1. 用户准入相关信息的查询与人工处理
2. 实名认证、学历认证、头像认证的审核承接
3. 资料附加照片与开放性文字内容审核
4. 与准入相关的参数、文案、字典配置
5. 权限、日志、脱敏等基础安全能力对上述功能的支撑

### 4.1.3 用户生命周期管理

对齐总体后台文档：`3.2.3 用户生命周期管理`

本期重点只细化与准入相关的部分：

1. 启用/停用：
   停用后用户不可正常使用小程序，且不进入真实曝光链路。
2. 冻结/解冻：
   冻结后用户不可登录，或按业务规则禁止匹配、发言等能力。
3. 注销处理：
   继续沿用总体文档定义，不在本文件重复展开。
4. 黑白名单：
   本期非 PRD-01 必需，不做细化。

### 4.1.4 客服操作

对齐总体后台文档：`3.2.4 客服操作`

结合小程序当前是 `微信授权登录` 模式，本期客服操作建议调整为：

1. 清除异常登录态
2. 冻结/解冻账号
3. 处理解封申诉
4. 查看认证驳回原因并引导用户重新提交

说明：

1. 总体后台文档中的 `帮用户重置密码` 更适用于账号密码体系。
2. 对当前微信授权登录场景，该项不作为 PRD-01 本期必做。

### 4.1.5 认证审核能力

这是本文件对 `3.2 用户管理中心` 的必要细化，不新增一级菜单，建议作为 `用户管理中心` 下的二级页或详情 Tab。

#### 实名认证审核

目标：承接微信官方人脸核身或兜底方案的后台查看与处理。

建议字段：

2. 昵称
3. 提交时间
4. 当前状态

说明：

2. `人工直接通过实名认证` 属高风险动作，应挂靠总体文档的权限与审计能力，默认不向普通角色开放。

#### 头像认证审核

3. 必要时标记需用户重新上传

### 4.1.6 必要新增说明

总体后台文档虽然写到了 `认证审核`，但没有把三类认证的后台承接方式拆开。本文件在 `用户管理中心` 下细化 `实名认证/学历认证/头像认证` 三类审核视图，属于 `必要细化`，不建议包装成新的独立一级模块。

## 4.2 内容审核

对应总体后台文档：`3.5.3 内容审核`

本期范围只覆盖用户资料体系内的审核，不展开社区动态、评论、举报处理的完整后台链路。

### 4.2.1 资料附加照片审核

审核对象：

1. 笑起来的样子
2. 生活中的样子
3. 旅行照
4. 才艺照
5. 其他资料页附加照片

列表建议字段：

1. 审核记录 ID
2. 用户 ID
3. 昵称
4. 照片缩略图
5. 提交时间
6. 审核状态
7. 驳回原因

操作建议：

1. 通过
2. 驳回

规则要求：

1. 单张资料附加照片审核驳回，仅影响该照片展示。
2. 资料附加照片审核结果不得写入 `头像认证状态`。

### 4.2.2 开放性文字审核

审核对象：

1. 关于我
2. 希望 TA 了解
3. 后续扩展的自由输入文本

列表建议字段：

1. 审核记录 ID
2. 用户 ID
3. 昵称
4. 文本字段类型
5. 文本摘要
6. 提交时间
7. 审核状态
8. 驳回原因

操作建议：

1. 通过
2. 驳回

规则要求：

1. 文本审核驳回，仅影响对应文本展示。
2. 文本审核状态不得与图片审核或认证状态混用。

### 4.2.3 头像审核边界

需要特别说明：

1. `头像认证` 属于用户管理中心下的认证审核能力。
2. `资料附加照片审核` 属于内容审核能力。
3. 两者必须分开存储状态、分开展示结果、分开记录驳回原因。

对应总体后台文档：`3.1 权限与账号体系`、`3.1.5 审计与安全`

本文件不新增模块，只补充 PRD-01 场景下的要求：

## 5. 数据与状态要求

## 5.1 后台需要可见的核心状态

1. 首登资料完成状态
2. 资料完整度分
3. 实名认证状态
4. 学历认证状态
5. 头像认证状态
6. 资料附加照片审核状态
7. 开放性文字审核状态
8. 核心准入状态
9. 用户状态

## 5.2 状态隔离要求

必须明确隔离以下 5 类状态：

1. 实名认证
2. 学历认证
3. 头像认证
4. 资料附加照片审核
5. 开放性文字审核

约束：

1. 主头像认证通过，不代表资料附加照片自动通过。
2. 资料附加照片通过，不代表开放性文字自动通过。
3. 某一类状态驳回，不影响其他类别已通过状态。

## 6. 异常与边界场景

本期建议在现有页面内承接，不单独做 `异常任务中心`。

### 6.1 典型异常

1. 用户注册成功但未完成首登资料：
   后台用户详情中可见停留状态。
2. 实名认证中途退出：
   后台状态保留为处理中或待继续。
3. 学历认证超时未返回：
   列表中仍显示 `审核中`，并允许人工跟进。
4. 头像认证超时未返回：
   列表中仍显示 `审核中`，并允许人工跟进。
5. 用户更换主头像：
   需触发头像重新审核。
6. 用户修改影响认证结果的字段：
   需提示可能需要重新审核。

### 6.2 本期承接方式

1. 通过列表筛选 `审核中/异常` 状态处理
2. 通过详情页查看驳回原因与处理记录
3. 通过操作日志追溯后台人工动作

说明：

1. 若后续希望建设 `异常自动派单、领取、流转、SLA`，建议作为增值服务二期处理。

## 7. 接口与实现建议

本期以总体后台文档的共享服务层思路为准，不额外放大为独立中台。

### 7.1 后台侧接口分组建议

1. `用户列表/详情接口`
2. `认证记录查询接口`
3. `内容审核查询与处理接口`
4. `参数配置接口`
5. `字典管理接口`
6. `认证模板与文案配置接口`

### 7.2 实现建议

1. 后台与移动端共享用户、认证、审核主数据。
2. 配置类数据尽量挂到现有参数配置与移动端配置体系下。
3. 不建议为 PRD-01 单独建设一套新的后台工程边界。

## 8. 本期交付建议

结合总体后台报价文档，PRD-01 建议纳入本期交付的内容如下：

1. 用户列表补充准入与认证相关字段
2. 用户详情补充准入、认证、资料完成度信息
3. 用户管理下增加三类认证审核视图或详情页能力
4. 内容审核下增加资料附加照片、开放性文字审核
5. 参数配置中补充首登、准入、认证、上传限制相关配置项
6. 字典管理中补充学校、标签、学历、地区等基础字典维护
7. 移动端配置中补充认证模板与准入引导文案配置
8. 权限、日志、脱敏对以上功能生效

## 9. 后续增值服务建议

以下建议不写入本期报价必交范围，更适合作为后续增值服务：

1. 独立工作台与漏斗看板
2. 独立异常任务中心
3. 审核 SLA、预警看板、质检抽检
4. 更细粒度的人工准入覆盖策略
5. 首登与认证转化数据专题报表
6. 更强的批量审核与批量运营能力

## 10. 结论

本文件已经按总体后台需求文档进行收敛和对齐：

1. 不新增一级模块。
2. 绝大多数能力挂靠在已有的 `用户管理中心`、`内容审核`、`系统管理中心`、`移动端配置管理` 下实现。
3. 仅将 `三类认证审核视图` 视为对现有 `认证审核` 能力的必要细化，而非新增独立系统。
4. 其余可做但非本期必需的能力，统一归为后续增值服务。
