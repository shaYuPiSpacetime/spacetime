## 2. 模块定位

本模块用于承接小程序前期拉新裂变需求，目标不是短期盈利，而是快速做注册增长与基础活跃增长。

本模块分为两条业务线：

1. `普通用户邀请裂变`
   用户将小程序分享给同学或好友，对方完成三项认证后，计为成功邀请；同时可在注册成功、资料完成、三项认证完成等节点分阶段发放成家币奖励。
2. `校园代理推广`
   由后台生成校园代理二维码，由校园代理在线下或私域渠道推广，后台按注册、资料完成、三项认证、付费等维度统计数量，其中成功邀请口径统一按三项认证完成计算，并据此发放奖金。

## 3. 核心业务目标

### 3.1 普通用户裂变目标

- 让愿意主动转发的用户，快速获得足够体验核心玩法的奖励币。
- 满足甲方“前期只要成功邀请 2 位同学完成三项认证，就让邀请人基本可以免费玩”的运营目标。

### 3.2 校园代理目标

- 支持学校、社团、代理渠道的专属推广。
- 能清晰统计每位代理带来的注册、登录、激活与消费数量。
- 能导出奖金结算依据，方便线下发放奖金或佣金。

## 5. 业务定义

## 5.1 角色定义

### 5.1.1 邀请人 `Inviter`

- 已注册的普通小程序用户
- 生成个人邀请二维码，邀请新用户扫码进入小程序

### 5.1.2 被邀请人 `Invitee`

- 通过普通用户二维码或校园代理二维码扫码进入小程序的新用户

### 5.1.3 校园代理 `CampusAgent`

- 由后台录入和管理的合作代理人
- 生成校园代理二维码获客
- 奖励以 `奖金/佣金` 为主，不走普通用户发币逻辑

## 5.2 关键事件定义

### 5.2.1 邀请点击

- 用户扫码进入小程序
- 仅记录来源，不立即建立有效奖励关系

### 5.2.2 首次注册并成功登录

- 被邀请用户为新用户
- 完成首次注册并成功进入小程序首页
- 该事件用于建立邀请关系，并可作为分阶段奖励事件之一

### 5.2.3 资料完成

- 被邀请用户完成基础资料填写
- 可作为普通用户裂变的追加奖励事件

### 5.2.4 三项认证完成

- 被邀请用户完成 `实名认证 + 头像认证 + 学历认证`
- 这是普通用户与校园代理两条线统一的 `成功邀请` 口径
- 阶梯奖励、成功邀请人数统计、代理成功人数统计均按该口径计算

### 5.2.5 首次付费

- 被邀请用户首次购买 VIP
- 或首次充值成家币

首版保留能力，但前期可关闭，不作为主激励目标。

## 6. 邀请关系绑定规则

### 6.1 绑定来源

统一通过二维码进入小程序，后台来源仅区分二维码生成方：

1. 普通用户：由已注册普通用户生成二维码，被邀请人扫码后归属该普通用户邀请
2. 校园代理：由后台校园代理生成二维码，被邀请人扫码后归属该校园代理

### 6.2 绑定时机

邀请关系按以下规则建立：

1. 被邀请人通过带 `inviterId / qrCode` 的二维码路径进入小程序
2. 系统记录来源信息
3. 若该用户此前从未注册：
   - 在首次注册并成功登录后，建立邀请关系
4. 若该用户已注册：
   - 不再建立新的邀请关系

### 6.3 关系唯一性

1. 一个新用户只能绑定一个邀请来源
2. 一旦绑定成功，不允许被后续来源覆盖
3. 普通邀请和代理邀请不可重复计奖

### 6.4 失效规则

以下情况关系无效：

1. 被邀请用户不是新用户
2. 邀请人与被邀请人为同一账号
3. 命中设备、手机号、UnionId 等反作弊规则
4. 被后台判定为异常裂变

## 7.3 邀请记录状态

| 状态 | 说明 |
| --- | --- |
| registered | 已注册 |
| profile_completed | 已完成基础资料 |
| verify_success | 已完成三项认证 |
| frozen | 风险冻结中 |

## 7.4 普通用户奖励规则

### 7.4.1 首版设计原则

1. 裂变奖励以 `首次注册并成功登录` 为主要触发条件
2. 邀请关系在首次注册并成功登录后建立，但 `成功邀请人数` 统一按三项认证完成计算
3. 奖励金额支持后台配置
4. 奖励可采用阶梯式配置

### 7.4.2 推荐默认策略

为满足“邀请 2 位同学就能基本免费玩”的目标，建议后台默认开启：

1. `注册成功登录奖励`
   - 被邀请人首次注册并成功登录后发放
2. `资料完成奖励`
   - 被邀请人完成基础资料后发放
3. `三项认证完成奖励`
   - 被邀请人完成三项认证后发放
   - 并计入成功邀请人数与阶梯统计
2. `阶梯奖励`
   - 按 `0-2 / 3-8 / 9-20 / 21-50` 四档配置
   - 阶梯统计以“三项认证完成的成功邀请人数”为准
5. `消费返利奖励`
   - 首期默认关闭或弱化

### 7.4.3 奖励到账方式

- 奖励以成家币形式到账
- 到账后进入资产流水
- 同时通过通知中心推送到账通知
- 不同奖励事件可分次到账

### 7.4.4 到账时机

- 被邀请人满足对应触发条件后，奖励实时到账
- 若命中风控，则状态改为 `冻结中`，待人工复核后到账

## 7.5 邀请记录页

### 7.5.1 列表项

1. 被邀请人脱敏昵称/手机号
2. 绑定时间
3. 当前状态
4. 已发奖励
5. 无效原因/冻结原因

### 7.5.2 筛选

- 全部
- 已成功
- 已奖励
- 冻结中
- 无效

## 7.6 规则页

规则页需至少明确：

1. 必须是新用户才计入邀请
2. 必须通过邀请链接或二维码进入才计入关系
3. 首次注册并成功登录后建立邀请关系
4. 完成三项认证后才算成功邀请
5. 注册成功、资料完成、三项认证完成可分别触发奖励
6. 作弊、刷量、自邀不计奖
7. 平台有权冻结异常奖励并人工复核

## 8. 校园代理推广

## 8.1 业务定位

校园代理不走“普通用户邀请发币”模式，而是走“代理渠道统计 + 后台发奖金/佣金”模式。

## 8.2 承接方式

### 8.2.1 渠道识别

每位代理拥有独立：

1. qrCode
2. 专属二维码
3. 专属小程序路径
4. 可选专属海报

### 8.2.2 移动端用户感知

对被代理带来的新用户而言：

1. 正常进入小程序注册
2. 不要求看到复杂代理页面
3. 只需后台准确识别来源并计入代理统计

### 8.2.3 普通邀请与代理邀请关系

1. 同一新用户只归属一个来源
2. 渠道优先，若命中校园代理二维码则优先归代理渠道
3. 不再给某个普通邀请人重复发币

## 8.3 代理统计维度

移动端链路需支持后台统计以下事件：

1. 扫码/点击量
2. 首次注册人数
3. 首次成功登录人数
4. 基础资料完成人数
5. 三项认证完成人数
6. 首次购买 VIP 人数
7. 首次充值成家币人数

说明：

1. 校园代理 `成功邀请人数` 与普通用户一致，统一按 `三项认证完成` 计算。

## 9. 奖励通知与资产联动

## 9.1 通知类型

新增以下通知：

1. 邀请成功奖励到账
2. 阶梯奖励升级提醒
3. 邀请奖励冻结提醒
4. 邀请奖励无效提醒

## 9.2 资产联动

- 普通用户奖励到账后，写入成家币流水
- 流水类型建议新增：
  - `invite_register_reward`
  - `invite_profile_reward`
  - `invite_verify_reward`
  - `invite_ladder_reward`
  - `invite_risk_release`

## 10. 反作弊与风控规则

## 10.1 基础限制

以下场景默认不计奖：

1. 自己邀请自己
2. 老用户重复注册
3. 同手机号重复邀请
4. 同设备批量注册
5. 同一身份证/实名信息命中重复关系
6. 同一支付账号反复命中异常

## 10.2 风控状态

奖励状态新增：

1. pending
2. success
3. frozen
4. invalid

## 10.3 前台提示

- 被冻结时提示：
  - `奖励正在核验中，请稍后查看`
- 被判无效时提示：
  - `该邀请未满足活动规则，未计入奖励`

## 14. 已确认结论

| 已确认事项 | 最终确定 |
| --- | --- |
| 普通用户奖励币名称 | 现阶段正式文档继续统一使用 `成家币` |
| 普通用户成功邀请口径 | 完成 `实名认证 + 头像认证 + 学历认证` 三项认证后，才算成功邀请 |

| 普通邀请与代理邀请归属 | 渠道优先，命中代理渠道时优先归代理 |
| 校园代理成功邀请口径 | 与普通用户一致，三项认证完成才算邀请成功 |
| 校园代理奖金发放方式 | 线下人工结算奖金 |
| 单日奖励上限与风控阈值 | 后台可控制，默认启用奖励上限与风控冻结机制 |
