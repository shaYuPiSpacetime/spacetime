# 成家立业移动端细化 PRD-08

## 1. 文档信息

- 文档主题：推荐算法（觅缘匹配与精选）
- 合并范围：
  - 觅缘推荐流（恋爱匹配卡片流）
  - 精选页面（高质量用户推荐）
- 关联前置文档：
  - `细化PRD-01_用户准入与资料认证初始化`（用户资料、认证状态、准入门槛）
  - `细化PRD-02_关系反馈与互动链路`（喜欢/不喜欢/超级喜欢、匹配后的互动行为）
  - `细化PRD-04_商业化（VIP、成家币、解锁与资产中心）`（推荐相关的付费能力）
- 目标：输出一份同时覆盖产品需求与算法技术设计的完整 PRD，可直接支撑产品设计、后端算法工程、联调与测试。
- 算法依据文档：`docs/相关资料/平台算法.docx`

## 2. 模块定位

本模块是"成家"核心链路的大脑——决定**谁在觅缘卡片流中看到谁、以什么顺序看到、精选页面展示哪些人**。

核心职责：

1. 为新用户（行为数据不足）提供冷启动推荐能力。
2. 为存量用户提供基于混合推荐（协同过滤 + 内容过滤）的召回与粗排。
3. 通过互惠算法计算双向匹配度，提升匹配成功率。
4. 通过 CatBoost 进行精排，输出 Top 20 推荐结果。
5. 利用 SHAP 对内解释推荐理由，支撑运营调优。

适用范围：

- **觅缘推荐流**：恋爱匹配卡片流，用户左右滑动，进入匹配→聊天链路。
- **精选页面**：高质量用户聚合展示，与觅缘共享同一套模型，可叠加额外筛选规则（如资料完整度 > 80%）。
- **不覆盖**：觅知音（兴趣社交朋友推荐）——该场景使用简单的时间/热度排序，不走本算法。

## 3. 设计原则

1. 双向互惠优于单向偏好：匹配必须同时考虑 A→B 和 B→A 两个方向的偏好，双向都高才高分。
2. 冷启动可用但不粗糙：新用户无行为数据时，靠静态属性（学校、年龄、兴趣标签、婚姻状态等）做出有意义的推荐。
3. 行为数据驱动精排：随用户行为积累，逐步从纯内容过滤过渡到混合模型 + CatBoost 精排。
4. 可配置、可解释：特征权重、阈值、打分规则均可在后台配置；SHAP 为内部运营提供推荐理由。
5. 架构轻量：首版离线批量训练，架构预留增量更新能力。
6. 尊重准入规则：推荐结果受 PRD-01 定义的实名认证准入规则约束——未实名用户仅可浏览，不参与曝光。

## 4. 推荐场景定义

### 4.1 觅缘推荐流

| 维度 | 说明 |
| --- | --- |
| 入口 | 底部导航"成家" → 觅缘卡片流 |
| 交互方式 | 左滑（不喜欢）、右滑（喜欢）、超级喜欢、点击进入用户主页 |
| 推荐数量 | 单次拉取 20 张卡片，支持翻页（每次 20 张） |
| 排序逻辑 | CatBoost 精排后 Top 20，已曝光未交互的卡片降权 |
| 去重规则 | 24 小时内不重复展示同一用户；用户左滑后不再出现（除非全量耗尽） |
| 过滤规则 | 异性过滤（性别维度，非打分项）；已拉黑/已举报用户不展示；注销/冻结账号不展示 |
| 准入条件 | 用户需完成首登资料 + 实名认证通过方可参与曝光和匹配操作（浏览无限制） |

### 4.2 精选页面

| 维度 | 说明 |
| --- | --- |
| 入口 | "精选"Tab 或成家模块内精选入口 |
| 定位 | 高质量用户聚合展示，增强信任感与留存 |
| 推荐逻辑 | 与觅缘共享同一模型，叠加质量过滤（资料完整度 > 80%、头像认证通过） |
| 排序 | CatBoost 精排分数降序，可按同城优先置顶 |
| 展示数量 | 分页加载，每页 20 条 |

### 4.3 觅知音 / 悦目 / 诚意贴（不在此算法范围）

这些场景使用简单规则排序（发布时间倒序 / 热度加权），不走推荐算法，不在本文档展开。

## 5. 算法总体架构

```
                    ┌─────────────────────────┐
                    │   用户行为数据积累       │
                    │  (滑动/消息/浏览/互动)   │
                    └────────────┬────────────┘
                                 │
              total_actions >= 10? ──否──→ 冷启动路径
                                 │              │
                                是              ↓
                                 │      ┌───────────────┐
                                 ↓      │ 内容过滤      │
                    ┌─────────────────┐  │ (静态属性匹配) │
                    │ 混合召回        │  └───────┬───────┘
                    │ 协同过滤+内容过滤│          │
                    └───────┬─────────┘          │
                            │                    │
                            └────────┬───────────┘
                                     │
                                     ↓
                            ┌─────────────────┐
                            │ 互惠算法        │
                            │ 双向偏好计算    │
                            └───────┬─────────┘
                                    │
                                    ↓
                            ┌─────────────────┐
                            │ CatBoost 精排   │
                            │ + SHAP 可解释性 │
                            └───────┬─────────┘
                                    │
                                    ↓
                            ┌─────────────────┐
                            │ Top 20 推荐结果 │
                            └─────────────────┘
```

### 5.1 用户分组逻辑

```
if total_actions >= 10:
    分组 = "Existing User（现有用户）"
    → 混合推荐模型（协同过滤 + 内容过滤）
else:
    分组 = "New/Inactive User（新用户/不活跃用户）"
    → 反向匹配模型（冷启动方案）
```

- `total_actions`：用户在平台上的总行为次数，包括滑动（左/右）、超级喜欢、收藏、浏览主页、发消息。
- 阈值 `10` 为默认值，后台可配置。

### 5.2 三层管道

| 层级 | 算法 | 职责 | 输入 | 输出 |
| --- | --- | --- | --- | --- |
| 召回/粗排 | 混合推荐（协同过滤 + 内容过滤） | 从全量用户池中召回候选集 | 用户特征 + 行为矩阵 | 候选集（数百人） |
| 互惠排序 | 互惠算法（Reciprocal Recommendation） | 计算双向匹配度 | 候选集 + Preference/Attractiveness Profile | 互惠分排序（筛至百人内） |
| 精排 | CatBoost + SHAP | 精确排序，输出 Top 20 | 互惠排序结果 + 全部特征 | Top 20 卡片 + SHAP 解释 |

## 6. 特征体系

### 6.1 静态属性特征

以下特征全部来自 `UserProfile`（PRD-01 定义），用于内容过滤和冷启动：

| 特征 | 类型 | 匹配方式 | 说明 |
| --- | --- | --- | --- |
| 性别 | 离散 | 过滤条件 | 异性过滤，不参与打分 |
| 年龄 | 连续 | 差值梯度 | 差值 ≤ 3 岁满分 1.0，差值越大分数越低 |
| 学历 | 离散 | 同等级满分 | 等级相同 = 1.0 |
| 学校 | 离散 | 同校加权 | 同校 = 额外加权分 |
| 专业 | 离散 | 相似度 | 同专业或相近专业 = 加权 |
| 家乡 | 离散 | 层级匹配 | 同省 > 同市 > 同区县 |
| 居住地 | 离散 | 层级匹配 | 同城满分 1.0，同省内递减 |
| 婚姻状态 | 离散 | 匹配则满分 | 相同 = 1.0，不同 = 0.3 |
| 感情状态 | 离散 | 匹配则满分 | 同为"正在寻觅"等配对逻辑 |
| 脱单目标 | 离散 | 匹配则满分 | 同为"1-2 年内结婚"等配对逻辑 |
| 身高 | 连续 | 范围匹配 | 在用户设定的偏好范围内 = 1.0 |
| MBTI | 离散 | 配对表 | 基于 MBTI 最佳配对表给分 |
| 兴趣标签 | 多值离散 | Jaccard 相似度 | 两用户标签集合的交并比 |
| 照片数量 | 连续 | 正相关 | 照片越多，资料可信度越高 |

### 6.2 "匹配则满分"规则详表

以下离散属性采用"二值匹配"——相同给满分 1.0，不同给低分 0.3（后台可配）：

| 属性 | 满分条件 | 满分值 | 不匹配值 | 来源 |
| --- | --- | --- | --- | --- |
| 婚姻状态 | 两人状态完全相同 | 1.0 | 0.3 | 算法文档 |
| 感情状态 | 两人状态形成合法配对 | 1.0 | 0.3 | 算法文档 |
| 脱单目标 | 目标一致 | 1.0 | 0.3 | 算法文档 |
| 学历 | 等级完全相同 | 1.0 | 梯度递减 | PRD-01 |

以下连续属性采用梯度打分：

| 属性 | 满分条件 | 满分值 | 递减规则 |
| --- | --- | --- | --- |
| 年龄 | 差值 ≤ 3 岁 | 1.0 | 差 5 岁=0.8，差 10 岁=0.4 |
| 地理位置 | 同城 | 1.0 | 同省不同城=0.6，不同省=0.2 |

### 6.3 行为特征

#### 6.3.1 活跃度指标

| 指标 | 权重 | 说明 | 是否可配 |
| --- | --- | --- | --- |
| 最近登录时间 | 30% | 越近越好 | 是 |
| 动态发布频率 | 25% | 近期发动态越多越活跃 | 是 |
| 互动频率（点赞/评论） | 20% | 参与度高 | 是 |
| 资料完整度 | 15% | 头像、自我介绍等 | 是 |
| 在线状态 | 10% | 当前是否在线 | 是 |

活跃度综合评分 = 各指标 × 权重求和，归一化到 [0, 1]。

#### 6.3.2 行为组合评分

| 行为组合 | 评分 |
| --- | --- |
| 访问 + 送爱心 + 发消息 | 3 |
| 访问 + 送爱心 | 2 |
| 访问 | 1 |

#### 6.3.3 应用使用时长标签

| 标签 | 含义 | 范围 | 用户特征 |
| --- | --- | --- | --- |
| Low | 低使用 | < 10 分钟 | 偶尔打开，不太活跃 |
| Medium | 中等使用 | 10-30 分钟 | 正常活跃用户 |
| High | 高使用 | > 30 分钟 | 高频使用，投入度高 |

#### 6.3.4 滑动模式

| 标签 | 右滑比例 | 用户类型 | 匹配质量预期 |
| --- | --- | --- | --- |
| Selective | < 30% | 挑剔谨慎 | 高（认真选择） |
| Average | 30%-70% | 正常用户 | 中等 |
| Generous | > 70% | 随意/海王 | 低（匹配多但质量差） |

说明：右滑比例过高的用户（海王型），匹配质量反而更低——不认真筛选导致聊天质量差，应在精排中对 Generous 型用户的匹配结果做降权。

#### 6.3.5 滑动时间段

| 时段 | 英文标签 | 时间范围 | 行为特征 |
| --- | --- | --- | --- |
| 凌晨 | After Midnight | 0-4 点 | 夜猫子，可能情绪化滑动 |
| 清晨 | Early Morning | 4-8 点 | 早起型，作息规律 |
| 上午 | Morning | 8-12 点 | 通勤时间，碎片使用 |
| 下午 | Afternoon | 12-18 点 | 工作间隙，随意浏览 |
| 晚上 | Evening | 18-22 点 | 黄金时段，用户最活跃 |
| 深夜 | Night | 22-24 点 | 睡前刷一刷 |

推荐质量假设：晚上(Evening)黄金时段匹配成功率最高，凌晨质量最低。CatBoost 训练时此特征作为 categorical feature 参与。

#### 6.3.6 互动行为变量

| 变量 | 含义 | 分析价值 |
| --- | --- | --- |
| likes_received | 收到的点赞数 | 反映资料吸引力 |
| mutual_matches | 相互匹配数 | 直接反映匹配成功量 |
| message_sent_count | 发送消息数 | 反映主动沟通意愿 |
| match_outcome | 匹配结果 | 训练标签来源 |

### 6.4 特征工程（Feature Engineering）

以下衍生特征在训练与推理时实时计算：

```
1. 匹配转化率 = mutual_matches / likes_received（likes_received=0 时分母取 1）
2. 消息成功率 = (match_outcome == 'Date Happened') / message_sent_count（分母为 0 取 1）
3. 挑剔指数 = 1 - swipe_right_ratio
4. 活跃度综合评分 = 
      app_usage_time_min / max(app_usage_time_min) × 0.3
    + message_sent_count / max(message_sent_count) × 0.4
    + swipe_right_ratio × 0.3
5. 夜间用户标识 = (last_active_hour >= 22 OR last_active_hour <= 4) → 1, 否则 0
6. 黄金时段用户 = (last_active_hour >= 18 AND last_active_hour <= 22) → 1, 否则 0
```

### 6.5 行为模式与匹配结果的关系

| 行为模式 | 预期匹配结果 |
| --- | --- |
| 高右滑比例 + 低消息数 | Chat Ignored（海王，不认真） |
| 低右滑比例 + 高消息数 | Date Happened（认真用户） |
| 晚上活跃 + 适度使用 + 高选择性 | Date Happened（高质量用户） |
| 凌晨活跃 + 过度使用 + 低选择性 | Chat Ignored（低质量） |

此关系不作为硬编码规则，而是由 CatBoost 从数据中自动学习。

## 7. 冷启动策略（基于内容过滤的"反向匹配"）

### 7.1 适用条件

`total_actions < 10` 的新用户或不活跃用户。

### 7.2 核心思路

新用户没有行为数据，无法使用协同过滤，因此：

1. **基于静态属性进行内容过滤**：使用第 6.1 节列出的全部静态属性特征进行相似度计算。
2. **反向匹配**：不是单向推荐"用户 A 可能喜欢谁"，而是双向计算：
   - **正向**：A 的"择偶偏好（Preference Profile）"与 B 的"自身条件（Attractiveness Profile）"的匹配度 → A 多大概率喜欢 B
   - **反向**：B 的"择偶偏好（Preference Profile）"与 A 的"自身条件（Attractiveness Profile）"的匹配度 → B 多大概率喜欢 A
   - **最终冷启动分** = 正向分 × 反向分

### 7.3 Preference Profile（择偶偏好）构成

| 来源 | 说明 |
| --- | --- |
| 脱单目标（datingGoal） | 用户填写的脱单时间目标 |
| 感情状态（emotionalStatus） | 用户当前感情状态 |
| 婚姻状态（maritalStatus） | 婚姻状态配对 |
| 理想型标签 | 用户填写的"理想型"偏好标签 |
| 年龄偏好 | 可接受的对方年龄范围（在后续资料编辑中设置，首版可用默认范围 ±5 岁） |
| 兴趣标签 | 用户自身的兴趣标签集 |

### 7.4 Attractiveness Profile（自身条件）构成

即 UserProfile 中的全部静态属性——年龄、学历、学校、身高、家乡、居住地、照片数量、MBTI、兴趣标签等。

### 7.5 冷启动评分公式

```
cold_start_score(A, B) = 
    forward_score(A_pref, B_attr) × backward_score(B_pref, A_attr)
```

其中每个方向的 score 由各静态属性加权求和得出，权重由 CatBoost 在全量数据上训练得出（新老用户共用同一套属性权重）。

### 7.6 冷启动兜底规则

当属性匹配度过低或候选集为空时，启用兜底规则：

1. 同城优先
2. 同校/同专业优先
3. 年龄相近（±5 岁）优先
4. 资料完整度高的优先

### 7.7 冷启动→正常推荐切换

当用户 `total_actions` 达到阈值（默认 10，后台可配），自动从冷启动路径切换至混合推荐路径。切换时需保证用户无感知——用户看到的依然是连续的卡片流。

## 8. 混合推荐（协同过滤 + 内容过滤）

### 8.1 适用条件

`total_actions >= 10` 的现有用户。

### 8.2 协同过滤：ALS 矩阵分解

首版采用 ALS（Alternating Least Squares，交替最小二乘法）进行矩阵分解，参考 hybrid-book-recommender-system 的实现范式。

**用户-行为矩阵：**

- 行 = 用户 u，列 = 被推荐用户 v（在此场景中，被推荐用户也视作"物品"）
- 值 = 行为类型权重：
  - 左滑 = -1（负反馈）
  - 浏览无操作 = 1
  - 右滑/喜欢 = 2
  - 超级喜欢 = 3
  - 发消息 = 5

**ALS 分解公式：**

将稀疏的用户-行为矩阵 R (m×n) 分解为两个低秩矩阵：

```
R ≈ U × V^T

其中：
  U = 用户隐向量矩阵 (m × k)，k 为隐因子维度（建议 k=50）
  V = 用户隐向量矩阵 (n × k)（被推荐用户的隐向量）
```

**预测公式（点积）：**

```
predict(u, v) = U_u · V_v^T = sum_{d=1}^{k} U_{u,d} × V_{v,d}
```

**候选召回策略：**

1. 对当前用户 u，计算其隐向量 `U_u` 与所有候选用户隐向量 `V_v` 的余弦相似度。
2. **KMeans 用户聚类**：对全部用户的 ALS 隐向量 `U` 做 KMeans 聚类（建议 k=10），同一簇内的用户在行为模式上相近，可作为协同过滤的候选池。
3. 取与当前用户最相似的 Top K=100 个候选者进入候选集。
4. 另取同一 KMeans 簇内活跃度高的用户作为补充候选。

**ALS 超参建议（Optuna 搜索范围）：**

| 超参数 | 搜索范围 | 说明 |
| --- | --- | --- |
| factors (k) | 50-200 | 隐因子维度 |
| regularization | 0.001-0.1 | L2 正则化系数 |
| iterations | 10-30 | ALS 交替迭代次数 |
| alpha | 5-40 | 隐式反馈置信度权重 |

### 8.3 内容过滤（Content-Based Filtering）

使用第 6.1 节静态属性特征计算用户间相似度，与冷启动路径的评分逻辑一致。

**文本特征处理（参考 hybrid 项目）：**

对可文本化的用户资料字段（关于我、希望 TA 了解、兴趣标签集合），可采用以下处理：

- **TF-IDF 向量化**：对短文本字段（如"关于我"）使用 `TfidfVectorizer(max_features=200)` 提取关键词权重向量
- **语义嵌入（二期）**：使用轻量级预训练模型（如 all-MiniLM-L6-v2, 384 维）对用户自述文本做语义编码，计算语义级相似度
- **标签集 Jaccard 相似度**：对兴趣标签直接计算 `|A ∩ B| / |A ∪ B|`

首版内容过滤以结构化属性匹配为主（6.1 节），文本语义嵌入留到二期。

### 8.4 混合方式：双塔候选生成

参考 hybrid-book-recommender-system 的 ALS + Qdrant 双塔候选生成模式，本系统采用 **ALS 相似度 + 内容相似度** 双通道：

```
候选集 = ALS召回(100人) ∪ 内容过滤召回(100人)

hybrid_score = α × als_cosine_similarity + (1 - α) × content_similarity
```

- α：协同过滤权重，默认 0.6（后台可配）
- als_cosine_similarity：ALS 隐向量的余弦相似度（归一化至 [0, 1]）
- content_similarity：静态属性加权相似度（归一化至 [0, 1]）

**去重与合并规则：**

- 同一候选用户同时出现在两个通道时，取较高分
- 候选集附带 `source` 标记（ALS / Content），作为 CatBoost 精排的 categorical feature
- 混合后的候选集规模控制在 200 人以内，进入下一层互惠算法

## 9. 互惠匹配算法

### 9.1 核心原理

传统推荐是单向的（"你可能喜欢 X"），但婚恋匹配必须是双向的（"你们互相喜欢"）。

互惠算法的本质是**同时计算 A→B 和 B→A 两个方向的偏好强度**：

```
reciprocal_score(A, B) = similarity(A_pref, B_attr) × similarity(B_pref, A_attr)
```

即：
- **正向分**：A 的 Preference Profile 与 B 的 Attractiveness Profile 的相似度
- **反向分**：B 的 Preference Profile 与 A 的 Attractiveness Profile 的相似度
- **互惠分**：正向分 × 反向分

### 9.2 Profile 构成

与冷启动中的定义一致（见 7.3、7.4）：

- **Preference Profile**：脱单目标、感情状态、婚姻状态、理想型标签、年龄偏好、兴趣标签
- **Attractiveness Profile**：年龄、学历、学校、身高、家乡、居住地、照片、MBTI、兴趣标签

### 9.3 算法优势

| 维度 | 传统单向推荐 | 互惠算法 | 效果 |
| --- | --- | --- | --- |
| 偏好方向 | 只看 A→B | A→B 且 B→A | 避免"我喜欢但对方不喜欢"的无效推荐 |
| 冷启动 | 新用户无数据 | 基于属性可直接计算 | 新用户也能获得合理推荐 |
| 匹配转化率 | 低 | 高 | 双方都感兴趣才推荐，提升聊天转化 |

### 9.4 排序优化参考：CLIMF（Collaborative Less-is-More Filtering）

LibRec 库（v3.0.0）中的 CLIMF 算法（Shi et al., RecSys 2012）直接优化**平均倒数排名（MRR）**，可作为互惠排序层的优化参考。

**核心思想：** 传统 BPR 优化的是"正样本排在负样本前面的概率"，而 CLIMF 直接优化整个排序列表的 MRR：

```
CLIMF 损失函数（平滑 MRR 下界）：

L(U,V) = -sum_{i in I_u} [ log(σ(predict(u,i)))
         - sum_{j in I_u, j≠i} log(1 - σ(predict(u,j) - predict(u,i))) ]
         + λ(||U||² + ||V||²)

其中 σ(x) = 1/(1 + exp(-x)) 为 logistic 函数
```

**与 BPR 的对比：**

| 维度 | BPR | CLIMF |
| --- | --- | --- |
| 优化目标 | 单对排序 (pairwise) | 全列表排序 (listwise) |
| 评估一致 | 间接优化 AUC | 直接优化 MRR |
| 适用场景 | 隐式反馈排序 | Top-N 推荐（对本场景更直接） |
| 计算开销 | 低（双采样） | 中（对每个正样本遍历其余正样本） |

**在本系统中的应用方式：**

互惠排序层的输出可送入 CatBoost 精排时，附带一个 `reciprocal_rank_score` 特征。CLIMF 的思想可用于后续将排序目标从"预测是否匹配"升级为"预测在推荐列表中的位置"，使优化目标与业务指标（MRR）对齐。

### 9.5 互惠排序的工程实现模式

参考 hybrid-book-recommender-system 的推理管道（`rec_pipeline.py`）：

```
1. 构建用户向量：
   user_vector = mean(已喜欢用户的 ALS 向量) 

2. 双向计算：
   forward_score  = cosine_similarity(A_pref_vector, B_attr_matrix)
   backward_score = cosine_similarity(B_pref_vector, A_attr_matrix)

3. 互惠分：
   reciprocal_score = forward_score × backward_score

4. 排序：按 reciprocal_score 降序，取 Top N
```

此模式可将 Preference Profile 和 Attractiveness Profile 分别编码为向量，通过对用户已喜欢/已互动对象的行为来隐式学习偏好，无需用户显式填写偏好。

## 10. CatBoost 精排

### 10.1 模型定位

CatBoost 作为精排模型，承接互惠算法筛选后的候选集（百人级别），对所有 (user_u, candidate_v) pair 做精确打分排序，最终输出 Top 20。

### 10.2 训练标签（Relevance Label）

| 标签值 | 含义 | 判定条件 |
| --- | --- | --- |
| 5 | 强正样本 | 双方互发消息，且有持续聊天（聊天轮次 ≥ 3 或 时长 ≥ 10 分钟） |
| 3 | 中正样本 | 一方主动发消息，对方回复了 |
| 2 | 弱正样本 | 有收藏 / 超级喜欢 / 长时间浏览，但未发消息 |
| 1 | 浏览正样本 | 仅浏览 / 右滑 / 点喜欢 |
| 0 | 负样本 | 左滑、明确拒绝、浏览后无任何后续动作、或已多次展示未交互 |

### 10.3 输入特征

CatBoost 模型接收的特征包括：

1. **互惠分**（来自上一层的 reciprocal_score）
2. **候选来源标记**（`source`：ALS 召回 / Content 召回，categorical feature）
3. **两个通道的排名**（`rank_als`、`rank_content`：该候选在各自通道的排名位置）
4. **静态属性相似度**（6.1 中各维度的单项分）
5. **行为特征**（6.3 中各行为变量的实时值）
6. **衍生特征**（6.4 中特征工程的 6 个指标）
7. **用户聚类标签**（来自 ALS 隐向量的 KMeans 聚类，categorical feature）
8. **用户画像嵌入**（可选，后续迭代——将 UserProfile 各维度做 Embedding 后作为特征输入）

### 10.4 训练细节

**样本构造：**

训练样本为 (user_u, candidate_v) pair，每个 pair 对应一个 relevance_label（5/3/2/1/0）。样本构成：

- **正样本**：用户有过互动（label ≥ 1）的真实 pair
- **负样本**：候选集中用户未互动的 pair，以及已曝光多次但无交互的 pair（隐式负样本）

由于正样本远少于负样本（参考 hybrid 项目中约 1:4 的比例），首版训练时需做负采样或配置 `class_weight`：

- 方案 A（推荐）：按正负 1:4 的比例对负样本做随机下采样
- 方案 B：直接使用 CatBoost 的 `scale_pos_weight` 参数

**训练/验证集划分：**

**按用户分组划分（Grouped Split）**，而非随机打散——同一用户的所有 (user, candidate) pair 必须在同一集合中，防止数据泄漏。参考 hybrid 项目中的 `groupby('user_id')` 划分方式。

**CatBoost 核心超参数（首版建议 + Optuna 搜索范围）：**

以下搜索空间参考 hybrid-book-recommender-system 中验证有效的范围：

| 超参数 | 搜索范围 | 说明 |
| --- | --- | --- |
| iterations | 200-1000 | 迭代轮数（树的数量） |
| depth | 4-10 | 树深度，较大值捕获更复杂特征交互 |
| learning_rate | 0.01-0.3（log scale） | 学习率，与 iterations 反比关系 |
| l2_leaf_reg | 1.0-20.0 | L2 正则化，防止过拟合 |
| border_count | 32-255 | 数值特征分箱数，影响训练速度和精度 |
| bagging_temperature | 0.0-1.0 | 贝叶斯 Bootstrap 温度，增加随机性 |
| random_strength | 1.0-20.0 | 分数随机噪声标准差，防过拟合 |
| rsm | 0.5-1.0 | 每棵树使用的特征比例 |
| grow_policy | SymmetricTree / Depthwise | 树增长策略 |

**评估指标（训练时）：**

- 分类指标：F1 macro、AUC
- 早停（Early Stopping）：验证集 F1 连续 30 轮不提升则停止

### 10.5 输出

- 每个 (user_u, candidate_v) pair 的精排分数（0-1）
- 按分数降序排序，取 Top 20 作为推荐结果
- SHAP 值（每个特征对该 pair 分数的贡献值），用于内部解释

### 10.6 CatBoost 内置特征重要度（SHAP 轻量替代）

SHAP 计算开销大，不适合高频使用。CatBoost 内置的 `get_feature_importance()` 可作为日常使用的轻量替代：

- **LossFunctionChange**：基于移除该特征后损失函数变化的特征重要度
- **PredictionValuesChange**：基于该特征对预测值平均贡献的重要度

特点：计算快（模型训练时自动产出），适合每周模型训练后快速查看特征重要度排名。

### 10.7 参考实现

hybrid-book-recommender-system（https://github.com/KonNik88/hybrid-book-recommender-system.git）中的完整 CatBoost 训练管道——包括 Optuna 搜索、双塔候选生成、特征拼接、per-user 分组评估——作为核心参考。该项目在 Goodbooks-10k 数据集上达到 MAP@10=0.57, MRR@10=0.81, AUC=0.90。

## 11. SHAP 可解释性（内部使用）

### 11.1 定位

SHAP（SHapley Additive exPlanations）用于**对内解释推荐理由**，帮助产品和运营理解"为什么推荐这个人给这个用户"。不对用户端展示。

### 11.2 输出内容

对每个推荐 pair，输出每个特征的 SHAP 值（正贡献/负贡献），例如：

```
推荐 A → B：总分为 0.87
  + 同校：+0.15
  + 兴趣标签相似：+0.12
  + 婚姻状态匹配：+0.10
  - 距离较远：-0.05
  ...
```

### 11.3 使用场景

1. 运营排查：某用户反馈推荐质量差时，通过 SHAP 值排查哪些特征拉低了分数。
2. 算法调优：分析全局 SHAP 均值，识别贡献度最高/最低的特征，针对性调整权重或特征工程。
3. 阶段性复盘：对比不同版本模型的特征重要度变化，验证优化方向。

### 11.4 实现方案

参考 hybrid-book-recommender-system 的做法——SHAP 在项目中 import 但实际生产中未被调用，原因是 TreeExplainer 对大规模样本的计算耗时过高。

**首版实际落地策略：**

1. **日常使用**：CatBoost 内置 `get_feature_importance()`，模型训练后自动产出特征重要度排名，满足日常监控。
2. **深度排查时**：对问题用户样本（百级）调用 `shap.TreeExplainer(model).shap_values(X_sample)` 生成 SHAP 值，定位具体特征贡献。首版仅限管理员手动触发，不自动化。
3. **二期目标**：每周离线对采样用户（万级）批量计算 SHAP 值，存入数据仓库供运营自助查看。

### 11.5 限制

- SHAP 计算开销较大，TreeExplainer 对 1000+ 样本、100+ 特征的场景需数分钟。
- 首版只在离线批处理时对采样样本计算，不对全量推荐请求实时计算。
- 首版只输出 SHAP 特征重要度排名（全局 Top 20），不做逐条推荐的实时解释。

### 13.2 优化目标

最大化 NDCG@20（Normalized Discounted Cumulative Gain @ 20）。

备选目标（当 NDCG 变化不显著时切换）：
- MRR@20（Mean Reciprocal Rank）—— 第一个正确推荐排在第几位
- MAP@20（Mean Average Precision）—— 综合衡量准确率和召回率

### 13.3 执行策略

- 每周模型训练时自动执行一次 Optuna 搜索（建议 30-50 trials，参考 hybrid 项目中使用 30 trials 即收敛）。
- 早停（Early Stopping）机制：验证集 F1 连续 30 轮不提升则提前终止当前 trial，节省搜索时间。
- 搜索结果（最优超参数组合）自动应用到当周模型训练。
- 超参数变更记录在操作日志中，历史最优参数集持久化存储，可回溯对比。

## 14. 评估指标

### 14.1 离线评估

**评估方式：Per-User 分组评估**

参考 hybrid-book-recommender-system 的做法——离线评估必须按用户分组（group by user_id），同一用户的所有 pair 在同一集合中，先按用户分别计算指标，再取所有用户的均值。这模拟了真实推荐场景（每个用户看到一个排序列表）。

**评估函数定义（参考 hybrid 项目实现）：**

```
AP@K（Average Precision @ K）：
  AP_u@K = (1/min(K, |relevant_u|)) × sum_{k=1}^{K} P@k × rel(k)
  其中 P@k = (前k个中相关数)/k，rel(k) = 1 如果第k个相关，否则 0
  
  MAP@K = (1/|U|) × sum_{u in U} AP_u@K

MRR@K（Mean Reciprocal Rank @ K）：
  MRR@K = (1/|U|) × sum_{u in U} (1 / rank_u)
  其中 rank_u = 用户u的第一个相关推荐在列表中的位置

NDCG@K（Normalized Discounted Cumulative Gain @ K）：
  DCG_u@K = sum_{k=1}^{K} (2^{rel_k} - 1) / log2(k + 1)
  NDCG_u@K = DCG_u@K / IDCG_u@K（IDCG 为理想排序下的 DCG）
  NDCG@K = (1/|U|) × sum_{u in U} NDCG_u@K
```

**评估指标与目标：**

| 指标 | 公式 | 说明 | 参考目标 |
| --- | --- | --- | --- |
| AUC | 标准 ROC-AUC | 模型区分正负样本的能力 | > 0.85 |
| MAP@20 | 见上 | 综合衡量 Top 20 的准确率和召回率 | > 0.30 |
| NDCG@20 | 见上 | Top 20 推荐排序质量，含位置衰减 | 持续优化 |
| MRR@20 | 见上 | 第一个相关推荐的排名倒数 | > 0.70 |
| Precision@20 | 相关数/K | Top 20 中正样本（label ≥ 3）的比例 | > 0.30 |
| Recall@20 | 相关数/总相关 | 正样本中被推荐的比例 | 持续优化 |

### 14.2 在线评估（业务指标）

| 指标 | 说明 | 监控方式 |
| --- | --- | --- |
| 右滑率 | 推荐卡片中被右滑的比例 | 后台实时统计 |
| 匹配转化率 | 右滑中双方互滑的比例 | 后台实时统计 |
| 消息转化率 | 匹配后发消息的比例 | 后台实时统计 |
| 约会转化率 | 持续聊天（≥ 3 轮）的比例 | 后台实时统计 |
| 人均滑动次数 | 用户每次会话的滑动量 | 后台实时统计 |
| 推荐覆盖率 | 全量推荐候选用户池占平台总用户的比例 | 离线周报 |

### 16.4 推理管道（Predict-Time Pipeline）

参考 hybrid-book-recommender-system 的 `rec_pipeline.py` 实现模式：

```
predict(user_id):
  1. 加载用户特征：静态属性 + 行为特征 + ALS 隐向量
  2. 候选召回：
     a. ALS 余弦相似度 → Top 100
     b. 内容过滤相似度 → Top 100
     c. 合并去重（取高分）→ 候选集 ~150 人
  3. 互惠计算：
     对每个候选 v:
       forward  = similarity(user_pref, v_attr)
       backward = similarity(v_pref, user_attr)
       reciprocal_score = forward × backward
  4. 特征拼接：候选集 + reciprocal_score + 全部静态/行为/衍生特征
  5. CatBoost 推理：对 150 个 (user, candidate) pair 打分
  6. 排序取 Top 20，组装卡片响应
```

向量存储方式：
- ALS 隐向量以 NumPy `.npy` 文件存储，通过 `mmap_mode='r'` 加载，支持内存高效的余弦相似度计算
- 首版训练后由 Python 导出因子矩阵，推荐服务通过文件读取加载
- 二期可考虑使用向量数据库（如 Milvus/Qdrant）替代内存计算

### 已确认重点

1. 推荐场景限定为觅缘 + 精选，觅知音不在此算法范围
2. 冷启动采用反向匹配（正向 × 反向），阈值为 total_actions < 10
3. 首版离线批量训练，架构预留增量更新
