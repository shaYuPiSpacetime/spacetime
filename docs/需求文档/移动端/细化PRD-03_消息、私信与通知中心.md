# 成家立业移动端细化 PRD-03

## 1. 文档信息

- 文档主题：消息、私信与通知中心
- 对应原始关联模块：
  - `消息` Tab
  - 私信会话
  - 悄悄话
  - 通知中心
  - 通知管理
- 关联前置文档：
  - `细化PRD-01_用户准入与资料认证初始化`
  - `细化PRD-02_关系反馈与互动链路`
- 目标：输出一份可直接支撑产品设计、前后端开发、联调、测试的细化 PRD

## 2. 模块定位

本模块承接 `PRD-02` 中已经形成的社交反馈结果，让用户从“被喜欢 / 被访问 / 匹配成功”进入真实互动，同时承接审核结果、举报结果、资产变动、活动通知等系统消息触达。

它承担两类职责：

1. 社交互动职责：
   承接私信、悄悄话回复、匹配成功后的聊天。
2. 信息触达职责：
   承接系统通知、互动提醒、资产通知、活动通知。

## 3. 设计原则

1. 先匹配，后常规私信：普通私信能力以 `匹配成功` 为前置，不再沿用总 PRD 中 `互相关注即可私信` 的旧口径。
2. 系统消息独立开放：即使用户未完成三项认证，也可查看系统消息与审核结果。
3. 会话与通知分层：用户会话和系统通知分开展示，避免信息噪音互相干扰。
4. 保护优先：对女性用户的聊天保护机制优先于普通会话自由发送。
5. 轻量首发：首版只做文本消息、悄悄话、通知中心，不展开图片、语音、撤回、通话等复杂能力。
6. 与支付解耦：本模块只定义悄悄话、VIP、成家币的触发点与规则，不展开完整支付页。

## 4. 页面范围

本模块涉及以下页面：

1. `消息列表页`
2. `私信对话页`
3. `悄悄话卡片/回复态`
4. `通知中心页`
5. `通知管理页`

## 5. 与 PRD-01、PRD-02 的关系

本模块严格继承以下已确认规则：

1. 未完成 `实名认证 + 头像认证 + 学历认证` 任一项的用户，不开放真实聊天链路。
2. `匹配成功` 定义沿用 PRD-02：
   - 双方互送爱心
   - 精选页送出 `对 ta 心动` 后对方回送爱心
   - 悄悄话后对方回复
3. 匹配成功后，主页中的 `Yo` 改为 `聊天 icon`，底部主按钮改为 `聊天 icon + 聊天`。
4. 悄悄话属于普通私信前的更进一步互动入口。
5. 被拉黑、被封禁、被注销等异常状态会中断聊天链路。

补充明确：

1. `消息 Tab` 对所有已登录用户可见。
2. 未完成三项认证的用户：
   - 可查看 `官方消息 / 系统通知`
   - 不展示用户私信会话
   - 不可发送悄悄话
3. 普通私信开口统一收敛为：`匹配成功后才开放`
4. 设置中的“谁可以给我发私信”相关旧口径，需同步收敛为适配 `匹配成功后开放` 的新文案。

## 6. 用户流程总览

## 6.1 匹配成功后进入聊天

1. 用户 A 与用户 B 已形成 `匹配成功`
2. 任一方点击：
   - 匹配成功弹窗中的 `去聊天`
   - 相互喜欢列表中的 `聊天`
   - 用户主页中的 `聊天 icon`
3. 系统创建或打开双方会话
4. 用户进入 `私信对话页`
5. 按女性保护机制、拉黑状态、账号状态判断是否允许发送

## 6.2 悄悄话链路

1. 用户 A 在前置社交场景中对用户 B 发出 `悄悄话`
2. 悄悄话以特殊消息卡片形式进入双方消息系统
3. 用户 B 可：
   - 回复
   - 忽略
4. 若 B 回复，则系统判定 `匹配成功`
5. 会话从 `悄悄话等待态` 进入 `正常私信态`

## 6.3 系统通知链路

1. 用户触发审核、举报、充值、到期、活动等系统事件
2. 系统生成对应通知
3. 用户在 `消息列表 -> 官方消息 / 通知中心` 查看
4. 点击通知跳转对应详情页

## 9. 核心状态与规则

## 9.1 模块开放规则

1. `消息 Tab`：
   所有已登录用户可见。
2. `官方消息 / 通知中心`：
   所有已登录用户可查看。
3. `用户私信会话`：
   仅对完成三项认证且满足会话条件的用户开放。
4. `悄悄话`：
   仅对完成三项认证且具备对应权益的用户开放。

## 9.2 普通私信开放规则

首版普通私信统一采用以下规则：

1. 双方均完成 `实名认证 + 头像认证 + 学历认证`
2. 双方账号状态正常
3. 双方不在彼此黑名单中
4. 双方已形成 `匹配成功`

明确废止旧口径：

1. 不再沿用 `互相关注后可免费私信`
2. 不再把 `关注` 作为私信开放前置条件

## 9.3 悄悄话规则

1. 悄悄话是 `匹配成功前` 的特殊互动消息
2. 悄悄话发送资格：
   - 当前用户已完成三项认证
   - 对方为可见且正常用户
   - 普通用户可直接使用成家币发送
   - VIP 用户每日享 `1` 次免费悄悄话，用完后也可继续使用成家币发送
3. 悄悄话发送后：
   - 发送方会话状态为 `pending_whisper`
   - 接收方可在消息系统中看到悄悄话卡片
   - 在上一条悄悄话未被回复或未失效前，不允许向同一目标用户重复发送新的悄悄话
4. 接收方回复悄悄话后：
   - 判定双方 `匹配成功`
   - 会话切换为正常私信态
5. 接收方忽略悄悄话：
   - 不形成匹配成功
   - 不开放普通私信
   - 本次悄悄话关闭
   - 发送方 `7 天内` 不可再次向同一用户发送悄悄话
   - 页面展示轻提示，说明对方暂未回应

## 9.4 女性保护机制

已确认规则：

1. 匹配成功后的前 `3 天` 为保护期
2. 保护期从 `匹配成功时间` 开始连续计算，不依赖是否进入过聊天页
3. 若对话双方为 `男 + 女` 组合，则保护期内：
   - 女性未发送首条消息前，男性不可主动发送普通私信
   - 仅当女性已发送消息后，男性才可继续发送
4. 只有女性发送的真实用户消息，才算解除男性禁发：
   - 普通文本消息
   - 对悄悄话的明确回复
5. 系统提示消息不计入“女性已回复”
6. 若女性先发出消息或先回复悄悄话，则会话立即解除男性禁发限制
7. 保护期结束后，双方按普通私信规则正常发送

产品表现：

1. 男性端输入框置灰
2. 页面提示：`已开启女生保护机制，待对方回复后即可继续聊天`
3. 女性端正常展示输入框

## 9.5 拉黑、举报、异常状态规则

1. 当前用户拉黑对方后：
   - 会话立即不可继续发送
   - 保留历史聊天记录
   - 会话进入 `invalid` 失效态
2. 对方拉黑当前用户后：
   - 保留历史聊天记录
   - 当前会话进入 `invalid` 失效态
   - 当前用户不可继续发新消息
   - 消息列表继续展示该会话，但标记不可聊天
3. 任一方被冻结、封禁、注销中、已注销后：
   - 会话状态改为 `invalid`
   - 输入框置灰，不可继续发送
4. 举报入口继续保留在聊天页右上角更多菜单中

## 9.6 未读与已读规则

1. 会话列表未读数按会话维度累计
2. 进入会话页后，本会话消息置为已读
3. 通知中心点击单条后，该条置已读
4. 支持 `全部已读`
5. `消息` Tab 红点为：
   `私信未读 + 官方消息未读 + 通知中心未读`
6. 红点数字超过 `99` 时展示 `99+`
7. 若用户在 `通知管理` 中关闭某类通知：
   - 不再触发该类外部提醒
   - 不再计入消息 Tab 红点
   - 但通知中心历史记录继续保留并可查看


## 9.7 记录保留规则

1. 私信记录长期保留
2. 通知信息一直保留
3. 前台默认按正常分页加载历史，不因为保留期额外裁剪数据

## 10. 页面级详细需求

## 10.1 消息列表页

### 10.1.1 页面目标

聚合用户私信会话与官方消息入口，承接未读提醒和消息分发。

### 10.1.2 页面入口

1. 底部 Tab `消息`
2. 匹配成功后的回流入口

### 10.1.3 页面结构

1. 顶部标题：`消息`
2. 右上角：`通知` 图标
3. 固顶 `官方消息` 卡片
4. 私信会话列表

### 10.1.4 官方消息卡片

展示内容：

1. 官方头像/图标
2. 名称：`官方消息`
3. 最新消息摘要
4. 时间
5. 未读红点

点击行为：

1. 进入 `通知中心页`

### 10.1.5 私信会话列表

每条展示：

1. 对方头像
2. 对方昵称
3. 最后消息摘要
4. 时间：
   - 刚刚
   - X 分钟前
   - 昨天
   - 日期
5. 未读数
6. 会话状态标识：
   - 悄悄话待回复
   - 已匹配
   - 已失效

排序规则：

1. 按最后消息时间倒序
2. 官方消息固定在私信列表上方

### 10.1.6 未完成三项认证时

1. 页面仍可进入
2. 可查看 `官方消息`
3. 不展示用户私信会话
4. 私信会话区展示空态与认证引导
5. 文案示例：
   `完成实名认证、头像认证、学历认证后，才可开启真实聊天`

### 10.1.7 空态

1. 无私信会话：
   `还没有新的聊天，去成家看看谁和你更有缘`
2. 无通知未读：
   `暂时没有新的通知`

## 10.2 私信对话页

### 10.2.1 页面目标

承接匹配成功后的双向聊天，以及悄悄话回复后的会话转换。

### 10.2.2 页面入口

1. 消息列表点击私信会话
2. 匹配成功弹窗点击 `去聊天`
3. 相互喜欢页点击 `聊天`
4. 主页点击 `聊天 icon`

### 10.2.3 页面结构

1. 顶部导航：
   - 返回
   - 对方昵称
   - 更多菜单
2. 消息流区域
3. 底部输入区域

### 10.2.4 消息类型

首版支持：

1. 普通文本消息
2. 悄悄话卡片消息
3. 系统提示消息：
   - 已匹配成功

可发送态：

1. 查看对方主页
2. 举报
3. 拉黑

### 10.2.7 历史消息加载

2. 上滑加载更早历史
3. 每次分页加载 `20` 条

## 10.3 悄悄话卡片/回复态

### 10.3.1 发送方视角

1. 会话内展示：
   - 你向 ta 发送了一条悄悄话
   - 等待对方回复
2. 在对方未回复前：
   - 不开放普通私信输入
   - 不重复触发匹配成功
   - 不可再次向同一用户发送新的悄悄话

### 10.3.2 接收方视角

1. 会话内展示悄悄话卡片
2. 支持操作：
   - 回复
   - 忽略

### 10.3.3 回复后

1. 立即判定为 `匹配成功`
2. 写入一条系统提示：
   `你们已成功匹配，快开始聊天吧`
3. 会话切换为正常私信态

## 11. 商业化边界

本模块只定义消息能力涉及的商业触发点：

1. 悄悄话发送：
   - VIP 可享每日免费次数
   - 超出后或普通用户按成家币消耗
2. 成家币不足时：
   - 弹出充值引导
3. 完整价格、套餐、到账流程不在本模块展开

## 14. 异常与边界场景

1. 用户未完成三项认证进入消息 Tab：
   可看官方消息与通知中心，不展示用户私信会话
2. 匹配成功后对方又拉黑：
   保留历史聊天记录，会话失效，不可继续发送
3. 悄悄话发送成功但对方长期未回复：
   保持 `pending_whisper` 状态，不自动转普通私信
4. 同一对象在上一条悄悄话未回复前：
   不允许重复发送新的悄悄话
5. 悄悄话被忽略后：
   本次悄悄话关闭，发送方 `7 天内` 不可再次发送
6. 悄悄话回复后：
   立即转为匹配成功与正常会话
7. 男性在保护期内先进入会话：
   输入框置灰，等待女性先回复
8. 女性在保护期内发送真实用户消息或回复悄悄话：
   男性侧立即解除禁发限制
9. 任一方账号被冻结或注销：
   会话置为失效态
10. 通知开关关闭后：
   关闭该类外部提醒与红点，但不影响站内通知沉淀
