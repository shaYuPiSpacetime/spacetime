# 成家立业移动端细化 PRD-01

## 1. 文档信息

- 文档主题：用户准入与资料认证初始化
- 合并范围：
  - 微信授权登录
  - 首登资料完善
  - 个人资料编辑
  - 认证中心
- 对应原始大模块：
  - 原顺序第 1 步 `首登注册登录 + 新用户资料初始化`
  - 原顺序第 2 步 `个人资料编辑体系 + 资料完整度规则`
  - 原顺序第 3 步 `认证中心`
- 目标：输出一份可直接支撑产品设计、前后端开发、联调、测试的细化 PRD。

## 2. 模块定位

该模块是整个产品的准入层与身份建立层，决定用户是否能顺利进入后续核心婚恋链路。

当前阶段目标用户以 `在校大学生` 为主，因此本模块首版资料模型优先围绕校园身份、基础画像、认证准入展开，不引入工作、职业、年薪、公司等社会化字段。

核心职责：

1. 完成微信授权登录与账号自动创建。
2. 在首登阶段建立最小可用用户档案。
3. 通过资料完善度机制持续引导用户补齐信息。
4. 通过实名认证、学历认证、头像认证建立平台信任基础。
5. 为后续 `觅缘推荐流`、`精选页面`、`理想型搜索`、`喜欢我的`、`最近看过我的`、`消息私信` 提供准入判定依据。

## 3. 设计原则

1. 首登即补资料：借鉴婚恋类产品常见路径，用户首次进入后不直接放开核心能力，而是先建立基础资料。
2. 认证前置但不一步锁死：允许用户注册成功进入产品，但核心交友能力受认证状态控制。
3. 资料完善与认证解耦：资料可逐步补，认证可分步完成，但两者共同影响推荐曝光与功能权限。
4. 首版先做最小闭环：优先支持“可注册、可补资料、可认证、可判断是否可进入觅缘”。
5. 可运营、可配置：必填项、资料完整度权重、认证准入门槛、文案均支持后台配置。
6. 已确认的首版限制规则：未实名认证用户仅可浏览 `觅缘卡片`，不可进行喜欢、不喜欢、打招呼等匹配操作。

## 5. 用户流程总览

### 5.1 标准新用户路径

1. 进入小程序
2. 未登录，进入 `微信授权登录页`
3. 用户点击微信授权登录
4. 授权成功后自动完成登录；若为平台首登用户，则自动创建账号
5. 判断是否为新用户
6. 新用户进入 `首登资料完善页`
7. 完成首批必填资料后，进入 `认证中心引导页`
8. 用户可按顺序完成认证，或先进入产品非核心区
9. 当满足准入条件后，开放 `觅缘推荐流` 等核心能力

### 5.2 老用户路径

1. 进入小程序
2. 已有登录态则直接进入首页
3. 系统检查资料完整度与认证状态
4. 若未满足核心能力准入条件，在相关页面显示认证/补资料横幅

### 5.3 竞品借鉴方向

本模块在流程上借鉴 `青藤之恋` 一类产品的通用策略：

1. 首次进入后先完善资料，而不是直接放开浏览。
2. 学历/身份类认证对平台信任感影响较大，应尽早引导。
3. 用户资料页不仅是编辑后台，也是后续推荐卡片与主页展示的源数据。

说明：以上为竞品路径抽象借鉴，用于指导流程设计，不视为对竞品具体页面的逐项复刻。

## 10. 核心状态机

## 10.1 账号状态机

`未注册 -> 已注册未首登初始化 -> 已完成首登初始化 -> 正常可用 -> 冻结/注销中/已注销`

说明：

1. 注册成功后即进入 `已注册未首登初始化`。
2. 完成首批必填资料后进入 `已完成首登初始化`。
3. 账号是否能进入核心婚恋链路，还需叠加认证状态判断。

## 10.2 资料状态机

`未填写 -> 草稿保存 -> 已提交完整资料 -> 后续补充更新`

说明：

1. 首登资料页支持分步提交。
2. 仅当首批必填字段全部完成时，才视为 `已提交完整资料`。
3. 后续补资料不影响既有账号状态，但会影响资料评分。

## 10.3 认证通用状态机

`未认证 -> 待提交 -> 审核中 -> 已通过`

`未认证 -> 待提交 -> 审核中 -> 已拒绝 -> 重新提交 -> 审核中 -> 已通过`

补充：

1. 对于实名认证，可拆成 `填写信息 -> 人脸识别 -> 提交成功 -> 审核中/即时通过`。

## 10.4 核心能力准入状态机

`不可用 -> 可浏览非核心内容 -> 可进入成家核心链路`

已确认准入规则：

1. 注册成功但未完成首登资料：仅允许停留在资料页和认证页。
2. 已完成首登资料但未完成实名认证：仅允许浏览 `觅缘卡片`，不开放任何匹配操作。
3. 已完成首登资料且实名认证通过：开放曝光、匹配及相关互动能力。

首版已确认的核心认证门槛：

1. 实名认证通过
2. 学历认证、头像认证为增强信任项，不作为首版曝光与匹配的前置门槛

补充说明：

1. “浏览”当前仅指浏览 `觅缘卡片`。
2. “曝光”包含以下全部场景：
   - 出现在别人觅缘推荐流
   - 出现在搜索/筛选结果
   - 出现在喜欢我的
   - 出现在最近来访
3. “匹配”包含以下能力：
   - 喜欢/不喜欢行为真实生效
   - 可形成互相喜欢
   - 可进入后续私信等真实互动链路
4. 社区动态发布需在实名认证通过后才开放。

## 11. 页面级详细需求

## 11.1 微信授权登录页

### 11.1.1 页面目标

完成微信授权登录，并在首次登录时自动创建账号。

### 11.1.2 页面元素

参考 demo `Login.vue`，首版保留小程序授权所需最小元素：

1. Logo/产品名
2. 协议勾选
3. 用户协议/隐私政策入口
4. 微信授权登录按钮

### 11.1.3 交互规则

1. 用户必须勾选协议后，才可点击微信授权登录按钮。
2. 用户点击微信授权登录后，拉起小程序授权流程。
3. 授权成功后，服务端以 `openId/unionId` 完成登录态创建。
4. 若用户为首次登录，则自动完成注册，无需额外注册步骤。
5. 登录成功后必须判断是否新用户，并进入首登资料完善流程。

### 11.1.4 状态

1. 默认态
2. 协议未勾选态
3. 授权中
4. 登录失败
5. 登录成功
6. 首登自动注册成功

### 11.1.5 异常

1. 微信授权拒绝
2. 账号被冻结
3. 网络失败
4. openId/unionId 获取失败
5. 服务端登录态创建失败

## 11.2 首登资料完善页

### 11.2.1 页面目标

在新用户首次登录后，快速收集最小必要资料，建立基础匹配画像。

### 11.2.2 页面定位

该页面不同于“完整资料编辑页”，重点是：

1. 收集最关键字段
2. 尽快让用户完成首登
3. 尽快进入后续认证引导

### 11.2.3 页面结构建议

建议拆成 3 步：

1. `基础身份`
2. `学校与婚恋目标`
3. `头像与关于我`

### 11.2.4 字段

第一步：

1. 昵称
2. 性别
3. 生日
4. 身高
5. 当前居住地
6. 家乡

第二步：

1. 学校
2. 最高学历
3. 感情状态
4. 脱单目标
5. 婚姻状态

第三步：

1. 头像
2. 至少 1 张本人照
3. 关于我
4. 选择 3 个标签

### 11.2.5 交互规则

1. 性别一旦提交不可修改，仅允许客服后台处理。
2. 生日用于计算年龄，年龄需满足平台准入范围。
3. 学校支持搜索联想与兜底手输。
4. 头像需明确提示“用于头像认证审核”。
5. 关于我需有最小字数限制，建议 `20` 字。
6. 标签至少选择 `3` 个。

### 11.2.6 按钮策略

1. 每步提供 `下一步`
2. 支持 `返回上一步`
3. 支持草稿保存
4. 最后一页点击 `完成并进入认证`

### 11.2.7 页面流转

1. 微信授权登录成功且判定为新用户 -> 首登资料第 1 步
2. 完成资料第 3 步 -> 认证中心引导页
3. 若中途退出 -> 下次登录回到未完成步骤
4. 已完成资料但未实名认证 -> 进入觅缘时仅展示卡片浏览态，并持续提示“完成实名认证后，才可曝光和匹配”

## 11.3 编辑资料总页

### 11.3.1 页面目标

作为资料与认证的总入口，展示资料评分、认证进度、基础资料、扩展资料、预览效果。

### 11.3.2 参考 demo

参考 `ProfileEdit.vue`，首版建议保留以下结构：

1. 资料评分
2. 认证提示卡片
3. 头像/背景区
4. 照片上传区
5. 标签区
6. 四重认证区
7. 基础资料卡片
8. 关于我
9. 希望 TA 了解
10. 语音介绍

### 11.3.3 交互规则

1. 点击基础资料卡片进入 `基础资料编辑页`
2. 点击认证项进入对应认证页
3. 点击照片位进入上传/替换
4. 点击标签进入标签编辑页
5. 点击预览切换到“我的主页预览”
6. `主头像`、`资料附加照片`、`开放性文字内容` 审核需隔离：
   - 主头像走 `头像认证`

### 11.6.5 微信官方人脸核身方案可用性结论

首版实名认证建议优先评估接入微信官方人脸核身能力，参考官方服务端接口：

1. `getVerifyId`
2. `queryVerifyInfo`

当前结论如下：

1. `理论可用`，可作为实名认证核心方案候选。
2. `不能只调服务端接口`，必须和小程序前端拉起官方核身流程联动使用。
3. `不保证所有小程序默认可用`，官方文档未明确写死适用账号范围，需以真实 `appid` 调用结果为准。
4. 官方返回码中存在 `10006 appid 没有权限`，说明该能力存在权限校验，不是纯开放接口。

### 11.6.6 微信官方人脸核身接入前置条件

根据微信官方文档，接入该能力至少要满足以下条件：

1. 必须由 `服务端` 调用官方接口，不可由小程序前端直接调用。
2. 需要有可用的 `access_token`。
3. 需要先拿到当前登录用户的 `openid`。
4. 用户需提交真实姓名和身份证号码。
5. `getVerifyId` 时使用的 `openid` 必须和前端发起核身时的用户身份一致。
6. 生成 `verify_id` 所使用的小程序 `appid` 必须与前端实际发起核身的小程序 `appid` 一致。
7. `verify_id` 有效期默认 `3600 秒`，过期后需重新生成。
8. `queryVerifyInfo` 时，`verify_id`、`out_seq_no`、`openid` 必须与 `getVerifyId` 时保持一致。
9. `queryVerifyInfo` 还需要传入 `cert_hash`，需按微信官方规则由证件信息摘要计算得到。
10. 用户设备需支持人脸检测，并允许相机、麦克风权限，否则核身无法完成。

### 11.6.7 微信官方人脸核身业务流程

建议实名认证流程调整为：

1. 用户在实名认证页填写：
   - 真实姓名
   - 身份证号
2. 前端提交给业务服务端。
3. 业务服务端调用微信 `getVerifyId`：
   - 传 `out_seq_no`
   - 传 `cert_info`
   - 传当前用户 `openid`
4. 微信返回 `verify_id` 和有效期。
5. 业务服务端把 `verify_id` 返回给小程序前端。
6. 小程序前端拉起微信官方人脸核身流程。
7. 用户完成刷脸后，业务服务端调用微信 `queryVerifyInfo` 查询最终结果。
8. 当且仅当：
   - `errcode = 0`
   - `verify_ret = 10000`
     时，判定实名认证通过。
9. 业务服务端更新本地实名认证状态，并回写用户资料与准入权限。

### 11.6.8 微信官方人脸核身返回结果判定

实名认证通过判定：

1. `errcode = 0`
2. `verify_ret = 10000`

常见失败/阻断原因：

1. `10006`：`appid 没有权限`
2. `84022`：`verify_id 已过期`
3. `84017`：`openid 不匹配`
4. `84018`：`cert_hash 无效或不匹配`
5. `90008`：相机权限未授权
6. `90009`：麦克风权限未授权
7. `90010`：相机和麦克风权限都未授权
8. `90001 / 90109`：设备不支持人脸检测
9. `10003`：身份证号不匹配
10. `10004`：比对人脸信息不匹配
11. `10005`：正在检测中
12. `10300`：未完成核身

### 11.6.9 产品与研发落地建议

1. 首先用真实小程序 `appid` 做一次联调验证，确认是否具备调用权限。
2. 若校验返回 `10006 appid 没有权限`，则需走微信侧能力申请/开通流程，不能直接以该方案排期上线。
3. 在 PRD 中将该方案定义为：
   - `实名认证优先方案：微信官方人脸核身`
   - `兜底方案：人工审核/第三方实名核验`
4. 前端需在发起核身前明确申请和解释相机、麦克风权限用途。
5. 服务端需保留完整核身流水，包括：
   - `out_seq_no`
   - `verify_id`
   - `openid`
   - `cert_hash`
   - 微信返回结果
   - 本地实名认证状态

## 11.8 头像认证页

### 11.8.1 页面目标

校验主头像是否为本人真实清晰头像。

### 11.8.2 首版建议流程

1. 用户上传或更换主头像
2. 主头像保存成功后，自动触发头像认证，无需额外进入独立页面手动提交
3. 服务端调用第三方头像核验服务商 API
4. 前台状态进入 `审核中`
5. 第三方异步返回核验结果
6. 服务端更新头像认证状态为 `已通过` / `已拒绝`

### 11.8.3 规则

1. 必须上传清晰正脸照。
2. 禁止卡通、宠物、多人合照、遮挡严重照片作为认证头像。
3. 审核不通过需给出驳回原因。

### 11.8.4 第三方头像认证接入原则

1. 首版头像认证不做平台自研人工认证主流程。
2. 首版接入第三方头像核验服务商 API。
3. 认证结果采用 `异步后置` 返回模式，不要求用户等待同步结果。
4. 当前不在 PRD 中锁定具体服务商名称。
5. 当前不在 PRD 中锁定具体第三方 API 字段。
6. 若第三方核验返回失败或不确定结果，可进入人工复核兜底，但人工复核不是首版主链路默认方案。
7. 头像认证与主头像上传流程融合，采用“上传即自动审核”模式。

### 11.8.5 与资料附加照片审核的边界

1. `头像认证` 只针对 `主头像`。
2. `笑起来的样子`、`生活中的样子`、`旅行照` 等资料附加照片，不属于头像认证。
3. 资料附加照片需单独进入 `资料照片审核` 链路。
4. 主头像认证状态与资料附加照片审核状态互不影响、独立存储。

## 11.9 资料内容审核隔离规则

首版必须将以下 5 类内容分开处理，不可共用同一审核记录与状态：

1. `实名认证`
   - 审核对象：姓名、身份证、人脸核身结果
   - 目标：确认真实身份
2. `学历认证`
   - 审核对象：学历信息与第三方验证结果
   - 目标：确认学校/学历真实性
3. `头像认证`
   - 审核对象：主头像与第三方头像核验结果
   - 目标：确认主头像为本人真实头像
4. `资料附加照片审核`
   - 审核对象：笑起来的样子、生活中的样子、旅行照、才艺照等
   - 目标：确保图片合规、真实、可展示
5. `开放性文字审核`
   - 审核对象：关于我、希望 TA 了解等自由输入文本
   - 目标：过滤违规、低俗、广告、联系方式导流等内容

### 11.9.1 资料附加照片审核流程

1. 用户上传资料附加照片
2. 系统生成照片审核记录
3. 照片状态进入 `审核中`
4. 审核通过后，该照片可在个人主页展示
5. 审核驳回后，该照片不可展示，并返回驳回原因

### 11.9.2 开放性文字审核流程

1. 用户提交 `关于我`、`希望 TA 了解` 等文本
2. 系统生成文字审核记录
3. 文本状态进入 `审核中`
4. 审核通过后，对应文本可在个人主页展示
5. 审核驳回后，对应文本不可展示，并返回驳回原因

### 11.9.3 审核隔离要求

1. 主头像通过，不代表资料附加照片自动通过。
2. 资料附加照片通过，不代表开放性文字自动通过。
3. 某一类内容审核驳回，不影响其他类别已通过的状态。
4. 后台审核台需至少拆分为：
   - 资料照片审核
   - 开放性文字审核
5. 前端状态展示也需拆分，不允许只展示一个笼统的“资料审核中”。

   说明：提交当前主头像后，由服务端调用第三方头像核验服务商 API，认证结果异步回写。

## 16. 校验规则

1. 昵称：2-12 字，不允许纯表情，不允许敏感词。
2. 生日：年龄需满足平台允许区间。
3. 身高：140-220 cm。
4. 学校：长度限制 2-50 字。
5. 关于我：最少 20 字，最多 300 字。
6. 标签：至少 3 个，最多 10 个。
7. 主头像：必须上传 1 张。
8. 相册：首版建议最少 3 张，最多 9 张。
9. 身份证号：按国家规则校验格式。

## 17. 异常与边界场景

1. 用户注册成功但首登资料未完成：
   下次登录直接回到未完成步骤。
2. 用户已完成资料，但认证中途退出：
   返回认证中心，保留已完成项。
3. 认证被驳回：
   页面需展示驳回原因并支持重新提交。
4. 用户修改已认证相关字段：
   若修改会影响认证结果，需提示“修改后可能需要重新审核”。
5. 用户更换主头像：
   若头像认证依赖主头像，需重新进入头像审核。
6. 用户年龄不符合平台准入要求：
   拒绝注册或限制继续使用。
7. 学校未检索到：
   允许手输，但进入人工核验风险更高。
8. 第三方学历认证超时或无明确返回：
   前台保持 `审核中`，后台需支持人工跟进或补偿重试。
9. 第三方头像认证超时或无明确返回：
   前台保持 `审核中`，后台需支持人工复核兜底。
10. 资料附加照片审核被驳回：
    仅影响对应照片展示，不影响主头像认证状态。
11. 开放性文字审核被驳回：
    仅影响对应文案展示，不影响照片审核或认证状态。
