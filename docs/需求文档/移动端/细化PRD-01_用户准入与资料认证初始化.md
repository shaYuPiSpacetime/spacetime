# 成家立业移动端细化 PRD-01

## 1. 文档信息

- 文档主题：用户准入与资料认证初始化
- 合并范围：
  - 微信授权登录
  - 首登资料完善
  - 个人资料编辑
  - 认证中心
- 对应原始大模块：
  - 原顺序第 1 步 `首登注册登录 + 新用户资料初始化`
  - 原顺序第 2 步 `个人资料编辑体系 + 资料完整度规则`
  - 原顺序第 3 步 `认证中心`
- 目标：输出一份可直接支撑产品设计、前后端开发、联调、测试的细化 PRD。

## 2. 模块定位

该模块是整个产品的准入层与身份建立层，决定用户是否能顺利进入后续核心婚恋链路。

当前阶段目标用户以 `在校大学生` 为主，同时兼容 `职场人` 身份进入。因此本模块首版资料模型采用 `统一准入流程 + 按身份分支展示字段` 的方式建设：

1. 所有新用户首次登录都先进入同一套首登引导流程。
2. 基础共性字段统一采集，如性别、生日、学历、居住地等。
3. `在校生` 与 `职场人` 在完整资料页展示不同字段：
   - 在校生重点补充 `学校`、`专业`
   - 职场人重点补充 `职业`、`年收入`
4. 不做 `admin` 与 `miniapp` 之外的跨模块资料拼装，所有准入资料仍由本模块统一承接。

核心职责：

1. 完成微信授权登录与账号自动创建。
2. 在首登阶段建立最小可用用户档案。
3. 通过资料完善度机制持续引导用户补齐信息。
4. 通过实名认证、学历认证、头像认证建立平台信任基础。
5. 为后续 `觅缘推荐流`、`精选页面`、`理想型搜索`、`喜欢我的`、`最近看过我的`、`消息私信` 提供准入判定依据。

## 3. 设计原则

1. 首登即补资料：借鉴婚恋类产品常见路径，用户首次进入后不直接放开核心能力，而是先建立基础资料。
2. 认证前置但不一步锁死：允许用户注册成功进入产品，但核心交友能力受认证状态控制。
3. 资料完善与认证解耦：资料可逐步补，认证可分步完成，但两者共同影响推荐曝光与功能权限。
4. 首版先做最小闭环：优先支持“可注册、可补资料、可认证、可判断是否可进入觅缘”。
5. 可运营、可配置：必填项、资料完整度权重、认证准入门槛、文案均支持后台配置。
6. 已确认的首版限制规则：未实名认证用户仅可浏览 `觅缘卡片`，不可进行喜欢、不喜欢、打招呼等匹配操作。

## 5. 用户流程总览

### 5.1 标准新用户路径

1. 进入小程序
2. 未登录，进入 `微信授权登录页`
3. 用户点击微信授权登录
4. 授权成功后自动完成登录；若为平台首登用户，则自动创建账号
5. 判断是否为新用户
6. 新用户进入图 1 对应的 `首登轻量引导流程`
7. 首登轻量引导按顺序展示：`性别 -> 出生日期 -> 身份 -> 学历 -> 常住地`
8. 轻量引导支持跳过；若用户已填写部分字段，系统需立即保存，并在后续完整资料页中 `反显预填写结果`
9. 轻量引导结束后，统一进入图 2 对应的 `完善资料和认证` 强引导页
10. 若用户继续填写，则按页面顺序完成 `基本资料 -> 添加头像 -> 自我介绍 -> 三重认证`
11. 若用户在强引导页再次跳过，则进入 `游客模式`
12. 游客模式下仅允许查看部分公共信息，不进入真实推荐、搜索、曝光、匹配和互动链路
13. 当用户补齐首批必填资料并满足准入条件后，开放 `觅缘推荐流` 等核心能力

### 5.2 老用户路径

1. 进入小程序
2. 已有登录态则直接进入首页
3. 系统检查资料完整度与认证状态
4. 若用户处于 `游客模式`，优先回到 `完善资料和认证` 强引导页
5. 若未满足核心能力准入条件，在相关页面显示认证/补资料横幅

### 5.3 竞品借鉴方向

本模块在流程上借鉴 `青藤之恋` 一类产品的通用策略：

1. 首次进入后先完善资料，而不是直接放开浏览。
2. 学历/身份类认证对平台信任感影响较大，应尽早引导。
3. 用户资料页不仅是编辑后台，也是后续推荐卡片与主页展示的源数据。

说明：以上为竞品路径抽象借鉴，用于指导流程设计，不视为对竞品具体页面的逐项复刻。

当前 `UserProfile` 中保留的字段全部属于首发范围，不再区分“首发建议”与“二期增强”。

补充说明：

### 8.1 身份分支字段规则

1. `identityType` 为首登轻量引导必填字段，且会影响完整资料页的字段展示和校验规则。
2. 当用户选择 `在校生` 时：
   - 必填字段为 `学校`、`学历`
   - `职业`、`年收入` 不展示也不校验
3. 当用户选择 `职场人` 时：
   - 必填字段为 `职业`、`年收入`
   - `学校`、`专业` 改为选填或历史教育信息补充项
4. 用户若修改 `identityType`，系统需提示“已填写的身份相关字段可能被清空并重新校验”。
5. 身份分支规则同时影响：
   - 前端字段展示
   - 必填校验
   - 资料完整度计算
   - 后台资料查看口径

## 9. 资料完整度规则

## 9.1 目标

资料完整度用于：

1. 驱动用户持续补资料
2. 作为推荐排序加权因子
3. 作为商业化和认证引导文案基础
4. 作为个人主页“评分”展示

## 9.2 首版评分建议

总分 100 分。

| 项目 | 分值 |
| --- | --- |
| 昵称 | 5 |
| 主头像 | 10 |
| 性别 | 5 |
| 生日/年龄 | 5 |
| 身份 | 5 |
| 身高/体重 | 5 |
| 居住地 | 5 |
| 家乡 | 5 |
| 脱单目标 | 5 |

1. 用户资料页展示 `资料评分/完整度` 进度条。
2. 完整度低于后台阈值时，展示醒目补全引导。
3. 完整度不作为注册成功前置条件，但可作为推荐准入条件之一。

## 10. 核心状态机

## 10.1 账号状态机

`未注册 -> 已注册未首登初始化 -> 已完成首登初始化 -> 正常可用 -> 冻结/注销中/已注销`

说明：

1. 注册成功后即进入 `已注册未首登初始化`。
2. 完成首批必填资料后进入 `已完成首登初始化`。
3. 账号是否能进入核心婚恋链路，还需叠加认证状态判断。

## 10.2 资料状态机

`未填写 -> 草稿保存 -> 已提交完整资料 -> 后续补充更新`

说明：

1. 首登资料页支持分步提交。
2. 仅当首批必填字段全部完成时，才视为 `已提交完整资料`。
3. 后续补资料不影响既有账号状态，但会影响资料评分。

## 10.3 认证通用状态机

`未认证 -> 待提交 -> 审核中 -> 已通过`

`未认证 -> 待提交 -> 审核中 -> 已拒绝 -> 重新提交 -> 审核中 -> 已通过`

补充：

1. 对于实名认证，可拆成 `填写信息 -> 人脸识别 -> 提交成功 -> 审核中/即时通过`。

## 10.4 核心能力准入状态机

`不可用 -> 游客模式 -> 可浏览非核心内容 -> 可进入成家核心链路`

已确认准入规则：

1. 注册成功后，所有用户都必须先进入图 1 对应的首登轻量引导流程。
2. 首登轻量引导允许跳过；若跳过或仅填写部分字段，系统仍需保存已填内容。
3. 轻量引导结束后统一进入图 2 对应的 `完善资料和认证` 强引导页。
4. 若用户在强引导页再次跳过，则进入 `游客模式`。
5. 游客模式下仅允许查看公共信息，不进入真实推荐、曝光、搜索、匹配和互动链路。
6. 已完成首登资料但未完成实名认证：允许浏览 `觅缘卡片`，但不开放任何匹配操作。
7. 已完成首登资料且实名认证通过：开放曝光、匹配及相关互动能力。

首版已确认的核心认证门槛：

1. 实名认证通过
2. 学历认证、头像认证为增强信任项，不作为首版曝光与匹配的前置门槛

补充说明：

1. `游客模式` 下的“浏览公共信息”不等于浏览真实觅缘推荐流。
2. “浏览”当前仅指 `已完成资料但未实名认证` 用户可浏览 `觅缘卡片`。
3. “曝光”包含以下全部场景：
   - 出现在别人觅缘推荐流
   - 出现在搜索/筛选结果
   - 出现在喜欢我的
   - 出现在最近来访
4. “匹配”包含以下能力：
   - 喜欢/不喜欢行为真实生效
   - 可形成互相喜欢
   - 可进入后续私信等真实互动链路
5. 社区动态发布需在实名认证通过后才开放。

4. 提交成功后进入审核中态。

### 11.6.5 微信官方人脸核身方案可用性结论

首版实名认证建议优先评估接入微信官方人脸核身能力，参考官方服务端接口：

1. `getVerifyId`
2. `queryVerifyInfo`

当前结论如下：

1. `理论可用`，可作为实名认证核心方案候选。
2. `不能只调服务端接口`，必须和小程序前端拉起官方核身流程联动使用。
3. `不保证所有小程序默认可用`，官方文档未明确写死适用账号范围，需以真实 `appid` 调用结果为准。
4. 官方返回码中存在 `10006 appid 没有权限`，说明该能力存在权限校验，不是纯开放接口。

### 11.6.6 微信官方人脸核身接入前置条件

根据微信官方文档，接入该能力至少要满足以下条件：

1. 必须由 `服务端` 调用官方接口，不可由小程序前端直接调用。
2. 需要有可用的 `access_token`。
3. 需要先拿到当前登录用户的 `openid`。
4. 用户需提交真实姓名和身份证号码。
5. `getVerifyId` 时使用的 `openid` 必须和前端发起核身时的用户身份一致。
6. 生成 `verify_id` 所使用的小程序 `appid` 必须与前端实际发起核身的小程序 `appid` 一致。
7. `verify_id` 有效期默认 `3600 秒`，过期后需重新生成。
8. `queryVerifyInfo` 时，`verify_id`、`out_seq_no`、`openid` 必须与 `getVerifyId` 时保持一致。
9. `queryVerifyInfo` 还需要传入 `cert_hash`，需按微信官方规则由证件信息摘要计算得到。
10. 用户设备需支持人脸检测，并允许相机、麦克风权限，否则核身无法完成。

### 11.6.7 微信官方人脸核身业务流程

建议实名认证流程调整为：

1. 用户在实名认证页填写：
   - 真实姓名
   - 身份证号
2. 前端提交给业务服务端。
3. 业务服务端调用微信 `getVerifyId`：
   - 传 `out_seq_no`
   - 传 `cert_info`
   - 传当前用户 `openid`
4. 微信返回 `verify_id` 和有效期。
5. 业务服务端把 `verify_id` 返回给小程序前端。
6. 小程序前端拉起微信官方人脸核身流程。
7. 用户完成刷脸后，业务服务端调用微信 `queryVerifyInfo` 查询最终结果。
8. 当且仅当：
   - `errcode = 0`
   - `verify_ret = 10000`
     时，判定实名认证通过。
9. 业务服务端更新本地实名认证状态，并回写用户资料与准入权限。

### 11.6.8 微信官方人脸核身返回结果判定

实名认证通过判定：

1. `errcode = 0`
2. `verify_ret = 10000`

常见失败/阻断原因：

1. `10006`：`appid 没有权限`
2. `84022`：`verify_id 已过期`
3. `84017`：`openid 不匹配`
4. `84018`：`cert_hash 无效或不匹配`
5. `90008`：相机权限未授权
6. `90009`：麦克风权限未授权
7. `90010`：相机和麦克风权限都未授权
8. `90001 / 90109`：设备不支持人脸检测
9. `10003`：身份证号不匹配
10. `10004`：比对人脸信息不匹配
11. `10005`：正在检测中
12. `10300`：未完成核身

### 11.6.9 产品与研发落地建议

1. 首先用真实小程序 `appid` 做一次联调验证，确认是否具备调用权限。
2. 若校验返回 `10006 appid 没有权限`，则需走微信侧能力申请/开通流程，不能直接以该方案排期上线。
3. 在 PRD 中将该方案定义为：
   - `实名认证优先方案：微信官方人脸核身`
   - `兜底方案：人工审核/第三方实名核验`
4. 前端需在发起核身前明确申请和解释相机、麦克风权限用途。
5. 服务端需保留完整核身流水，包括：
   - `out_seq_no`
   - `verify_id`
   - `openid`
   - `cert_hash`
   - 微信返回结果
   - 本地实名认证状态

3. 头像认证由后台人工审核承接，不走自动第三方核验主链路。
4. 用户在审核结果返回前，前端统一展示 `审核中`。
5. 审核不通过需给出驳回原因。

### 11.8.4 头像认证人工审核原则

1. 首版头像认证由后台人工审核承接。
2. 头像认证与主头像上传流程融合，采用“上传并确认资料后自动提审”模式。
3. 审核中状态由服务端直接写入，前端不等待同步结果。
4. 后台需支持查看当前头像大图、历史头像认证记录、驳回原因、审核操作日志。
5. 用户更换主头像后，原通过结果失效，需重新进入 `审核中`。

### 11.8.5 与资料附加照片审核的边界

1. `头像认证` 只针对 `主头像`。
2. `笑起来的样子`、`生活中的样子`、`旅行照` 等资料附加照片，不属于头像认证。
3. 资料附加照片需单独进入 `资料照片审核` 链路。
4. 主头像认证状态与资料附加照片审核状态互不影响、独立存储。

## 11.9 资料内容审核隔离规则

首版必须将以下 5 类内容分开处理，不可共用同一审核记录与状态：

1. `实名认证`
   - 审核对象：姓名、身份证、人脸核身结果
   - 目标：确认真实身份
2. `学历认证`
   - 审核对象：学历信息与第三方验证结果
   - 目标：确认学校/学历真实性
3. `头像认证`
   - 审核对象：主头像与后台人工审核结果
   - 目标：确认主头像为本人真实头像
4. `资料附加照片审核`
   - 审核对象：笑起来的样子、生活中的样子、旅行照、才艺照等
   - 目标：确保图片合规、真实、可展示
5. `开放性文字审核`
   - 审核对象：关于我、希望 TA 了解等自由输入文本
   - 目标：过滤违规、低俗、广告、联系方式导流等内容

### 11.9.1 资料附加照片审核流程

1. 用户上传资料附加照片
2. 系统生成照片审核记录
3. 照片状态进入 `审核中`
4. 审核通过后，该照片可在个人主页展示
5. 审核驳回后，该照片不可展示，并返回驳回原因

### 11.9.2 开放性文字审核流程

1. 用户提交 `关于我`、`希望 TA 了解` 等文本
2. 系统生成文字审核记录
3. 文本状态进入 `审核中`
4. 审核通过后，对应文本可在个人主页展示
5. 审核驳回后，对应文本不可展示，并返回驳回原因

### 11.9.3 审核隔离要求

1. 主头像通过，不代表资料附加照片自动通过。
2. 资料附加照片通过，不代表开放性文字自动通过。
3. 某一类内容审核驳回，不影响其他类别已通过的状态。
4. 后台审核台需至少拆分为：
   - 资料照片审核
   - 开放性文字审核
5. 前端状态展示也需拆分，不允许只展示一个笼统的“资料审核中”。

## 13. 功能准入判定

## 13.1 觅缘推荐流准入

已确认首版条件：

1. 已登录
2. 已完成首登资料必填项
3. 实名认证通过
4. 账号状态正常

## 13.2 认证前浏览与交互限制

已确认规则如下：

1. `游客模式` 用户仅允许浏览公共信息，不允许进入真实 `觅缘卡片` 浏览态。
2. 已完成资料但未实名认证用户，仅允许浏览 `觅缘卡片`。
3. 认证前不允许执行 `喜欢/不喜欢/打招呼` 等按钮操作。
4. 认证前不进入真实曝光链路，包含：
   - 不进入别人的觅缘推荐流
   - 不进入搜索/筛选结果
   - 不进入喜欢我的
   - 不进入最近来访
5. 认证前不进入真实匹配链路，包含：
   - 喜欢关系不生效
   - 不形成互相喜欢
   - 不开放私信
6. `精选` 在 `未完成资料或未实名认证` 时，照片需以 `模糊态` 展示。
7. `未完成资料或未实名认证` 用户点击 `精选` 中的 `心动` 按钮时，弹出提示文案：
   - `认证后，再查看详细信息`
8. 未实名认证用户不开放 `理想型搜索`。
9. 认证前不开放社区动态发布。
10. 页面需持续提示：`完成实名认证后，才可曝光和匹配`。

## 13.3 认证后开放能力

实名认证通过后开放以下能力：

1. 进入别人推荐流，获得真实曝光。
2. `喜欢/不喜欢` 行为真实生效。
3. 可形成互相喜欢。
4. 开放 `喜欢我的`。
5. 开放 `最近来访`。
6. `精选` 页面照片以 `高清态` 展示，默认展示 `10` 个用户。
7. `精选` 页面底部展示购买服务入口，用于承接后续会员/成家币体系转化。
8. 开放 `理想型搜索`。
9. 开放社区动态发布。
10. 开放后续消息/私信等真实互动链路。

## 13.4 认证前仍可使用的基础能力

1. 微信授权登录
2. 首登资料完善
3. 编辑资料
4. 查看认证说明并发起实名认证
5. 浏览觅缘卡片
6. 浏览 `模糊态精选`
7. 查看协议/隐私文档

## 13.5 精选与理想型搜索的文档边界

1. 本文档只定义 `认证前后可见性、可点击性、开放条件`。
2. `精选` 底部购买服务入口仅在本文档中定义为产品边界和触发点。
3. 会员套餐、成家币消耗、支付流程、购买页、权益明细等，统一放到后续 `会员与成家币体系` 文档中详细展开。
4. `理想型搜索` 在本文档中仅定义是否开放，不展开搜索条件、解锁方式和支付规则细节。

## 14. 接口草案

## 14.1 微信授权登录

1. `POST /api/auth/wechat/login`
   说明：微信授权登录接口。若用户首次登录，服务端自动创建账号并返回登录态，不再区分“注册接口”和“登录接口”。

## 14.2 首登资料

1. `GET /api/user/profile/init`
   说明：返回首登轻量引导/强引导状态、游客模式状态、已填写字段及预填写数据。
2. `POST /api/user/profile/init`
   说明：分步保存首登轻量引导或强引导资料。
3. `POST /api/user/profile/init/skip`
   说明：记录跳过动作；若在强引导页跳过，则切换为游客模式。
4. `GET /api/user/profile/detail`
5. `PATCH /api/user/profile`
6. `POST /api/user/profile/avatar/upload`
7. `POST /api/user/profile/photos/upload`

## 14.3 认证

1. `GET /api/user/verify/status`
2. `POST /api/user/verify/real-name/submit`
   说明：提交姓名、身份证号后，由服务端调用微信 `getVerifyId` 创建核身会话，并返回 `verify_id` 给前端。
3. `POST /api/user/verify/real-name/result`
   说明：前端完成微信核身流程后，由服务端调用微信 `queryVerifyInfo` 查询最终结果，并更新本地实名认证状态。
4. `POST /api/user/verify/education/submit`
   说明：提交学历认证资料后，由服务端调用第三方学历认证服务商 API，认证结果异步回写。
5. `POST /api/user/verify/avatar/submit`
   说明：提交当前主头像后，由服务端创建头像认证申请，状态进入 `审核中`，等待后台人工审核。

## 14.4 资料内容审核

1. `POST /api/user/profile/photo/submit`
   说明：提交资料附加照片，生成照片审核记录。
2. `GET /api/user/profile/photo/audit-status`
   说明：查询资料附加照片审核状态与驳回原因。
3. `POST /api/user/profile/text/submit`
   说明：提交关于我、希望 TA 了解等开放性文本，生成文字审核记录。
4. `GET /api/user/profile/text/audit-status`
   说明：查询开放性文字审核状态与驳回原因。

## 16. 校验规则

1. 昵称：2-12 字，不允许纯表情，不允许敏感词。
2. 生日：年龄需满足平台允许区间。
3. 身份：必须选择 `在校生` 或 `职场人`。
4. 身高：140-220 cm。
5. 体重：30-200 kg。
6. 学校：长度限制 2-50 字；在校生必填。
7. 职业：职场人必填，优先使用职业字典；手动填写时限制 2-30 字。
8. 年收入：职场人必填，必须命中字典枚举。
9. 关于我：最少 20 字，最多 300 字。
10. 标签：至少 3 个，最多 10 个。
11. 主头像：必须上传 1 张。
12. 相册：首版建议最少 3 张，最多 9 张。
13. 身份证号：按国家规则校验格式。

## 17. 异常与边界场景

1. 用户注册成功但首登资料未完成：
   下次登录直接回到未完成步骤。
2. 用户在首登轻量引导中跳过：
   进入强引导页，已填写字段需反显。
3. 用户在强引导页再次跳过：
   进入游客模式，仅保留公共信息浏览权限。
4. 用户已完成资料，但认证中途退出：
   返回认证中心，保留已完成项。
5. 认证被驳回：
   页面需展示驳回原因并支持重新提交。
6. 用户修改已认证相关字段：
   若修改会影响认证结果，需提示“修改后可能需要重新审核”。
7. 用户更换主头像：
   需重置头像认证状态并重新进入 `审核中`。
8. 用户年龄不符合平台准入要求：
   拒绝注册或限制继续使用。
9. 学校未检索到：
   允许手输，但进入人工核验风险更高。
10. 第三方学历认证超时或无明确返回：
   前台保持 `审核中`，后台需支持人工跟进或补偿重试。
11. 头像认证人工审核长时间未处理：
    前台保持 `审核中`，后台需支持催审、转审或超时提醒。
12. 资料附加照片审核被驳回：
    仅影响对应照片展示，不影响主头像认证状态。
13. 开放性文字审核被驳回：
    仅影响对应文案展示，不影响照片审核或认证状态。
