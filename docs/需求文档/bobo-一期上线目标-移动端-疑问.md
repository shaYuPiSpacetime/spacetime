# 一期上线目标 - 移动端疑问点

> 口径：本文档只记录一期移动端上线前需要产品确认的疑问点，不展开完整页面清单。

## 产品确认疑问点

1. 双向喜欢才能开聊是否作为一期强规则？

   建议明确：普通私信必须建立在“相互喜欢”之后。单向喜欢、仅浏览主页、仅社区互动不应直接开启普通私信。若保留“悄悄话/小纸条”等破冰能力，需要单独定义次数、消耗、审核和举报规则，不能等同于普通私信。

2. 社区发私信是否会绕过“相互喜欢才能开聊”？

   当前一期范围里有社区打招呼、社区发私信、私信对话、相互喜欢等链路，需要产品确认：社区发私信是普通私信，还是一种受限破冰能力。如果是普通私信，建议同样受相互喜欢限制；如果是破冰能力，需要限制频次、内容类型和举报入口。

3. 聊天举报是否作为一期 P0 能力？

   建议保留。婚恋场景里，风险和骚扰很多发生在会话内，只做内容/用户举报不够。会话页应支持聊天举报，后台需要能看到举报人、被举报人、会话上下文、证据截图/消息片段、处理状态和处罚结果。

4. “举报 + 不再推荐/屏蔽此人”是否作为最小安全闭环？

   建议作为一期 P0。用户在用户主页、推荐卡、社区详情、会话页遇到不合适或风险对象时，除了举报，还应能立即“不再推荐/屏蔽此人”。完整黑名单列表、关键词屏蔽、动态屏蔽可以后置，但场景内切断联系和曝光的能力不建议缺失。

5. 拉黑、屏蔽、不再推荐三者是否需要区分？

   建议产品确认一期术语和效果：不再推荐偏推荐流负反馈，屏蔽此人偏双方不可见/不可联系，拉黑偏更强关系阻断。若一期不做完整黑名单管理页，也要明确场景内按钮点击后的实际效果。

6. 未相互喜欢时，用户主页上的“私信/打招呼/悄悄话”按钮如何展示？

   建议产品明确不同关系状态下的按钮：未喜欢、我喜欢 TA、TA 喜欢我、相互喜欢、已屏蔽、已被冻结。否则前端容易出现按钮可点但接口不允许，或用户误以为能直接聊天。

7. 悄悄话/小纸条是否允许发给未匹配用户？

   如果允许，需要确认是否消耗成家币/VIP 权益、每日次数上限、内容审核方式、对方是否可拒收、被举报后的权益是否退回。这个点会影响商业化，也会影响骚扰风险。

8. 取消喜欢、解除匹配是否一期要做？

   建议至少保留取消喜欢和解除匹配。相互喜欢不是永久关系，用户需要能主动退出；解除后是否保留历史会话、是否还能再次推荐，也需要产品确认。

9. 推荐卡负反馈是否只做“不喜欢”，还是要做“不再推荐此人”？

   建议区分。“不喜欢”可以作为普通推荐反馈，“不再推荐此人”应作为明确屏蔽该用户的安全/体验动作。两者对后续推荐的影响不同。

10. 被屏蔽/被举报后，双方关系和消息如何处理？

    需要确认：是否立即禁止发消息、是否从推荐流移除、是否隐藏历史会话、是否通知对方、是否影响相互喜欢关系。建议不通知被屏蔽方，避免引发二次骚扰。

11. 后台举报处理是否统一承接用户举报、内容举报、聊天举报？

    建议统一承接，但区分举报来源。后台至少要支持查看举报对象、举报类型、证据上下文、处理记录，并能执行冻结、下架内容、限制互动等处置。

12. 一期是否需要“安全兜底入口”？

    建议在用户主页、推荐卡、社区内容详情、会话页都能触达举报/屏蔽。设置页可以不做完整安全中心，但核心场景不能没有安全入口。

13. 付费权益和聊天规则是否冲突？

    需要确认 VIP 或成家币是否能突破“相互喜欢才能普通私信”。建议不要直接突破普通私信规则；如要做付费破冰，应作为悄悄话/小纸条独立权益，并有次数、审核和举报限制。

14. “喜欢我的”“最近看过我的”是否涉及隐私保护？

    需要确认普通用户和 VIP 用户分别能看到什么：头像是否模糊、昵称是否隐藏、能否直接进入主页、能否发起喜欢或破冰。这个影响会员价值，也影响用户隐私预期。

15. 一期是否需要最小验收标准？

    建议产品确认一条首发闭环验收标准：用户完成认证后能看到有效推荐；只有相互喜欢才能普通私信；任意关键场景可举报和屏蔽；后台能追踪举报、屏蔽、消息和处罚记录。
