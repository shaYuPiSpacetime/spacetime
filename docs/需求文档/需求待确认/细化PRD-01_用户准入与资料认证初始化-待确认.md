- 文档目的：标出 PRD-01 当前已覆盖内容、与设计稿或后台 PRD 不一致的内容、仍需产品确认的缺口。

## 2. 总体结论

当前移动端 PRD-01 已覆盖大部分主流程，包括微信授权登录、首登轻量引导、完整资料完善、头像认证、实名认证、学历认证、认证状态、游客模式、准入拦截、后台审核承接等。

但对照设计稿后，仍存在以下核心问题：

1. 部分流程口径与设计稿不一致，尤其是 `轻量引导是否允许跳过`、`登录协议弹窗形式`、`资料完善页是否可稍后再说`。
2. 准入门槛在不同文档之间存在冲突：移动端 PRD-01 写 `实名认证通过即可开放曝光和匹配`，但 PRD-04/PRD-05/PRD-06/PRD-07 多处写 `实名认证 + 头像认证 + 学历认证` 三项全部通过才开放核心互动或算成功邀请。
3. 后台 PRD 与移动端 PRD 对 `职业/年收入` 的首发范围存在冲突，后台文档仍写“职业字段在移动端 PRD-01 首版未纳入”，但移动端文档已明确职场人必填。
4. 设计稿展示了多种学历认证路径和学信网帮助页，但 PRD 对各路径的字段、适用条件、提交后状态、后台审核方式仍不够细。
5. 自我介绍、头像图片、资料附加照片、主头像认证、开放性文字审核之间的展示与审核边界已经提出，但对前端页面状态和失败重提链路描述仍不足。

## 4. 高优先级待确认问题

### 4.1 核心准入门槛存在跨文档冲突

当前移动端 PRD-01 的口径：

- 已完成首登资料但未完成实名认证：仅允许浏览觅缘卡片。
- 已完成首登资料且实名认证通过：开放曝光、匹配及相关互动能力。
- 学历认证、头像认证为增强信任项，不作为首版曝光与匹配前置门槛。

但其他需求文档出现了不同口径：

- PRD-04 写 `实名认证 + 头像认证 + 学历认证` 三项全部通过后，才开放真实曝光、匹配、关系反馈和真实聊天链路。
- PRD-05 写发布、评论、关注、@用户门槛必须三项认证全部通过。
- PRD-06 写完成三项认证后才能进入成家核心互动功能。
- PRD-07 写被邀请用户完成三项认证后才算成功邀请。

待确认：

1. 首版核心曝光和匹配到底是 `只要求实名认证`，还是 `三项认证全部通过`？
2. 不同功能是否分层：如浏览/曝光/喜欢/私信/社区发布/邀请成功分别采用不同门槛？
3. 后台 `核心准入状态` 应按哪个规则计算？

建议统一为一张准入矩阵，不要散落在多个 PRD 中各自定义。

### 4.2 登录协议交互与设计稿不一致

PRD 写法：

- 登录页有协议勾选。
- 用户必须勾选协议后，才可点击微信授权登录按钮。

设计稿表现：

- 登录页主按钮为 `立即使用`。
- 点击后弹出 `用户协议和隐私政策` 弹窗。
- 弹窗有 `不同意` 和 `同意`。
- 弹窗文案包含未成年人勿注册、个人信息收集、第三方共享、个人信息权利等说明。

待确认：

1. 登录页是否保留协议勾选，还是改为首次点击按钮后弹协议弹窗？
2. 用户不同意后，是停留登录页、关闭弹窗，还是退出小程序？
3. 协议弹窗是否仅首次展示？后续版本变更是否需要重新弹出？
4. 未成年人限制文案是否要求强制展示，是否与年龄准入规则联动？

### 4.3 首登轻量引导是否允许跳过

PRD 多处写：

- 除授权页外，轻量引导页均支持跳过。
- 跳过或填写部分字段后，仍保存已填内容并进入强引导页。

设计稿表现：

- 性别、年龄、身份、学历、地址页没有明显 `跳过` 按钮。
- 底部下一步按钮在未选择时呈禁用态，暗示字段必选。

待确认：

1. 性别、生日、身份、学历、常住地是否仍允许跳过？
2. 如果不允许跳过，应修改 PRD 中 `轻量引导支持跳过` 的描述。
3. 如果允许跳过，设计稿需要补跳过入口，并明确跳过后下一步按钮状态。
4. 性别和生日如果用于基础安全和推荐规则，建议不要允许跳过。

### 4.4 年龄准入规则不够明确

PRD 写了：

- 年龄需满足平台允许区间。
- 年龄范围可后台配置。
- 年龄不符合平台准入要求时，拒绝注册或限制继续使用。

但未明确：

1. 最小年龄是否为 18 周岁？
2. 最大年龄是否限制？
3. 年龄不符发生在生日选择页即时拦截，还是提交后拦截？
4. 未成年人在协议弹窗已提示不可注册，生日选择时是否二次提示？
5. 已注册用户后续因生日纠错导致年龄不符时如何处理？

建议补充为明确规则：按生日精确到日计算年龄，未满配置年龄时直接阻断后续流程，并展示固定提示文案。

### 4.5 后台 PRD 与移动端 PRD 字段范围冲突

移动端 PRD 当前已明确：

- `identityType` 首发必做。
- `occupation` 职场人必填。
- `annualIncomeRange` 职场人必填。
- 在校生不展示职业、年收入。

后台 PRD 仍写：

- `职业` 字段在移动端 PRD-01 首版未纳入，应按实际数据结构展示，避免先写死必填。

待确认：

1. 后台 PRD 是否需要同步改为 `职业/年收入为职场人首发必填字段`？
2. 用户列表和用户详情是否增加 `身份类型`、`职业`、`年收入` 字段？
3. 搜索条件是否需要支持身份、职业、收入筛选？

## 5. 移动端流程待补充问题

### 5.1 性别选择页文案和图标需确认

设计稿中性别页存在可疑点：

- `我是女生` 卡片右侧展示了疑似男性符号。
- `我是男生` 卡片也展示了相同或相近符号。

待确认：

1. 女生卡片和男生卡片的图标是否放反？
2. 性别值提交枚举是否为 `FEMALE/MALE`，前端展示是否需要防止映射错误？
3. 性别不可修改，因此该页是否需要二次确认或明确提示？

### 5.2 出生日期选择页需要补充选择器规则

设计稿为三列年月日选择器，PRD 仅写日期选择器。

待确认：

1. 年、月、日范围分别如何生成？
2. 闰年、大小月如何处理？
3. 默认选中日期是什么？是否允许默认为空？
4. 选择后展示为生日还是年龄？
5. 星座是否由生日实时计算并反显？

### 5.3 地址页需要明确定位和手动选择关系

设计稿地址页包含：

- `选择城市`
- `获取定位`

PRD 写常住地为省市区三级选择器，但没有展开定位授权。

待确认：

1. 点击 `获取定位` 后是否调用微信定位权限？
2. 定位失败、拒绝授权、城市无法识别时如何处理？
3. 定位结果是只填城市，还是填省/市/区？
4. 用户是否可手动覆盖定位结果？
5. 地址页是否要求精确到区县，还是城市即可？

### 5.4 强引导页与底部 Tab 的关系需明确

设计稿展示：

- 未认证用户进入 `成家` 首页后，有底部 Tab。
- 主页面有 `完善资料和认证，解锁更多专属权益` 引导。
- 点击 `立即完善` 后进入资料完善步骤页。

PRD 写法中强引导页和游客模式较强，但未完全说明是否能看到底部 Tab。

待确认：

1. 新用户完成轻量引导后，是直接进入带 Tab 的首页引导，还是独立强引导页？
2. `稍后再说` 后进入游客模式，是否仍显示完整底部 Tab？
3. 游客模式下各 Tab 可进入哪些页面，哪些必须拦截？
4. 首页引导是否可以关闭？关闭后再次出现频率是什么？

### 5.5 基本资料页字段与设计稿字段顺序需同步

设计稿基本资料页字段顺序：

- 昵称
- 性别
- 出生日期
- 学历
- 现居地
- 身高/体重
- 家乡
- 身份
- 职业

PRD 基础资料字段还包含学校、专业、脱单目标、感情状态、标签、希望 TA 了解、语音介绍等。

待确认：

1. 设计稿未展示的字段是否在本页隐藏、放其他页面、还是暂不实现？
2. 在校生的学校、专业放在哪里填写？
3. 脱单目标、感情状态是否首版必须有页面承载？
4. `关于我` 与 `自我介绍` 是否为同一字段 `aboutMe`？
5. `希望 TA 了解` 是否需要单独页面，还是后续资料编辑页再补？

### 5.6 身高/体重弹层校验需细化

设计稿中身高/体重为底部弹层输入，PRD 仅写范围。

待确认：

1. 身高和体重是否均必填？PRD 中体重为条件必填，但设计稿同屏展示。
2. 未填写体重时是否允许提交？
3. 输入单位是否固定为 cm/kg？
4. 是否允许小数体重？
5. 异常值提示文案是什么？

### 5.7 职业和年收入选项需要明确字典

设计稿显示：

- 职业弹层有 `产品经理`、`程序员`、`设计师` 等选项，并有 `手动填写`。
- 年收入选项为 `5-15`、`15-30`、`30-50` 等。

PRD 写由后台配置，但未给默认枚举。

待确认：

1. 首版默认职业字典有哪些？
2. 年收入单位是否为 `万/年`？
3. 年收入枚举是否包括 `50-100`、`100以上`、`暂不透露`？
4. 手动职业填写是否进入审核或敏感词过滤？

### 5.8 自我介绍审核与按钮状态需补充

设计稿显示：

- 空内容时 `下一步` 禁用。
- 输入框提示不少于 20 字。
- 有内容后按钮可用。

PRD 写关于我最少 20 字、最多 300 字，并走开放性文字审核。

待确认：

1. 自我介绍提交后是立即进入下一步，还是等待审核结果？
2. 审核中时个人主页是否展示该文案？
3. 审核驳回后回到哪个页面重写？
4. 敏感词命中是前端即时提示，还是提交后审核驳回？

## 7. 实名认证待补充问题

## 8. 学历认证待补充问题

### 8.1 学历认证入口和身份关系需确认

设计稿显示学历认证分多种路径：

- 在校学生
- 中国大陆
- 学信网验证编码
- 毕业证或者学位证书编号
- 上传毕业证或学位证书

PRD 写首版支持中国大陆用户三种方式：

- 学信网自助认证
- 在线验证码提交

待确认：

1. `在校学生` 是否是学历认证入口的第一层身份选择？
2. 在校生是否只能上传学生证，职场人是否走毕业证/学位证/学信网路径？
3. `中国大陆` 之外的港澳台/海外是否隐藏，还是展示但提示暂不支持？
4. 学生证上传审核是学历认证的一种方式，还是资料证明材料补充？

### 8.2 学信网验证编码流程需要细化

设计稿有完整帮助页：

- 打开学信网
- 选择在线验证报告
- 复制在线验证码

待确认：

1. 是否由小程序内 WebView 打开学信网？
2. `打开学信网` 按钮跳转方式是什么？
3. 在线验证码格式、长度、有效期如何校验？
4. 验证码提交后是否调用第三方服务商，还是后台人工审核？
5. 第三方失败时是否允许用户改走上传证书路径？

### 8.3 毕业证/学位证编号路径需要补充规则

待确认：

2. 学位证编号适用于哪些授予年份？设计稿里提示 2008 年 9 月 1 日获得学位。

### 8.4 上传证书材料路径需要明确

设计稿展示 `上传毕业证或学位证书`，字段包括学校、学历、上传证明材料。

## 13. 二次核对补充：登录与认证设计图问题归并

### 13.1 A 级：必须先拍板的流程/架构问题

#### A1. 实名认证到底走不走人脸核身

前文已指出设计稿和 PRD 不一致，但需要升级为 A 级架构决策。

冲突：

1. 设计图 `实名认证-身份` 只有 `真实姓名 + 证件号码 + 提交`，没有刷脸后续页。
2. 移动端 PRD 11.6 大量描述微信官方人脸核身，包括 `getVerifyId`、`queryVerifyInfo`、前端拉起刷脸、`verify_ret` 判定。

影响：

1. 前端是否需要接微信人脸核身组件。
2. 后端接口是一次提交，还是 `创建核身会话 -> 查询核身结果` 两步。
3. 后台审核详情展示的是第三方实名核验结果，还是微信人脸核身流水。

待拍板：

1. 首版实名认证是否做人脸核身？
2. 若不做人脸核身，PRD 11.6 应改为 `姓名 + 身份证号 -> 第三方/后台核验`。
3. 若做人脸核身，设计稿必须补刷脸流程、失败页、重新核身页。

#### A2. 学历认证依赖实名认证，PRD 漏写

设计图 `三重认证-顺序提示` 弹窗写明：提交实名认证后，学历认证才会被审核。

当前 PRD 缺失：

1. 认证状态机没有体现 `学历认证待审核` 依赖 `实名认证已提交/已通过`。
2. 学历认证提交规则没有写实名前置。
3. 后台学历审核列表没有写 `实名未完成时不进入审核` 或 `暂停审核` 状态。

待确认：

1. 学历认证是依赖 `实名认证已提交`，还是必须 `实名认证已通过`？
2. 用户能否先填写学历材料但后台暂不审核？
3. 后台应显示为 `待实名后审核`、`未满足前置条件`，还是不生成审核任务？

建议补入状态：

`学历材料已提交 -> 待实名前置 -> 学历审核中 -> 已通过/已拒绝`

#### A3. 轻量引导能否跳过必须重定

前文已覆盖该冲突，补充结论：这不是小交互问题，会影响 `游客模式`、`首登状态机`、`资料草稿保存`。

2. 未选择时底部箭头禁用。
3. 无跳过入口。

1. 除授权页外均支持跳过。
2. 跳过后进入强引导，再跳过进入游客模式。

待拍板：

1. 轻量引导是否全部必填？
2. 如果全部必填，删除轻量引导跳过规则，只保留强引导 `稍后再说`。
3. 如果允许跳过，设计稿必须补 `跳过` 入口和跳过后的字段缺失提示。

#### A4. 学历认证需要按在校生/毕业生分流

前文已提学历路径不足，但需明确结构性缺失。

设计图路径：

1. 在校学生：学校名称 + 学历 + 上传学生证材料，最多 4 张。
2. 毕业生/中国大陆：三选一：
   - 学信网验证编码，推荐。
   - 毕业证或学位证书编号，24 小时审核。
   - 上传毕业证或学位证书，24 小时审核。

1. 学信网自助认证。
2. 在线验证码提交。
