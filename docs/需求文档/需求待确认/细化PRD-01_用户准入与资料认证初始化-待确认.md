# PRD-01 用户准入与资料认证初始化 - 待确认清单

## 1. 文档说明

- 适用范围：
  - 移动端 `细化PRD-01_用户准入与资料认证初始化.md`
  - 管理后台 `管理后台细化PRD-01_用户准入与资料认证初始化.md`
  - 移动端登录、首登、资料完善、头像/实名/学历认证 UI
  - 管理后台用户管理 UI
- 本文目的：对前序问题清单进行去重、归类和优先级收敛，区分 `产品拍板` 与 `技术拍板`。
- 处理原则：
  - `产品拍板`：影响业务规则、页面形态、字段范围、准入口径、用户体验，由产品/业务方确认。
  - `技术拍板`：影响接口、状态机、存储、安全、权限、幂等、事件、上线策略，由技术负责人确认；若技术方案改变产品体验，需产品共同确认。

## 2. 总体结论

当前 PRD-01 已覆盖账号登录、首登资料、资料完善、三重认证、后台审核、准入判断等主线，但仍存在若干必须先拍板的问题。

最核心的阻塞点有 8 个：

1. 实名认证到底是否做人脸核身。
2. 首登轻量引导是否允许跳过。
3. 学历认证是否依赖实名认证，以及是否按在校生/毕业生分流。
4. 核心准入门槛到底是 `实名认证通过` 还是 `实名 + 头像 + 学历` 三项通过。
5. 首登强引导和老用户编辑资料页是否拆成两套页面模型。
6. 设计稿未承载的字段是否仍为本期必做。
7. 后台用户管理 UI 是否按设计稿卡片宫格实现，并删除红娘、统一成家币命名。
8. 敏感个人信息、认证材料、后台导出和权限边界如何处理。

## 3. P0 产品拍板

### P0-P01 实名认证方案

**问题**

设计稿实名认证页只有 `真实姓名 + 证件号码 + 提交`，没有刷脸流程；移动端 PRD 11.6 大篇幅描述微信官方人脸核身。

**需拍板**

1. 首版实名认证是否做人脸核身？
2. 如果不做人脸核身，是走第三方姓名+身份证核验，还是后台人工审核？
3. 如果做人脸核身，设计稿是否补刷脸流程、失败页、重试页？
4. 设计稿字段 `证件号码` 是否改成 `身份证号`，还是支持多证件类型？

**影响**

接口数量、前端能力、微信权限、后台审核详情、认证状态机、合规材料留存。

### P0-P02 首登轻量引导是否允许跳过

**问题**

PRD 多处写轻量引导可跳过；设计图性别、年龄、身份、学历、地址页没有跳过入口，未选择时按钮禁用。

**需拍板**

1. 性别、生日、身份、学历、现居地是否全部必填？
2. 是否删除轻量引导跳过逻辑，仅保留强引导 `稍后再说`？
3. 如果保留跳过，设计稿是否补跳过入口？

**影响**

游客模式、首登状态机、资料草稿保存、准入拦截。

### P0-P03 学历认证前置与认证分流

**问题**

设计图提示“提交实名认证后，你提交的学历认证才会被审核”，PRD 未写学历认证依赖实名。同时设计图学历认证区分在校生和毕业生，PRD 只写学信网/验证码/毕业证编号三种方式。

**需拍板**

1. 学历认证依赖 `实名认证已提交`，还是必须 `实名认证已通过`？
2. 是否允许用户先提交学历材料，但后台待实名完成后再审核？
3. 学历认证是否按 `在校生/毕业生` 分流？
4. 在校生是否走 `学校 + 学历 + 学生证上传`？
5. 毕业生是否走 `学信网验证码 / 证书编号 / 上传证书` 三选一？
6. 海外/港澳台学历首版是否隐藏、禁用，还是展示待支持？

**影响**

学历认证状态机、后台审核列表、材料字段、学历接口、测试用例。

### P0-P05 首登强引导与老用户编辑资料页拆分

**问题**

设计图首登强引导是首页 Tab 内的任务卡 + 四段 stepper：`基本资料 -> 添加头像 -> 自我介绍 -> 三重认证`。PRD 11.3 把它写成编辑资料总页，包含资料评分、照片、标签、语音介绍等多入口。

**需拍板**

1. 是否拆成 `首登强引导向导页` 和 `老用户编辑资料总页`？
2. 首登向导是否只按设计图四步实现？
3. 标签、相册、希望 TA 了解、语音介绍等是否放到老用户编辑资料页？

**影响**

页面路由、资料保存接口、进度条状态、UI 验收。

### P0-P06 本期字段范围与承载位

**问题**

PRD 将 `脱单目标、感情状态、希望 TA 了解、标签、相册、语音介绍、MBTI` 等列入首发字段，但设计图首登流程没有承载位。PRD 又在研发建议中写语音介绍可延后，内部冲突。

**需拍板**

1. 以下字段本期是否必须做页面承载？
   - 脱单目标
   - 感情状态
   - 希望 TA 了解
   - 标签
   - 相册
   - 语音介绍
   - MBTI/性格标签
2. 若不在首登强引导中出现，是否只做数据结构预留？
3. 资料完整度中这些字段的分值是否删除或重分配？
4. 后台用户详情是否展示这些字段的空态？

**影响**

数据库字段、前端页面、资料完整度、后台详情、验收范围。

### P0-P07 性别、异性匹配与身份修改

**问题**

设计图副标题多次出现“为你推荐匹配的异性”，PRD 未说明是否只支持异性匹配；性别首版不可改，但后台允许客服改性别。

**需拍板**

1. 产品是否只支持异性匹配？
2. 是否需要性取向字段或明确不支持同性匹配？
3. 性别提交后是否需要二次确认？
4. 后台客服修改性别后，是否触发推荐关系、曝光池、已有匹配关系重算？
5. 身份 `在校生/职场人` 后续是否允许修改，修改后学历认证是否失效？

**影响**

推荐算法、资料字段、后台权限、用户纠错流程。

### P0-P08 年龄与未成年人处理

**问题**

协议弹窗提示未成年人勿注册，但账号可能在生日采集前已创建。年龄范围可配置，但未明确最小年龄、最大年龄和不符合时处理。

**需拍板**

1. 最小年龄是否 18 周岁？
2. 最大年龄是否限制？
3. 年龄按哪个时区和规则计算？
4. 年龄页是否允许默认日期直接提交？
5. 未成年人已创建账号后如何处理：删除、冻结、注销中、保留但不可用？
6. 已采集资料是否删除？

**影响**

合规、账号状态机、生日选择器、后台未成年人拦截记录。

### P0-P09 手机号绑定

**问题**

后台用户管理展示手机号并支持手机号搜索，但移动端 PRD 未写手机号授权/绑定流程。

**需拍板**

1. 首版是否强制绑定手机号？
2. 手机号绑定在登录后、轻量引导前，还是资料完善中？
3. 用户拒绝手机号授权能否继续？
4. 后台手机号为空时如何展示和搜索？

**影响**

后台用户管理、客服识别、账号合并、登录安全、测试数据。

### P0-P10 后台用户管理 UI 口径

**问题**

后台设计图是卡片式客户管理 + 右侧画像详情；后台 PRD 是传统用户管理字段清单。设计图包含红娘、千寻币、牵线等旧口径。

**需拍板**

1. 页面名称用 `客户管理`、`App 用户管理`，还是 `用户准入`？
2. 用户列表是否按设计稿做卡片宫格，而不是表格？
3. `画像` 是否就是查看详情入口？
4. `红娘` 是否全部删除？
5. `千寻币` 是否全部统一为 `成家币`？
6. `牵线/快速牵线` 是否保留；若不做红娘体系，入口是否删除？
7. 卡片右上角 `头像审核` 是否可直接审核，还是跳转认证审核页？

**影响**

后台页面实现、菜单、权限、设计还原、测试用例。

### P0-P11 后台字段闭环

**问题**

后台图展示 `公司、择偶要求、VIP、成家币、性格标签`，但 PRD-01 移动端未完整采集或属于其他 PRD。

**需拍板**

1. 职场人首版是否采集 `公司`？
2. `择偶要求` 来源于哪个字段或模块？
3. VIP、成家币是否在 PRD-01 用户详情只读展示？
4. PRD-04 未上线时，资产模块隐藏还是空态？
5. 性格标签来源于用户标签、测评结果，还是算法画像？

**影响**

后台详情、移动端字段、跨 PRD 依赖、空态验收。

### P0-P12 敏感信息、导出与材料留存

**问题**

本模块涉及身份证、证书编号、学生证/毕业证/学位证照片、人脸核身结果、手机号、定位、收入、婚姻状况等敏感信息。

**需拍板**

1. 敏感材料保存多久？
2. 用户注销后如何删除认证材料？
3. 后台是否允许下载证件/学历材料原图？
4. 用户列表/详情导出允许哪些字段？
5. 身份证号、证书编号、手机号是否加密存储？
6. 查看完整敏感信息是否需要二次确认和理由？
7. 后台页面是否加水印？

**影响**

合规、安全、后台权限、数据库设计、操作审计。
