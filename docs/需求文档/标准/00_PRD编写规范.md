# PRD 编写规范（总纲）

> 本文件定义"一份合格 PRD 应该长什么样"。所有 0→1 模块的 PRD 必须遵循本规范。
> 配套模板：`05_PRD评审清单.md`、`10_模板_全局定义_共享层.md`、`11_模板_全局定义_端专属层.md`、`20_模板_模块PRD.md`、`30_模板_页面规格.md`。

---

## 0. 快速上手（5 分钟）

**第一步：先建项目级全局定义。** 打开 `10_模板_全局定义_共享层.md`（跨模块术语+枚举+状态机，全项目唯一）和 `11_模板_全局定义_端专属层.md`（管理后台/移动端的全端共性基线，每端一份）。别跳过——这是项目级"单一事实源"，后面所有页面文档只引用不复述。

**第二步：建模块 PRD。** 打开 `20_模板_模块PRD.md`，定范围、边界（本期不做）、依赖、模块专属定义、模块端内定义、页面清单。别上来就写页面细节，先把方向对齐。

**第三步：逐页写规格。** 每个页面一份 `30_模板_页面规格.md`，按骨架填：字段表（10列）→ 操作表（8列）→ 状态穷举 → 验收标准。

**第四步：对照评审清单自检。** 打开 `05_PRD评审清单.md`，逐条打勾。

**铁律：字段只写名字 = 不合格。状态只写 happy path = 不合格。"待确认"遗留到定稿后 = 不合格。**

## 1. 核心原则

PRD 不是"功能描述"，是"决策集合"。

凡是能被合理追问"到底是哪一个？"的点，都必须在 PRD 里拍死。散文能藏歧义，AI 和 UI 都不会脑补：没写死的地方，AI 会随机猜一个，UI 会按自己理解画一版，分歧最终全堆到开发期爆炸。

> **反面案例（来自本项目）：**
> - "实名通过即可曝光" vs "三项全过才行" → 同一事实在两个 PRD 里各抄一遍，抄出矛盾
> - "首登轻量引导可跳过" vs 设计稿无跳过按钮 → 文字和设计各说各话
> - "支持学信网/验证码/毕业证编号三种方式" vs 设计图区分在校生/毕业生 → 流程分支没写死

## 2. 一份合格 PRD 的 12 条硬要求

| # | 要求 | 含义 | 不满足时的后果 | 本项目真实教训 |
|---|------|------|----------------|---------------|
| 1 | AI 可识别可实现 | 字段/操作/状态全部写死，无需脑补 | AI 随机猜，产出与预期不符 | 实名认证是否含人脸核身 → 接口数量、状态机、前端全部待定 |
| 2 | 无歧义、无内部矛盾 | 同一事实只在一处定义，他处引用 | 多文档各抄一遍，越抄越乱 | 准入门槛在 PRD-01/04/05/06/07 写了 5 个版本 |
| 3 | 页面级颗粒度 | 菜单→页面→按钮→字段→状态层层落地 | "不知道有几个菜单几个按钮" | 后台用户管理：卡片 vs 表格、红娘删不删全待确认 |
| 4 | UI 可直接出稿 | 布局 + 字段表 + 全状态齐全 | UI 反复回来追问 | 首登向导 vs 编辑资料页未拆分，UI 无从下手 |
| 5 | 可反向生成说明书 | 操作表 + 验收标准覆盖用户视角 | 说明书要另起炉灶 | — |
| 6 | 可验收 | 每个功能配 Given/When/Then 验收标准 | QA 无据可依，"完成"无定义 | — |
| 7 | 可追溯 | 需求/页面/字段/规则都有稳定 ID | 文档漂移无法被发现 | — |
| 8 | 单一事实源 | 跨模块定义抽全局，模块内复用定义抽模块公共定义，页面只引用 | 千寻币/成家币反复打架 | 红娘/千寻币/牵线口径在每个 PRD 里各写一遍 |
| 9 | 边界明确 | 单列"本期不做"及处理方式（隐藏/置灰/空态/仅预留结构） | 满屏"首版是否做 X" | 待确认清单 P0 项一半是"首版是否做" |
| 10 | 非功能性需求 | 权限、安全合规、性能、并发幂等、埋点 | 开发期集中爆炸 | P0-T07~T12 全部是非功能性缺失 |
| 11 | 可变更 | 版本号 + 变更记录；拍板后回写正文，删除待确认条目 | 待确认文档变成第二事实源 | 580 行待确认清单本身就是上一版 PRD 的缺陷日志 |
| 12 | PRD 间一致性 | 跨 PRD 的口径、状态、字段对齐，矛盾在定稿前发现 | 不同模块按不同假设开发，联调时撞车 | PRD-01 写"实名通过即可"，PRD-04 写"三项全过才行" |

## 3. 文档分层结构

写 PRD 不要写一坨大模块，分两层 + 一个入口，物理上消除矛盾。

```
某模块 PRD
├── 入口：模块 PRD（20_模板）—— 目标、范围、边界、依赖、模块专属定义、页面清单
│
├── 第一层：定义层（单一事实源，分三类）
│   ├── 共享层（10_模板）—— 全项目唯一一份，跨 2 个及以上业务模块共用
│   │   ├── 项目术语表 + 命名口径
│   │   ├── 跨模块数据字典 & 枚举
│   │   ├── 跨模块状态机（含流转表 + 触发条件）
│   │   ├── 跨模块业务规则汇总
│   │   ├── 第三方服务与依赖清单
│   │   ├── 配置项清单 + 通知与消息模板
│   │   └── 错误码表 + 接口约定
│   ├── 模块公共定义（20_模板 第 9 节或模块根目录公共定义）—— 单模块内跨端/跨页面复用
│   │   ├── 模块术语、枚举、状态机、规则
│   │   ├── 模块配置、通知、事件、错误码
│   │   └── 模块接口草案与公共约定
│   └── 全局端专属层（11_模板）—— 后台一份、移动端一份，只放全端共性
│       ├── 全端信息架构原则/导航基线
│       ├── 全端角色/登录态/权限控制原则
│       └── 全端 UI 通用空态/错误态/降级态 + 跨模块端文案 key
│
├── 模块端内定义（可独立文件或写在 20_模板 第 9 节）—— 单模块在某一端内复用
│   ├── 单模块菜单/页面树 + 页面 ID
│   ├── 单模块权限矩阵 + 敏感操作审计
│   └── 单模块端内 UI 状态 + 仅本端复用的端内文案
│
└── 第二层：页面规格（每个页面一份，固定骨架）
    └── 路由/入口 → 页面定位 → 布局 → 字段表 → 操作表 → 数据联动 → 状态与异常 → 验收 → 关联
```

页面规格里凡涉及枚举、状态、规则、配置、错误码、事件，一律**引用全局定义或模块公共定义的 ID，不复述**；涉及权限引用全局端专属层或模块端内定义；模块文案引用模块公共定义。
复述一遍就会错一遍——这是文档漂移的头号根源。
**共享层只放跨模块共性：** 同一模块内后台和移动端共用的定义放模块公共定义，不塞进共享层；真正跨 PRD 复用时再提升为 `GLB-*`。
**端专属层只放全端共性：** 单模块的菜单树、页面树、权限矩阵、端内 UI 状态放模块端内定义，不塞进全局端专属层；跨端或多页面复用的 `Mxx-TXT-*` 模块文案放模块公共定义。

## 4. ID 命名规范（可追溯的基石）

所有需求、页面、字段、操作、规则都挂稳定 ID。设计稿、代码、测试用例、Bug 全部引用 ID。

### 4.1 端前缀

| 前缀 | 含义 |
|------|------|
| `ADM` | 管理后台 |
| `APP` | 移动端小程序 |
| `GLB` | 项目级全局定义（跨 2 个及以上业务模块共享） |
| `Mxx` | 模块公共定义（如 `M07`，仅该模块内跨端/跨页面复用） |

### 4.2 模块号分配

| 模块号 | 模块名 | 对应 PRD |
|--------|--------|----------|
| 01 | 用户准入与资料认证 | PRD-01 |
| 02 | 关系反馈与互动链路 | PRD-02 |
| 03 | 消息、私信与通知中心 | PRD-03 |
| 04 | 商业化（VIP、成家币、资产） | PRD-04 |
| 05 | 推荐模块（朋友、社区与内容） | PRD-05 |
| 06 | 认证与安全设置、我的页 | PRD-06 |
| 07 | 推广裂变与邀请奖励 | PRD-07 |
| 08 | 推荐算法配置 | PRD-08 |
| 09 | 量表与测评 | PRD-09 |

> 模块号一经分配不可更改。新增模块顺延编号。

### 4.3 ID 结构

```
<端>-<模块号>-<对象类型>-<对象标识>[-<子项>]

示例：
ADM-01-PAGE-user-list              后台 PRD-01 的"用户列表"页面
ADM-01-PAGE-user-list-FIELD-phone  该页面的"手机号"字段
ADM-01-PAGE-user-list-ACT-freeze   该页面的"冻结"操作
ADM-01-PAGE-user-list-AC-freeze    该页面的"冻结成功"验收标准
GLB-ENUM-edu-level                 全局"学历"枚举
GLB-ENUM-auth-status               全局"认证状态"枚举
GLB-SM-auth                        全局"认证状态机"
GLB-SM-first-login                 全局"首登状态机"
GLB-ROLE-matrix                    全局权限矩阵
GLB-RULE-exposure-gate             全局"曝光准入门槛"规则
GLB-ERR-1001                       错误码 1001
GLB-TXT-login-protocol             全局文案-登录协议
M07-ENUM-invite-reward-event       PRD-07 模块内"邀请奖励事件"枚举
M07-SM-invite-reward               PRD-07 模块内"邀请奖励状态机"
M07-RULE-invite-attribution        PRD-07 模块内"渠道归属优先级"规则
APP-01-FLOW-first-login            移动端首登流程
APP-01-PAGE-profile-edit           移动端编辑资料页
```

### 4.4 对象类型枚举

| 类型 | 含义 | 示例 |
|------|------|------|
| `PAGE` | 页面 | `ADM-01-PAGE-user-list` |
| `FLOW` | 流程/向导 | `APP-01-FLOW-first-login` |
| `FIELD` | 字段 | `ADM-01-PAGE-user-list-FIELD-phone` |
| `ACT` | 操作/按钮 | `ADM-01-PAGE-user-list-ACT-freeze` |
| `STATE` | 状态 | `GLB-STATE-auth-rejected` / `M07-STATE-xxx` |
| `RULE` | 业务规则 | `GLB-RULE-exposure-gate` / `M07-RULE-invite-attribution` |
| `ENUM` | 枚举 | `GLB-ENUM-edu-level` / `M07-ENUM-invite-reward-event` |
| `SM` | 状态机 | `GLB-SM-auth` / `M07-SM-invite-reward` |
| `ROLE` | 角色/权限项 | `GLB-ROLE-auditor` |
| `ERR` | 错误码 | `GLB-ERR-1001` / `M07-ERR-7001` |
| `AC` | 验收标准 | `ADM-01-AC-freeze-success` |
| `TXT` | 文案/提示 | `GLB-TXT-login-protocol` / `M07-TXT-invite-home-title` |
| `CFG` | 后台配置项 | `GLB-CFG-age-min` / `M07-CFG-invite-ladder` |
| `EVT` | 事件/通知 | `GLB-EVT-auth-passed` / `M07-EVT-invite-bind` |
| `SRV` | 第三方服务 | `GLB-SRV-wechat-face` / `M07-SRV-wx-qrcode` |

### 4.5 规则

1. ID 一经发布**不可改、不可复用**。废弃的需求标记 `[已废弃]`，但 ID 保留占位。
2. 字段/操作 ID 用业务语义命名（`phone`），不用序号（`field-03`），便于阅读和搜索。
3. ID 全小写 kebab-case，层级用 `-` 连接。
4. 验收标准 ID 放在对应页面下（`ADM-01-AC-xxx`），不单独成体系。

## 5. 字段表标准（页面规格内）

字段口径只写名字（"手机号、性别、年龄……"列一串）= 字段级空洞。每个字段必须一行写全 10 列：

| # | 列 | 说明 | 必填 |
|----|----|------|------|
| 1 | 字段 ID | `...-FIELD-xxx` | ✅ |
| 2 | 显示名 | 界面展示文案 | ✅ |
| 3 | 类型 | string / int / enum / date / datetime / image / file / bool / json | ✅ |
| 4 | 必填 | 是 / 否 / 条件必填（注明条件） | ✅ |
| 5 | 取值范围 | 枚举引用 `Mxx-ENUM-xxx` / `GLB-ENUM-xxx`，或长度/数值范围/正则 | ✅ |
| 6 | 校验规则 | 格式、上下限、唯一性、前后端校验分工 | ✅ |
| 7 | 默认值 | 无则写"无" | ✅ |
| 8 | 可编辑 | 谁（角色）、何时（状态）、改后副作用（是否触发重新审核、事件等） | ✅ |
| 9 | 敏感级别 | 普通 / 敏感。敏感需注明：脱敏规则（如 138****1234）、是否加密存储、留存时长、是否可导出、注销后处理 | ✅ |
| 10 | 数据来源 | 用户填写 / 系统计算 / 第三方接口（注明服务）/ 其他模块 | ✅ |

> **反面案例（本项目）：** "标签、相册、语音介绍、MBTI" 列在 PRD 字段清单里，但设计稿首登流程没有承载位，也没写默认值和处理方式 → P0-P06 整条待确认。

## 6. 操作表标准（页面规格内）

每个按钮/操作一行写全 8 列：

| # | 列 | 说明 |
|---|----|------|
| 1 | 操作 ID | `...-ACT-xxx` |
| 2 | 操作名 | 按钮/链接文案 |
| 3 | 触发条件 | 什么状态下可见、可点击（引用状态机状态） |
| 4 | 前置权限 | 引用 `GLB-ROLE-xxx` |
| 5 | 二次确认 | 是否需要弹窗确认、确认文案 |
| 6 | 成功态 | 成功后的反馈 + 状态流转（引用 `Mxx-SM-xxx` / `GLB-SM-xxx`） |
| 7 | 失败态 | 失败反馈 + 错误码（引用 `Mxx-ERR-xxx` / `GLB-ERR-xxx`） |
| 8 | 影响（副作用） | 触发的事件（引用 `Mxx-EVT-xxx` / `GLB-EVT-xxx`）、对其他对象/页面的影响、是否需要刷新列表 |

## 7. 状态与异常标准（页面规格内）

不要只写 happy path。每个页面必须穷举 6 类状态，每类说明：触发条件 → 页面表现 → 用户可做什么：

| 状态类型 | 必填 | 说明 |
|----------|------|------|
| 加载态 | ✅ | 骨架屏/loading/spinner |
| 空态 | ✅ | 无数据时的插画 + 文案 + 操作引导 |
| 错误态 | ✅ | 请求失败/超时，含重试入口 |
| 无权限态 | ✅ | 无权限时的表现（隐藏整页/置灰按钮/提示无权限） |
| 业务态 | ✅ | 按对象状态机逐一覆盖，每个状态对应不同的页面表现 |
| 降级态 | ✅ | 第三方不可用、字典加载失败、定位失败等的兜底方案 |

> **反面案例（本项目）：** PRD 只写了"提交实名认证 → 审核通过/驳回"两个状态，但实际有"第三方处理中、人工跟进中、已失效、已撤销"等至少 6 个状态没定义 → P0-T03。

## 8. 验收标准标准（页面规格内）

每个核心功能配至少一条 Given/When/Then。这一条同时喂饱 QA（转测试用例）、AI（"完成"的定义）、说明书（用户视角改写）。

```
AC-ID: ADM-01-AC-freeze-success
Given 一个状态为"正常"的用户，当前管理员具备冻结权限（GLB-ROLE-admin-freeze）
When  点击"冻结"按钮，在二次确认弹窗中点击"确认冻结"
Then  用户状态变为"已冻结"（GLB-SM-user → frozen），该用户 token 立即失效，
      列表状态胶囊更新为"已冻结"，操作日志记录操作人和时间
```

验收标准编写规则：
- **Given** 写前置条件（数据状态 + 角色权限 + 页面状态），引用全局/模块/端专属 ID
- **When** 写具体操作步骤
- **Then** 写可观测的结果（UI 变化 + 数据变化 + 副作用），引用状态机/错误码 ID
- 不只写成功路径，异常路径也要有验收标准（如"无权限时点击冻结"）

## 9. 常见反模式（避坑指南）

以下反模式全部来自本项目真实踩坑记录：

| 反模式 | 表现 | 正确做法 |
|--------|------|----------|
| **散文式叙述** | "系统支持多种认证方式，用户可根据情况灵活选择" | 写死：哪几种、每种的前置条件、互斥还是并行、失败后能否切换 |
| **只写字段名** | "用户列表展示：手机号、性别、年龄、认证状态" | 每个字段按 10 列表格写全，含类型/必填/枚举/敏感级别/来源 |
| **隐藏的状态遗漏** | 只写"提交→审核→通过/驳回" | 补充：审核中用户能否修改、驳回后能否重新提交、超时未审核怎么办、第三方处理中等 |
| **跨PRD复述** | PRD-01 和 PRD-04 各写一遍准入门槛 | 跨模块定义 `GLB-RULE-exposure-gate`，各 PRD 只引用；单模块内复用定义用 `Mxx-*` |
| **边界模糊** | "语音介绍可延后"、"标签首版可考虑" | 进"本期不做"清单，写明处理方式：隐藏/置灰/空态/仅预留结构 |
| **权限一句话带过** | "管理员可查看，审核员可审核" | 角色×操作矩阵，每格填"允许/禁止/脱敏/本人仅" |
| **"待确认"不闭环** | 定稿后仍保留大量"待确认"项 | 拍板后立即回写正文、删除待确认条目；若实在无法拍板，标记 `[阻塞]` + 负责人 + 截止日期 |
