# PRD 评审清单

> PRD 定稿前，由产品负责人和技术负责人分别逐项检查。
> 评审通过 = 所有 P0 项 ✅ + P1 项 80% 以上 ✅。
> 评审结果记录在 PRD 文档头的变更记录中。

---

## A. 产品维度（产品负责人评审）

### A1. 范围与边界

| # | 检查项 | P |
|---|--------|---|
| A1.1 | 模块目标用一句话能说清楚（解决什么问题、对谁有价值） | P0 |
| A1.2 | 本期要做能力的清单完整，每条有需求 ID + 优先级 | P0 |
| A1.3 | "本期不做"清单完整，每项有处理方式（隐藏/置灰/空态/仅预留结构） | P0 |
| A1.4 | 没有"首版是否做 X"未决策的遗留项 | P0 |
| A1.5 | 跨模块依赖已声明，依赖未就绪的兜底方案可接受 | P0 |
| A1.6 | 改动影响面已评估（影响哪些已有页面/功能/数据） | P1 |

### A2. 信息架构与页面

| # | 检查项 | P |
|---|--------|---|
| A2.1 | 所有页面都在信息架构树里，有页面 ID + 一句话职责 | P0 |
| A2.2 | 每个页面有路由/入口，能说清"从哪来、到哪去" | P0 |
| A2.3 | 关键业务流程有文字或时序图描述，分支和异常路径完整 | P0 |
| A2.4 | 布局描述足够 UI 直接出稿（区块结构 + 层级 + 弹层） | P0 |
| A2.5 | 页面之间的跳转关系清晰（A 页的某个操作跳到 B 页的哪个状态） | P1 |

### A3. 字段与操作

| # | 检查项 | P |
|---|--------|---|
| A3.1 | 每个字段都填全了字段表 10 列，无"只写名字" | P0 |
| A3.2 | 每个操作都填全了操作表 8 列，含失败态和权限 | P0 |
| A3.3 | 操作触发条件引用了状态机状态（不是模糊的"满足条件时"） | P0 |
| A3.4 | 二次确认的操作已明确文案和取消后的行为 | P1 |
| A3.5 | 字段联动规则已穷举（选 A 影响 B 的展示/可选值） | P0 |
| A3.6 | 敏感字段已标注并写明脱敏/加密/留存/导出策略 | P0 |

### A4. 状态与异常

| # | 检查项 | P |
|---|--------|---|
| A4.1 | 每个对象有状态机（状态清单 + 流转表 + 触发条件 + 副作用） | P0 |
| A4.2 | 状态机覆盖了异常分支，不只 happy path | P0 |
| A4.3 | 每个页面穷举了 6 类状态（加载/空/错误/无权限/业务/降级） | P0 |
| A4.4 | 空态有文案 + 操作引导 | P1 |
| A4.5 | 降级态有兜底方案（第三方不可用、字典加载失败等） | P0 |
| A4.6 | 错误提示文案已明确（用户能看懂，不是技术术语） | P1 |

### A5. 验收与质量

| # | 检查项 | P |
|---|--------|---|
| A5.1 | 每个核心功能有 Given/When/Then 验收标准 | P0 |
| A5.2 | 验收标准含异常路径（不只成功路径） | P0 |
| A5.3 | 验收标准引用了全局 ID（状态机/错误码），可直接转为测试用例 | P1 |
| A5.4 | 可反向生成产品使用说明书（操作表 + 验收覆盖用户视角） | P1 |

## C. 一致性维度（两人共同评审）

### C1. PRD 内部一致性

| # | 检查项 | P |
|---|--------|---|
| C1.1 | 同一事实不在多个 PRD 中重复定义（引用全局定义而非复述） | P0 |
| C1.2 | 页面规格中枚举/状态/权限引用指向全局定义，无硬编码值 | P0 |
| C1.3 | 术语统一：全项目使用术语表中定义的标准术语，无旧称/别名 | P0 |
| C1.4 | PRD 文档间没有矛盾（如 PRD-01 和 PRD-04 对同一规则描述不同） | P0 |

### C2. PRD ↔ 设计稿一致性

| # | 检查项 | P |
|---|--------|---|
| C2.1 | PRD 中的所有字段在设计稿中有承载位 | P0 |
| C2.2 | PRD 中的所有操作在设计稿中有对应按钮/入口 | P0 |
| C2.3 | 设计稿中的每个元素在 PRD 中有对应描述（反向校验，防止设计稿多画） | P1 |
| C2.4 | PRD 状态表中定义的状态在设计稿中有对应页面/区块 | P0 |
| C2.5 | 术语口径在 PRD 和设计稿中一致 | P0 |

### C3. PRD ↔ 定义层一致性

| # | 检查项 | P |
|---|--------|---|
| C3.1 | 全局共享层或模块公共定义中的枚举值覆盖了所有 PRD 页面的使用场景 | P0 |
| C3.2 | 全局共享层或模块公共定义中的状态机覆盖了所有 PRD 页面的状态引用 | P0 |
| C3.3 | 全局端专属层或模块端内定义中的权限口径覆盖了所有 PRD 页面的操作权限 | P0 |
| C3.4 | 全局共享层或模块公共定义中的第三方服务清单覆盖了所有 PRD 引用的外部服务 | P1 |
| C3.5 | 共享层、全局端专属层、模块公共定义、模块端内定义之间无重复定义；页面只引用不复述 | P0 |

### C4. PRD ↔ 技术方案一致性（建议项）

> 你们 PRD / 技术方案(tcdesign) / 测试 / 前端四套文档并行，PRD↔技术方案最易漂移。
> 技术方案通常在 PRD 之后产出，故此维度为**建议项**：技术方案定稿时回头对一遍。

| # | 检查项 | P |
|---|--------|---|
| C4.1 | 技术方案的接口前缀/路径与全局定义第 9 节一致 | P1 |
| C4.2 | 技术方案的状态机/状态枚举与全局定义一致，无私自新增状态 | P1 |
| C4.3 | 技术方案的数据模型字段覆盖了 PRD 字段表的全部字段 | P1 |
| C4.4 | 技术方案的错误码与全局错误码表一致 | P1 |
