## 1. 模块目标

管理后台 PRD-01 承接移动端用户准入、资料与认证审核的运营能力，核心目标是让审核员、客服、运营在一个可控、可审计的后台中完成 App 用户查看、认证审核、资料内容审核、准入配置，并通过系统字典管理维护所需字典。

**后台目标：** App 用户管理采用卡片宫格 + 右侧画像抽屉，并支持上传 Excel 批量导入线下收集的 App 用户资料；实名、头像、学历认证独立审核；资料图片和开放性文字独立审核；审核页需区分机审/人工审核来源，并支持人工复核机审结论；敏感字段按权限脱敏，原图下载与完整字段查看必须审计。

**核心指标：** 待审核量、审核通过率、驳回率、学历 24 小时内处理率、核心准入开放率、敏感字段查看次数。

## 3. 关键业务流程

### 3.1 App 用户管理流程

```
进入 App 用户管理
-> 按搜索/筛选查看卡片宫格
-> 可下载导入模板并上传 Excel 批量导入线下用户信息
-> 点击用户卡片打开右侧画像抽屉
-> 查看基础资料、认证状态、准入状态、千寻币/VIP摘要
-> 需要处理头像认证时点击“头像审核”
-> 跳转到头像认证审核页对应记录
```

### 3.1.1 App 用户 Excel 批量导入流程

```
进入 App 用户管理
-> 点击“批量导入”
-> 下载标准 Excel 模板
-> 按模板填写线下收集的手机号、实名、轻量资料、基础资料、扩展资料等字段
-> 上传 Excel
-> 系统预校验字段格式、必填项、字典值、手机号/身份证号重复
-> 展示校验结果：总行数、可导入行数、失败行数、失败原因文件
-> 管理员确认导入
-> 成功行创建 App 用户资料；失败行不写入
-> 生成导入批次、审计日志和结果报告
```

### 3.2 认证审核流程

```
用户提交实名/头像/学历材料
-> 第三方 API/机审服务先给出通过、驳回或需人工处理结论
-> 机审通过：写入大状态已通过、来源机审，业务上等同审核通过，并在后台审核页可被人工复核
-> 需人工处理或机审驳回需复核：后台认证审核页生成待处理记录
-> 审核员查看材料、机审结果与审核历史
-> 人工通过：状态变更为已通过、来源人工审核，站内信通知用户，重新计算核心准入
-> 人工驳回：状态变更为已驳回、来源人工审核，自由输入驳回原因，站内信原文告知用户并引导联系客服
-> 当前列表实时刷新，已处理记录移出待审核筛选结果
```

### 3.3 配置与字典流程

```
管理员进入准入与认证配置
-> 修改年龄、资料完整度、准入门槛、上传限制、审核 SLA、文案等
-> 高风险配置二次确认
-> 记录变更前后值与操作人
-> 如需维护学历、职业、地区等枚举项，进入系统管理 / 字典管理
-> 新增数据按新配置和系统字典启用项生效，历史数据按冗余值展示
```

## 6. 数据与状态要求

后台用户资料、认证材料、字段显示名、字段来源与字段配置统一以 `../../PRD-01_模块公共定义.md` 的 `M01-DATA-user-input-fields` 为准。后台页面可以按运营场景组合摘要、筛选、脱敏、导出和审核字段，但不另行定义一套用户字段全集。

App 用户 Excel 批量导入模板以本文件 `ADM-01-DATA-user-import-fields` 为准。导入字段必须从 `../../PRD-01_模块公共定义.md` 的 `M01-DATA-user-input-fields` 中选取，只承接可线下采集和后台录入的用户信息，不包含短信验证码、微信授权、协议勾选、图片/音频材料、定位经纬度和系统状态字段。导入不绕过认证与核心准入：导入成功后实名认证进入审核中，头像/学历认证默认未提交，核心准入仍按三重认证通过计算。

实名、头像、学历、资料图片、开放性文字审核记录必须同时保存大状态 `auditStatus` 与审核来源 `auditSource`。大状态引用 `../../PRD-01_模块公共定义.md` 的 `M01-ENUM-verification-status` 或同口径内容审核状态；审核来源引用 `M01-ENUM-audit-source`；后台展示标签引用 `M01-ENUM-audit-result-display`。核心准入、资料展示、图片/文本可见性等业务逻辑只判断大状态是否 `approved`，不因机审/人工来源不同而产生差异。后台审核列表、详情、历史和审计必须展示来源，并允许有权限人员对机审通过或机审驳回记录发起人工复核，人工结论覆盖当前状态但保留完整历史。

| 系统状态 | 首登状态、核心准入状态、资料完整度、各类认证/审核状态、用户状态 | 由系统计算、审核或账号操作生成，不允许 Excel 直接写入 |

导入结果口径：

1. 导入只创建新的 App 用户资料，不覆盖已有用户；手机号或身份证号命中存量用户时，该行失败并在错误报告中提示。
2. 导入时不发送短信、不校验 `smsCode`；用户后续登录仍需按登录规则完成微信授权或手机号验证码校验。
3. 导入成功后，手机号、真实姓名、身份证号进入后台资料记录；实名认证状态默认为审核中，不自动通过；头像、学历认证状态默认未提交，除非后续通过对应认证流程提交材料。
4. 资料完整度、年龄、星座、首登状态、核心准入状态由系统按导入字段重新计算；核心准入仍必须满足三重认证通过且账号正常。
5. 每次导入必须生成批次号、导入人、导入时间、成功/失败数量、失败原因文件，并记录审计。

| 状态对象 | 后台可见字段 | 后台可操作 |
|----------|--------------|------------|
| 首登状态 | 未开始、轻量引导中、强引导中、已完成 | 查看、筛选 |
| 核心准入 | 资料未完成、认证未完成、已开放 | 查看、筛选 |
| 实名认证 | 未提交、审核中、已通过、已驳回 | 通过、驳回、查看历史 |
| 头像认证 | 未提交、审核中、已通过、已驳回、已失效 | 通过、驳回、查看大图和历史 |
| 学历认证 | 未提交、审核中、已通过、已驳回、已失效 | 通过、驳回、查看材料 |
| 资料图片 | 未提交、审核中、已通过、已驳回 | 通过、驳回 |
| 开放文字 | 未提交、审核中、已通过、已驳回 | 通过、驳回 |

状态隔离原则：

1. 实名、头像、学历、资料图片、开放性文字五类审核状态独立。
2. 主头像通过不代表资料图片通过。
3. 资料图片通过不代表开放性文字通过。
4. 任一认证通过/驳回都要记录审核历史与操作人。

## 7. 权限、审计与脱敏

| 操作 | 权限要求 | 审计要求 | 二次确认 |
|------|----------|----------|----------|
| 查看完整手机号/姓名/身份证号 | 高敏字段权限 | 记录操作人、字段、用户、理由、时间 | 是 |

| 冻结/解冻、启用/停用 | 客服/风控权限 | 记录目标用户、动作、原因 | 是 |
| Excel 批量导入 App 用户 | 导入权限 | 记录批次号、导入人、文件名、总行数、成功/失败数量、失败原因文件 | 是 |
| 人工复核机审结论 | 对应审核权限 | 记录复核对象、机审原结论、人工新结论、原因、操作人、时间 | 是 |
| 修改配置/系统字典 | 管理员权限 | 记录变更前后值 | 是 |
| 审核通过/驳回 | 审核权限 | 记录对象、结果、驳回原因 | 是 |

导出字段固定为：用户姓名、用户昵称、用户身份证号、性别、出生日期、身份、婚姻状况、个人标签、资料完整度、首登引导状态、核心准入状态、头像认证状态、实名认证状态、学历认证状态。其中用户填写字段的字段 ID、显示名与敏感级别以 `M01-DATA-user-input-fields` 为准；状态类字段由系统状态枚举维护。
