## 1. 模块目标

移动端 PRD-02 是用户完成三重认证后感知“有人喜欢我、有人看过我、我和谁已经匹配成功”的核心反馈链路。它既负责真实关系反馈的展示与状态流转，也负责将模糊态、单条解锁、会员全量查看自然承接到 PRD-04 商业化能力。

**用户故事：** 作为三重认证通过的用户，我想查看谁喜欢我、谁看过我、谁与我相互喜欢，以便判断是否继续看主页、发送悄悄话或进入聊天。

**核心指标：** 关系反馈页访问率、模糊卡片点击率、单条解锁转化率、会员全量查看转化率、匹配成功后聊天发起率。

## 2. 用户与角色

| 角色 | 在本模块中做什么 | 引用全局角色/规则 |
|------|------------------|-------------------|
| 未登录访客 | 不能查看个人关系反馈，点击入口引导登录 | `GLB-ROLE-visitor` |
| 已登录但未完成三重认证用户 | 被拦截到认证引导，不形成真实关系反馈 | `M01-RULE-core-access` |
| 三重认证通过用户 | 查看喜欢、访客、相互喜欢；发起喜欢；触发匹配成功 | `GLB-ROLE-app-user`、`M02-RULE-core-access` |
| 时空邂逅会员用户 | 可按会员权益全量清晰查看喜欢/访客记录，可使用隐藏访问记录权益 | `M04-ENUM-vip-benefit-type` |

## 3. 关键业务流程

### 3.1 喜欢反馈流程

```text
入口：觅缘、精选、理想型、婚恋用户主页等场景的喜欢动作
正常路径：
  1. 用户 A 对用户 B 点击喜欢
  2. 后端校验 A/B 均满足 `M02-RULE-core-access`、异性、账号正常、未拉黑
  3. 写入 `M02-SM-like-record=active`
  4. 用户 B 的喜欢我的列表出现记录
  5. 若 B 已喜欢 A，生成 `M02-SM-mutual-match=matched`
  6. 当前操作触发首次匹配时展示 `APP-02-PAGE-match-success-modal`
异常：
  - 未完成三重认证：不写关系，展示认证引导
  - 重复喜欢：幂等返回已喜欢
  - 任一方状态异常：不写关系或置失效
出口：继续推荐、查看主页或进入相互喜欢/聊天
```

### 3.2 来访反馈流程

```text
入口：用户进入另一名用户的婚恋用户主页
正常路径：
  1. 判断当前页面是否为婚恋用户主页；社区动态详情、职业主页不计入
  2. 校验访问方/被访问方账号状态正常
  3. 若访问方开启隐藏访问记录且权益有效，不进入对方访客列表
  4. 若未隐藏，按 30 分钟去重窗口生成或更新访客记录
  5. 被访问方在最近看过我的列表看到最近 7 天记录
异常：
  - 访问者未完成核心准入：不生成真实访客记录
  - 记录超过 7 天：前台列表不展示，后台可查
```

### 3.3 解锁与会员全量查看流程

```text
入口：喜欢我的/最近看过我的模糊卡片或解锁全部按钮
正常路径：
  1. 普通用户看到模糊列表
  2. 点击单条模糊卡片，唤起 `APP-04-PAGE-paywall-modal` 千寻币确认
  3. 扣币成功后当前记录永久清晰
  4. 点击解锁全部，唤起 `APP-04-PAGE-paywall-modal` 会员引导
  5. 会员有效期内对应列表全量清晰
异常：
  - 余额不足：由 PRD-04 余额不足弹窗承接
  - 目标关系已失效：不扣币，展示失效原因
  - 最近访客超过 7 天：不在最近看过我的前台列表展示，资产/解锁记录可追溯
```

### 3.4 悄悄话回复触发匹配流程

```text
入口：PRD-03 悄悄话被接收方回复成功
正常路径：
  1. PRD-03 发送悄悄话，不要求双方已普通匹配
  2. 接收方回复悄悄话成功
  3. PRD-03 发出内部事件
  4. PRD-02 生成 `matchSource=whisper_reply` 的匹配成功记录
  5. 双方进入相互喜欢列表，后续普通聊天遵守 PRD-03 女性保护规则
```

## 4. 核心数据模型

### 4.1 实体清单

| 实体 | 表名（建议） | 说明 | 所属模块 | 关键字段 |
|------|-------------|------|----------|----------|
| 喜欢记录 | `relation_like_record` | 用户喜欢关系 | 02 | likeNo, fromUserId, toUserId, sourceScene, likeStatus, invalidReason |
| 访客记录 | `relation_visit_record` | 婚恋用户主页访问反馈 | 02 | visitNo, visitorUserId, targetUserId, sourceScene, visitTime, visitStatus |
| 匹配成功记录 | `relation_mutual_match` | 相互喜欢/匹配成功关系 | 02 | matchNo, userAId, userBId, matchSource, matchStatus, invalidReason |
| 解锁关系映射 | `relation_unlock_ref` | PRD-02 业务对象与 PRD-04 解锁记录关联 | 02/04 | relationType, relationId, unlockNo, displayStatus |

### 4.2 实体关系

```text
用户 1──N 喜欢记录（fromUserId）
用户 1──N 喜欢记录（toUserId）
用户 1──N 访客记录（visitorUserId）
用户 1──N 访客记录（targetUserId）
用户 1──N 匹配成功记录（userAId/userBId）
PRD-04 解锁记录 1──1 PRD-02 喜欢/访客记录
```

### 4.3 跨模块字段引用

| 本模块使用字段 | 来源模块 | 来源实体 | 若来源模块未上线如何处理 |
|---------------|----------|----------|-------------------------|
| 核心准入状态 | PRD-01 | 用户认证/核心准入 | 关系反馈入口展示认证引导，不形成真实关系 |
| 会员状态/隐私权益 | PRD-04 | 用户资产/会员权益 | 按普通用户模糊态展示 |
| 千寻币单条解锁结果 | PRD-04 | 解锁记录/资产流水 | 单条解锁入口不可用或提示稍后重试 |
| 悄悄话回复事件 | PRD-03 | 悄悄话消息 | `whisper_reply` 匹配来源不触发 |
| 婚恋用户主页 | PRD-05/用户主页 | 用户主页 | 若主页未上线，访客记录入口不可用 |

## 9. 非功能性需求

### 9.1 权限

| 权限项 | 要求 |
|--------|------|
| 个人关系反馈数据 | 仅本人可查看，未登录不可访问 |
| 核心准入 | 未完成三重认证不可形成真实喜欢、访客、匹配 |
| 关系失效 | 失效后不可继续聊天、不可跳转失效对象主页；前台展示原因 |

### 9.2 性能

| 场景 | 要求 |
|------|------|
| 喜欢我的/最近看过我的列表 | 首屏 20 条以内，常规网络 800ms 内返回 |
| 相互喜欢列表 | 首屏 20 条以内，常规网络 800ms 内返回 |
| 匹配成功弹窗 | 当前喜欢操作返回后即时展示，不依赖轮询 |

### 9.3 并发与幂等

| 操作 | 幂等要求 |
|------|----------|
| 发起喜欢 | 同一 fromUserId + toUserId 当前 active 仅保留一条 |
| 取消喜欢 | 重复取消返回成功态，不重复触发事件 |
| 访客记录 | 同一 visitorUserId + targetUserId 在 30 分钟窗口内不新增多条展示记录 |
| 匹配成功 | 同一用户对同一来源在有效状态下不重复生成多条 active 匹配 |
| 单条解锁 | 同一用户同一记录重复解锁不重复扣币 |

### 9.4 埋点建议

| 埋点 ID | 触发时机 |

| `likes_me_page_show` | 喜欢我的列表曝光 |

## 10. 验收总则

```text
AC-ID: APP-02-AC-core-access
Given 用户未完成实名、头像、学历三重认证任一项
When  用户进入喜欢我的、最近看过我的或相互喜欢
Then  页面不展示真实列表，展示认证引导，且后端不返回真实关系反馈数据

AC-ID: APP-02-AC-relation-invalid
Given 相互喜欢关系因拉黑、注销、冻结或取消喜欢失效
When  用户查看相互喜欢列表
Then  前台记录置为失效态并展示失效原因，不再提供聊天入口
```
