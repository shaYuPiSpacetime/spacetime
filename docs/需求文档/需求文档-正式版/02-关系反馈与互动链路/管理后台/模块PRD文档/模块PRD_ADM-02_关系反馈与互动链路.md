## 1. 模块目标

管理后台 PRD-02 为运营、客服、风控提供关系反馈链路的查询与排查能力。首版目标不是建设独立关系运营系统，也不建设关系反馈配置页，而是在 ADM-01 既有 App 用户管理中，让后台能看清“某个用户为什么看不到喜欢/访客/匹配、某条关系为何失效、当前规则固定口径是什么”。

**用户故事：** 作为客服/运营，我想在用户列表中筛选关系反馈状态，并在用户详情中查看喜欢、访客、匹配和解锁记录，以便快速排查用户咨询。

## 3. 关键业务流程

### 3.1 客服排查关系反馈流程

```text
入口：用户管理中心 -> App 用户管理 -> 用户详情
正常路径：
  1. 客服搜索用户并打开用户详情
  2. 查看关系反馈摘要：核心准入、被喜欢数、最近 7 天访客 UV/PV、相互喜欢数、最近匹配时间
  3. 切换喜欢记录/访客记录/匹配记录/解锁记录 Tab
  4. 根据状态和失效原因解释用户问题
  5. 如涉及支付/解锁扣币，跳转或引用 PRD-04 资产流水
异常：
  - 无权限：Tab 隐藏或展示无权限态
  - 资产服务不可用：解锁记录展示基础信息，资产流水入口提示不可用
```

### 3.2 固定规则排查流程

```text
入口：用户管理中心 -> App 用户管理 -> 用户详情 -> 关系记录区块
正常路径：
  1. 客服/运营查看当前用户关系反馈摘要
  2. 按喜欢、访客、匹配、解锁记录 Tab 定位具体记录
  3. 结合固定参数解释前台表现：最近访客 7 天、访客去重 30 分钟、喜欢我的模糊展示上限 10、同一对用户匹配弹窗仅首次展示
  4. 如涉及隐藏访问记录、会员权益或解锁扣币，引用 PRD-04/PRD-06 的权益与入口说明
异常：
  - 无权限：Tab 隐藏或展示无权限态
  - 跨模块数据不可用：展示基础关系记录，并提示对应模块数据暂不可用
```

## 6. 边界（本期不做）

| 不做的能力 | 原因 | 本期处理方式 | 后续计划 |
|------------|------|--------------|----------|
| 独立关系反馈工作台 | 超出首版报价和原后台 PRD 范围 | 隐藏 | 二期 |
| 全局关系列表 | 已确认首版只在用户详情看 | 不建菜单，不建页面 | 客服排查量大时再评估 |
| 关系反馈后台配置页 | 本轮确认规则按代码固定实现，不做配置化 | 不建菜单，不建页面，不提供保存确认弹窗/配置日志/配置接口 | 如需运营可调，另立二期配置化需求 |
| 手工制造匹配/补喜欢/补访客 | 高风险，易破坏用户信任与审计 | 禁止 | 如需需单独审批 PRD |
| 手工恢复关系记录 | 高风险且需明确业务补偿规则 | 禁止 | 后续另立 |
| 关系链路 BI 看板 | 非首版必需 | 不展示 | 二期 |
| 女性保护发送顺序配置 | 主定义归 PRD-03 | PRD-02 仅引用；保护期天数默认 3 天、发送顺序和后台配置页必须由 PRD-03 承接 | PRD-03 承接 |

## 9. 非功能性需求

### 9.1 权限与审计

| 场景 | 要求 |
|------|------|
| 查看关系记录 | 按角色权限控制，记录查看日志 |
| 导出关系记录 | 首版默认不开放；若授权导出，必须二次确认并记录日志 |
| 查看敏感用户信息 | 继承管理后台全局脱敏与二次确认规则 |

### 9.2 性能

| 场景 | 要求 |
|------|------|
| 用户详情摘要 | 常规网络 800ms 内返回 |
| 关系记录 Tab | 默认分页 10 条，可切换 10/20/50 |

### 9.3 数据保留

| 数据 | 保留口径 |
|------|----------|
| 喜欢/访客/匹配底层记录 | 后台长期保留，用于客服排查和审计 |
| 最近看过我的前台展示 | 只展示最近 7 天 |
| 已解锁记录 | 前台清晰权益永久有效；资产追溯由 PRD-04 保留 |
