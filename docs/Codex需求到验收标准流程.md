# Codex 需求到验收标准流程

> 适用项目：Spacetime 成家立业  
> 适用对象：使用 Codex 推进需求分析、PRD 冻结、技术方案、开发、自测、验收的团队成员  
> 流程版本：v1.2  
> 核心目标：用同一套 Codex 流程，让一次性交互交付达到 90% 以上  
> 示例需求：`docs/需求文档/需求文档-正式版/01-用户准入与资料认证初始化/`

## 1. 流程目标

本流程用于处理一个完整需求目录。需求目录通常同时包含整体需求、模块公共定义、移动端需求、管理后台需求、页面规格、UI 缺口清单等文件。

标准链路：

```text
文档盘点 -> 需求合理性与闭环评审 -> PRD 冻结 -> 技术方案 -> 测试用例 -> 实现 -> 自测 -> 验收报告 -> 一致性校验
```

这套流程的目标不是“把文档读一遍”，而是让 Codex 在写代码前先发现 90% 以上会导致返工的问题：

1. 移动端流程是否闭环。
2. 管理后台流程是否闭环。
3. 移动端提交的数据、状态和审核结果是否能在后台承接。
4. 后台审核、配置、导入、导出、审计等动作是否能反向影响移动端。
5. 整体 PRD、端内 PRD、页面规格、UI 缺口清单是否互相一致。
6. 需求、技术方案、测试用例、代码、自测、验收是否能形成追溯链。

## 2. 默认执行协议

当用户只提供一个需求目录，并表达“按标准流程处理、全流程交付、从需求到验收、完成这个需求”等意图时，Codex 默认进入全流程模式。

用户只需要给出：

```text
按照 docs/Codex需求到验收标准流程.md，完整处理这个需求目录：
{需求目录路径}
```

Codex 必须按以下协议执行：

1. 先读取本流程文档、`AGENTS.md`、`TEAM_STANDARDS.md` 和需求目录。
2. 自动执行阶段 0 到阶段 8，不要求用户每一步手动下命令。
3. 每进入一个阶段，声明本阶段使用的 Codex skill。
4. 每完成一个阶段，写入对应阶段产物，并检查门禁。
5. 门禁满足时，自动进入下一阶段。
6. 门禁不满足时，停止推进，输出阻塞清单、影响范围和建议处理方式。
7. 需要环境、Token、账号、测试数据时，先从仓库配置读取；读取不到时只问缺失项，不编造。
8. 需要破坏性操作、外部网络、生产环境、真实资金或敏感数据时，必须先停下确认。
9. 自测、验收、一致性校验必须基于真实证据，不能用“应该可以”代替结果。
10. 最终输出交付总结，列出全部产物路径、通过/失败/跳过项、遗留问题和下一步建议。

### 2.1 自动推进与停止条件

| 场景 | Codex 行为 |
|------|------------|
| 文档完整且无 P0/P1 阻塞 | 自动从需求评审推进到 PRD 冻结 |
| PRD 已冻结或可冻结 | 自动输出技术方案 |
| 技术方案无核心缺口 | 自动生成测试用例 |
| 测试用例覆盖 P0/P1 | 自动进入实现 |
| 实现完成 | 自动执行可执行的自测 |
| 缺测试环境、Token、账号或数据 | 记录跳过项和补测条件，不伪造通过 |
| 发现 P0 阻塞 | 停止推进，不进入下一阶段 |
| 发现 P1 阻塞但有明确降级方案 | 可有条件推进，并在后续产物中持续追踪 |
| 发现需求超范围 | 记录超范围项，不静默实现 |
| 发现代码/测试失败 | 使用 `systematic-debugging` 定位根因后再修复 |

### 2.2 最终交付包

全流程完成后，至少应形成以下交付包：

```text
docs/需求分析/{yyyy-mm-dd}-{模块名}-requirement-review.md
docs/需求文档/需求冻结记录/{模块名}-prd-freeze.md
docs/技术方案/{yyyy-mm-dd}-{模块名}-tcdesign.md
docs/测试文档/{模块名}-testcase.md
docs/测试文档/{模块名}-testreport.md
docs/验收报告/{yyyy-mm-dd}-{模块名}-acceptance.md
docs/一致性校验/{yyyy-mm-dd}-{模块名}-consistency.md
```

如果本轮产生接口自动化脚本，还应包含：

```text
docs/测试文档/{模块名}-test-l1.sh
```

如果完成代码实现，还应在最终总结中列出：

| 类型 | 必须说明 |
|------|----------|
| 后端改动 | Controller、Service、DAO、Mapper、实体、SQL 等 |
| 管理后台改动 | 页面、路由、API、权限、状态管理等 |
| 移动端/小程序改动 | 页面、接口、状态、异常态等 |
| 测试改动 | 单测、接口脚本、页面测试、手工验收项 |
| 未执行项 | 缺什么环境或数据，如何补测 |

## 3. Codex Skill 使用原则

本流程以 Codex skills 为主入口，不依赖 Claude skills。

当前可用 Codex skills：

| 阶段 | Codex skill | 用途 |
|------|-------------|------|
| 需求合理性评审 | `brainstorming` | 识别需求边界、业务闭环、疑问与验收条件 |
| 技术方案 | `writing-plans` | 形成可实施的技术设计和任务拆解 |
| 实现 | `test-driven-development` | 新行为先定义测试或可验证条件 |
| 实施执行 | `executing-plans` | 按计划逐步开发 |
| 并行执行 | `subagent-driven-development` | 多个独立任务可并行推进时使用 |
| 调试 | `systematic-debugging` | 出现 bug 或测试失败时先定位根因 |
| 自测 | `verification-before-completion` | 完成前运行验证命令并记录证据 |
| Review | `requesting-code-review` | 提交前审查缺陷、风险、遗漏测试 |
| 处理 Review | `receiving-code-review` | 收到评审意见后逐条验证和修改 |

建议后续补充项目级 Codex skill：

| 建议 skill | 职责 |
|------------|------|
| `project-delivery-lifecycle` | 读取本流程，判断当前阶段，检查阶段产物与门禁，维护追溯矩阵 |

在该 skill 创建前，本文件就是团队流程的 source of truth。

### 3.1 阶段技能速查

| 阶段 | 必用 Codex skill | 可选 Codex skill | 触发提示词 |
|------|------------------|------------------|------------|
| 阶段 0：文档盘点 | `brainstorming` | 无 | “先盘点这个需求目录的文档结构和缺失项” |
| 阶段 1：需求合理性与闭环评审 | `brainstorming` | `systematic-debugging` | “检查需求是否合理、移动端和管理后台是否闭环” |
| 阶段 2：PRD 冻结 | `brainstorming` | 无 | “基于需求评审结论冻结 PRD，P0 不允许冻结” |
| 阶段 3：技术方案 | `writing-plans` | `brainstorming` | “基于冻结 PRD 输出技术方案” |
| 阶段 4：测试用例 | `test-driven-development` | `writing-plans` | “基于技术方案生成测试用例，覆盖接口和页面闭环” |
| 阶段 5：实现 | `test-driven-development`、`executing-plans` | `subagent-driven-development`、`systematic-debugging` | “按技术方案和测试用例实现” |
| 阶段 6：自测 | `verification-before-completion` | `systematic-debugging` | “执行接口测试、页面测试、构建测试并出自测报告” |
| 阶段 7：验收报告 | `verification-before-completion` | `requesting-code-review` | “基于测试证据输出验收报告” |
| 阶段 8：一致性校验 | `verification-before-completion`、`requesting-code-review` | `systematic-debugging` | “校验 PRD、方案、测试、代码、自测、验收是否一致” |

使用规则：

1. 每个阶段开始时，先声明本阶段使用的 skill。
2. 一个阶段有多个必用 skill 时，按表格顺序使用。
3. 如果阶段门禁不满足，不进入下一阶段。
4. 如果出现 bug、测试失败、闭环断链，立即使用 `systematic-debugging` 定位根因。
5. 如果要声明“完成、通过、可验收”，必须先使用 `verification-before-completion` 获取证据。

## 6. 阶段 0：文档盘点

**本阶段使用技能：**

| 类型 | Skill | 用法 |
|------|-------|------|
| 必用 | `brainstorming` | 读取需求目录结构，识别整体 PRD、端内 PRD、页面规格、UI 缺口和全局定义是否齐全 |

**本阶段不要使用：** `writing-plans`、`executing-plans`、`test-driven-development`。文档盘点还不是技术设计或实现阶段。

### 6.1 目标

确认需求输入是否完整，避免后续基于残缺 PRD 输出技术方案。

### 6.2 Codex 必读

```text
AGENTS.md
TEAM_STANDARDS.md
docs/需求文档/需求文档-正式版/全局定义/
docs/需求文档/需求文档-正式版/{需求编号}-{需求名称}/
```

### 6.3 盘点规则

Codex 必须先输出文档清单，不允许直接进入技术方案。

| 检查项 | 判断标准 |
|--------|----------|
| 公共定义 | 是否存在模块级术语、枚举、状态、配置、接口口径 |
| 移动端模块 PRD | 是否有目标、范围、边界、流程、数据、页面清单、验收标准 |
| 移动端页面规格 | 每个页面是否有定位、布局、字段、操作、状态异常、验收标准、关联 |
| 管理后台模块 PRD | 是否有角色、流程、范围、数据、权限、页面清单、验收标准 |
| 管理后台页面规格 | 每个页面是否有列表、字段、操作、状态异常、权限、验收标准、关联 |
| UI 缺口清单 | 是否记录缺失页面、异常态、优先级和处理方式 |
| 全局定义 | 是否存在项目级共享口径和端专属口径 |

### 6.4 输出

文档盘点写入需求评审报告的“文档清单”章节。

## 7. 阶段 1：需求合理性与闭环评审

**本阶段使用技能：**

| 类型 | Skill | 用法 |
|------|-------|------|
| 必用 | `brainstorming` | 判断需求是否合理，梳理移动端闭环、管理后台闭环、前后台联动和疑问分级 |
| 条件使用 | `systematic-debugging` | 当发现流程矛盾、状态断链、字段冲突时，用根因分析方式定位问题来源 |

**阶段输出前必须说明：** 已使用哪个 skill、发现了哪些 P0/P1/P2/P3 问题、是否允许进入 PRD 冻结。

### 7.1 目标

这是整个流程最重要的阶段。Codex 必须先判断需求是否合理、是否有问题、移动端和管理后台是否闭环，再决定能不能冻结 PRD。

### 7.2 评审维度

| 维度 | 必查问题 |
|------|----------|
| 业务目标 | 需求目标、用户价值、运营目标是否清楚 |
| 范围边界 | 本期做什么、不做什么、延期什么是否明确 |
| 移动端闭环 | 用户从入口、填写、提交、失败、重试、完成、进入核心链路是否闭环 |
| 管理后台闭环 | 后台从列表、详情、审核、配置、导入、导出、审计是否闭环 |
| 前后台联动 | 移动端提交的数据是否进入后台；后台处理结果是否回到移动端 |
| 数据闭环 | 字段、枚举、字典、状态、配置是否有统一来源 |
| 状态闭环 | 每个状态是否有进入条件、退出条件、展示文案和后续动作 |
| 权限闭环 | 后台角色、权限点、敏感字段、下载、导出、审计是否明确 |
| UI 闭环 | 页面规格和蓝湖/UI 缺口是否能支撑实现和验收 |
| 异常闭环 | 空态、失败、驳回、第三方不可用、权限拒绝、重复提交是否覆盖 |
| 测试可行性 | 每条核心需求是否能转成接口测试、页面测试或手工验收 |

### 7.3 移动端闭环检查

每条移动端主流程必须按以下链路检查：

```text
入口 -> 页面展示 -> 用户输入 -> 前端校验 -> 接口提交 -> 后端处理 -> 状态反馈 -> 用户下一步动作 -> 核心能力开放/阻断
```

最低检查项：

| 检查项 | 说明 |
|--------|------|
| 入口 | 用户从哪里进入该页面或流程 |
| 前置条件 | 是否需要登录、资料、认证、权限、定位、上传授权 |
| 字段 | 字段是否来自公共定义或端内定义 |
| 操作 | 主按钮、次按钮、跳过、返回、重试是否明确 |
| 接口 | 是否能推导出需要的提交、查询、上传、状态接口 |
| 状态 | 提交中、成功、失败、审核中、驳回、已通过是否明确 |
| 异常 | 年龄不符、定位失败、上传失败、第三方不可用等是否有处理 |
| 后续 | 完成后进入哪个页面或能力，未完成时如何拦截 |

### 7.4 管理后台闭环检查

每个管理后台页面必须按以下链路检查：

```text
菜单入口 -> 权限校验 -> 列表/筛选 -> 详情 -> 操作 -> 状态变更 -> 通知/回写 -> 审计日志 -> 列表刷新/后续处理
```

最低检查项：

| 检查项 | 说明 |
|--------|------|
| 菜单入口 | 页面属于哪个菜单、路由、权限点 |
| 角色权限 | 谁能看、谁能操作、谁能导出、谁能看敏感字段 |
| 列表字段 | 字段是否脱敏、是否支持筛选/排序/分页 |
| 详情信息 | 是否能支撑审核或运营判断 |
| 操作结果 | 通过、驳回、复核、配置、导入、导出后的状态变化 |
| 移动端回写 | 后台结果是否通知移动端、是否影响准入状态 |
| 审计 | 敏感查看、原图下载、导出、高风险配置是否留痕 |
| 异常 | 无权限、数据不存在、重复审核、导入失败、第三方失败是否覆盖 |

### 7.5 前后台闭环矩阵

需求评审必须输出前后台闭环矩阵。

```markdown
| 闭环 ID | 移动端触发 | 后端状态/数据 | 管理后台承接 | 后台处理结果 | 移动端反馈 | 是否闭环 | 问题 |
|---------|------------|---------------|--------------|--------------|------------|----------|------|
```

示例：

```text
用户提交学历认证 -> 生成学历审核记录 -> 后台学历认证审核页处理 -> 通过/驳回 -> 移动端认证页和站内信更新
```

### 7.6 问题分级

| 等级 | 定义 | 处理规则 |
|------|------|----------|
| P0 | 主流程无法成立、前后台断链、核心数据/状态缺失、安全合规缺口 | 不允许冻结 PRD |

| 必用 | `executing-plans` | 按技术方案和任务拆解逐步实现 |
| 条件使用 | `subagent-driven-development` | 多个互不依赖任务可并行时使用，例如后台页面和移动端接口并行 |
| 条件使用 | `systematic-debugging` | 实现或测试中出现异常时先定位根因再修改 |

### 11.3 实现原则

1. 严格遵守 `Controller -> Service -> ServiceImpl -> DAO -> DAOImpl -> Mapper`。
2. `admin/` 与 `miniapp/` 不互相 import，共享逻辑放 `common/`。
3. Controller 返回精确 `R<T>`。
4. 管理后台接口使用 `@RequirePermission` 和既有 RBAC 模式。
5. 业务表包含 `BaseEntity` 审计字段和逻辑删除。
6. 前端 API 放 `frontend/src/api/`，路由放 `frontend/src/router/`。
7. 页面组件只拼 UI 和调用 hook，业务逻辑放 hook/API/store。
8. 先实现最小闭环，再补充次要能力。
9. 不把未冻结的新需求混入当前实现。

| 必用 | `requesting-code-review` | 用 Review 视角查断链、超范围实现、遗漏测试和风险项 |
| 条件使用 | `systematic-debugging` | 发现断链时定位来源：PRD 缺失、方案遗漏、实现遗漏、测试遗漏或报告不实 |

**阶段输出前必须说明：** 一致性结论、断链项、超范围项和是否达到 90% 一次性交付口径。

### 14.1 目标

确认 PRD、需求评审、技术方案、测试用例、代码、自测报告、验收报告没有断链。

### 14.2 校验方向

| 方向 | 检查 |
|------|------|
| PRD -> 需求评审 | 核心需求是否被识别，问题是否被分级 |
| 需求评审 -> PRD 冻结 | P0/P1 是否处理后再冻结 |
| PRD 冻结 -> 技术方案 | 冻结范围是否都有设计 |
| 技术方案 -> 测试用例 | 接口、页面、状态、权限是否都有测试 |
| 测试用例 -> 实现 | 用例中的行为是否已实现 |
| 实现 -> 自测 | 改动是否被真实验证 |
| 自测 -> 验收 | 验收结论是否如实反映测试结果 |
| 全链路 | 是否有超范围实现或静默需求变更 |

### 14.4 结论规则

| 条件 | 结论 |
|------|------|
| P0/P1 无断链，P2/P3 断链有处理计划 | 一致 |
| P0/P1 无断链，但存在待补测或待确认项 | 有条件一致 |
| 任意 P0/P1 出现 PRD、方案、测试、代码、自测、验收断链 | 不一致 |

## 15. 一次性交互 90% 的口径

本流程中的“一次性交互达到 90% 以上”按以下方式判定：

```text
一次性交付率 = 首轮验收通过的 P0/P1 需求数 / 冻结范围内 P0/P1 需求总数
```

计入“首轮验收通过”的条件：

1. PRD 冻结后没有因需求理解错误返工。
2. 技术方案覆盖该需求。
3. 测试用例覆盖该需求。
4. 代码实现满足该需求。
5. 自测或验收能提供证据。
6. 一致性校验无 P0/P1 断链。

不计入通过的情况：

1. 需求评审漏掉核心问题，导致实现后返工。
2. 移动端和管理后台闭环断裂。
3. 后台状态无法回写移动端。
4. 测试用例缺失导致验收时才发现问题。
5. 自测报告伪通过或缺证据。

## 16. 团队标准提示词

### 16.1 全流程默认提示词

以后处理任意需求时，优先使用这一条：

```text
按照 docs/Codex需求到验收标准流程.md，完整处理这个需求目录：
{需求目录路径}

要求：从文档盘点、需求合理性与前后台闭环评审开始，依次完成 PRD 冻结、技术方案、测试用例、实现、自测、验收报告和一致性校验。每个阶段使用文档规定的 Codex skill，产出对应文档；遇到 P0 阻塞、缺环境/Token/账号/测试数据、外部服务或需要确认的高风险操作时停止并说明。
```

PRD-01 示例：

```text
按照 docs/Codex需求到验收标准流程.md，完整处理这个需求目录：
docs/需求文档/需求文档-正式版/01-用户准入与资料认证初始化/

要求：从文档盘点、需求合理性与前后台闭环评审开始，依次完成 PRD 冻结、技术方案、测试用例、实现、自测、验收报告和一致性校验。每个阶段使用文档规定的 Codex skill，产出对应文档；遇到 P0 阻塞、缺环境/Token/账号/测试数据、外部服务或需要确认的高风险操作时停止并说明。
```

### 16.2 只做需求评审

```text
按照 docs/Codex需求到验收标准流程.md，基于以下需求目录执行文档盘点和需求合理性/闭环评审：
docs/需求文档/需求文档-正式版/01-用户准入与资料认证初始化/

重点检查：需求是否合理、移动端是否闭环、管理后台是否闭环、前后台状态是否互通、页面规格和 UI 缺口是否影响实现。
```

### 16.3 PRD 冻结

```text
按照 docs/Codex需求到验收标准流程.md，基于需求评审报告冻结 PRD，输出 PRD 冻结记录。若存在 P0 问题，不要冻结。
```

### 16.4 技术方案

```text
按照 docs/Codex需求到验收标准流程.md，基于 PRD 冻结记录和需求评审报告输出技术方案，必须覆盖前后台闭环、接口、状态、权限、数据库、页面和测试策略。
```

### 16.5 测试用例

```text
按照 docs/Codex需求到验收标准流程.md，基于技术方案生成测试用例，覆盖 L1 接口测试、L2 Controller、L3 Service、L4 页面测试和前后台闭环测试。
```

### 16.6 实现

```text
按照 docs/Codex需求到验收标准流程.md，基于技术方案和测试用例执行实现。不得实现冻结范围之外的需求。
```

### 16.7 自测

```text
按照 docs/Codex需求到验收标准流程.md，对当前实现执行自测，覆盖接口测试、页面测试、构建测试和前后台闭环测试。缺少环境、Token、账号或测试数据时标记跳过，不要编造。
```

### 16.8 验收报告

```text
按照 docs/Codex需求到验收标准流程.md，基于 PRD、技术方案、测试用例和自测报告输出验收报告，说明需求完成度、闭环完成度、测试证据和遗留问题。
```

### 16.9 一致性校验

```text
按照 docs/Codex需求到验收标准流程.md，执行一致性校验，输出需求到代码到验收的追溯矩阵，并列出断链项和超范围项。
```

## 17. 禁止事项

1. 禁止跳过需求合理性与闭环评审直接写技术方案。
2. 禁止移动端和管理后台未闭环就冻结 PRD。
3. 禁止有 P0 问题时冻结 PRD。
4. 禁止 PRD 改了但不更新技术方案和测试用例。
5. 禁止技术方案只写“按 PRD 实现”。
6. 禁止测试报告中伪造通过结果。
7. 禁止编造 API 地址、Token、账号、测试数据。
8. 禁止管理后台和小程序端互相 import。
9. 禁止 Controller 直接调 DAO 或 Mapper。
10. 禁止 ServiceImpl 直接调 Mapper。
11. 禁止新增需求混入当前需求而不记录超范围项。
12. 禁止验收报告只写结论不写证据。
13. 禁止一致性校验跳过 P0/P1 断链。
