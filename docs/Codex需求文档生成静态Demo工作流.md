# Codex 需求文档生成静态 Demo 工作流

> 适用项目：Spacetime 成家立业  
> 适用对象：使用 Codex 将标准需求目录还原为可直接打开的静态 HTML demo 的团队成员  
> 流程版本：v1.5
> 核心目标：先生成页面覆盖和细节还原均达到 95% 以上的 `index.html`，用于产品演示和需求/技术对齐；对齐后再切换到正式研发流程落地前后端项目
> 关联流程：`docs/Codex需求到验收标准流程.md`

## 1. 流程目标

本流程只处理“标准需求文档目录 -> 静态 HTML demo”的阶段。

静态 demo 的用途是：

1. 给产品、需求方、技术看真实页面效果。
2. 在写真实前后端代码前，对齐页面、字段、按钮、弹窗、状态、异常态、控件形态、详情形式、分页统计、反馈提示和交互链路。
3. 用截图、细节核对矩阵和还原度矩阵证明需求页面已被充分还原。
4. 对齐完成后，再进入 `docs/Codex需求到验收标准流程.md` 做真实技术方案、前后端开发和测试验收。

本流程不做：

1. 不接真实接口。
2. 不写后端代码。
3. 不建数据库。
4. 不做生产权限。
5. 不把静态 HTML 当成真实业务功能交付。

标准链路：

```text
逐字读取需求目录 -> 提取页面元素与细节 -> 制定 HTML 方案与交互剧本 -> 生成 index.html -> 自测截图 -> 细节核对 -> 还原度评分 -> 交付静态 demo
```

推荐执行方式：

| 场景 | 建议 |
|------|------|
| 小需求或只提取清单 | 普通对话模式即可，按阶段产物推进 |
| 完整生成静态 demo | 建议开启目标模式，让 Codex 按评分结果持续修复 |
| 大需求、多页面 | 必须按第 4 节拆分页面组，逐组读取、实现、截图和评分 |
| 整体或细节低于 95% | 不能只改分数，必须按扣分项修复 HTML 后重新验证 |
| 信息缺失或工具受限 | 记录为缺口或阻塞，不得凭空补齐需求 |

目标模式不是“一次性加载所有文档然后无限生成”，而是按页面组小步循环：实现一组、截图一组、评分一组、修复一组，最后再汇总成总 HTML 和总还原度报告。

细节还原目标：

1. Codex 必须认真阅读需求文档的每一个页面、表格、注释、规则、字段说明和交互说明，不得只根据标题或常见后台模板补页面。
2. 还原目标不是“有页面框架”，而是页面覆盖率、字段控件还原、列表/详情/分页/统计、操作反馈、状态异常、交互连贯性和视觉结构都达到 95% 以上。
3. 需求中出现的“展示、隐藏、可编辑、不可编辑、必填、只读、禁用、默认值、占位文案、下拉选项、时间格式、分页、统计、排序、详情、确认、提示、跳转、刷新”等细节，都必须进入元素清单和评分矩阵。
4. 如果无法从需求文档判断控件类型、详情形式、分页规则或操作反馈，必须记录为待确认缺口，不得自行脑补后打高分。
5. 评分必须基于逐项核对结果和截图证据，不能凭整体观感、完成数量或“页面看起来差不多”给 95% 以上。

## 2. 标准输入目录

每个需求模块默认都是同一套目录结构：

```text
docs/需求文档/需求文档-正式版/{需求编号}-{需求名称}/
├── PRD-{编号}_模块公共定义.md
├── 蓝湖UI缺少页面清单.md                 # 可选
├── 移动端/
│   ├── APP-{编号}_端内定义.md
│   ├── 模块PRD文档/
│   └── 页面规格/
└── 管理后台/
    ├── ADM-{编号}_端内定义.md
    ├── 模块PRD文档/
    └── 页面规格/
```

其中：

| 类型 | 是否必需 | 说明 |
|------|----------|------|
| `PRD-{编号}_模块公共定义.md` | 必需 | 模块级术语、状态、枚举、公共规则 |
| `移动端/APP-{编号}_端内定义.md` | 必需 | 小程序端页面、规则、全局口径 |
| `移动端/模块PRD文档/` | 必需 | 小程序模块 PRD |
| `移动端/页面规格/` | 必需 | 小程序页面规格，静态 HTML 必须还原 |
| `管理后台/ADM-{编号}_端内定义.md` | 必需 | 管理后台端内定义、权限、菜单口径 |
| `管理后台/模块PRD文档/` | 必需 | 管理后台模块 PRD |
| `管理后台/页面规格/` | 必需 | 管理后台页面规格，静态 HTML 必须还原 |
| `蓝湖UI缺少页面清单.md` | 可选 | 如果存在则读取；不存在不阻塞 |

如果缺少必需目录或必需文件，Codex 必须先在 `00-文档读取与页面范围.md` 中记录缺口，不得静默假设。

## 3. 标准交付物

完整跑完本流程，标准产物是：

```text
docs/静态Demo/{模块名}/
├── 00-文档读取与页面范围.md
├── 01-页面元素清单.md
├── 02-静态HTML实现方案.md
├── 03-静态HTML自测与还原度报告.md
├── 04-静态HTML交付报告.md
├── html/
│   ├── index.html
│   ├── assets/
│   └── mock/
├── 页面组/                              # 可选：大需求拆分时使用
└── 截图证据/
```

也就是：

```text
5 个标准 Markdown 文档 + 1 个可直接打开的 HTML 演示包 + 截图证据
```

`html/index.html` 是核心交付物，必须能直接用浏览器打开。

如果需求很大，可以额外输出 `页面组/` 下的中间文档，用来记录分组页面范围、元素清单、截图和评分。额外文档不替代 5 个标准 Markdown 文档。

## 4. 大需求拆分执行策略

当需求文档较多、页面较多或页面规格较长时，Codex 不得一次性把所有内容加载进同一轮上下文并直接生成完整 HTML。必须先按模块、端和功能点拆分执行，避免上下文过载导致遗漏、错读和还原质量下降。

### 4.1 触发拆分的条件

满足以下任一条件，就必须拆分：

| 条件 | 处理方式 |
|------|----------|
| 页面规格文件超过 8 个 | 按移动端/管理后台和页面组拆分 |
| 移动端和管理后台都包含多个页面组 | 先分别处理，再汇总 |
| 单个页面规格很长 | 先提取该页面元素，再进入 HTML 实现 |
| 页面之间依赖复杂 | 先梳理流程和状态，再分组实现 |
| 一次读取后无法稳定保留细节 | 停止一次性实现，改为分组产出中间结果 |

### 4.2 推荐拆分方式

优先按端拆分：

```text
移动端页面组
管理后台页面组
```

再按功能点拆分：

```text
登录/授权页面组
资料填写页面组
认证/审核页面组
列表/详情页面组
配置/导入导出页面组
结果/异常页面组
```

实际分组必须以需求目录中的页面规格为准，不要为了套模板强行改名。

### 4.3 分组执行规则

每个页面组都要形成可追溯的中间结果：

| 中间结果 | 说明 |
|----------|------|
| 页面组范围 | 本组包含哪些页面、来源文件是什么 |
| 页面组元素清单 | 字段、按钮、弹窗、状态、交互、mock 数据 |
| 页面组 HTML 片段或页面 | 本组在 `index.html` 中对应的页面区域 |
| 页面组截图 | 本组页面截图证据 |
| 页面组评分 | 本组还原度和扣分项 |

如果页面组还原度低于 95%，必须先修复该页面组，不要急着汇总总 HTML。

### 4.4 汇总规则

所有页面组完成后，再汇总成总 HTML：

```text
页面组 HTML -> 总 index.html -> 全页面导航 -> 全量打开检查 -> 总还原度评分
```

总还原度必须基于所有页面组评分和总入口验证，不允许只用局部页面组分数代替总分。

### 4.5 上下文保护规则

1. 不在同一轮里全文加载所有页面规格再直接实现。
2. 先产出 `00-文档读取与页面范围.md`，再按页面组读取细节。
3. 每完成一组就写入 `01-页面元素清单.md`，作为后续上下文恢复依据。
4. 如果发生上下文压缩或中断，优先从已产出的 00/01/02 文档恢复，不重新凭记忆实现。
5. 页面组之间共享的状态、枚举、字段口径必须来自 `PRD-{编号}_模块公共定义.md`。
6. 分组实现不能减少最终标准交付物，最终至少输出 5 个 Markdown 文档、`html/index.html` 和 `截图证据/`；大需求允许额外输出 `页面组/` 中间文档。

## 5. 静态 HTML 交付约束

| 类型 | 约束 |
|------|------|
| 主入口 | `docs/静态Demo/{模块名}/html/index.html` |
| 打开方式 | 双击 `index.html` 或用浏览器打开本地文件 |
| 网络依赖 | 不请求真实 API，不依赖后端服务，不依赖登录态 |
| 资源路径 | CSS、JS、图片、mock 数据使用相对路径；简单 demo 可内联 |
| 小程序 | 在 HTML 中用手机外框模拟页面、导航、表单、底部操作栏 |
| 管理后台 | 在 HTML 中模拟菜单、面包屑、筛选区、表格、详情、弹窗、抽屉 |
| 交互 | tab、筛选、分页、弹窗、抽屉、状态切换用本地 JS 模拟 |
| 数据 | 使用 mock 数据覆盖正常态、空态、异常态和关键状态 |
| 浏览器 | 优先保证 Chrome/Edge 可直接打开和演示 |

## 6. 执行辅助说明：Codex skill 与 D2Admin

本流程中提到的 `brainstorming`、`writing-plans`、`test-driven-development`、`executing-plans`、`verification-before-completion` 是 Codex 执行阶段使用的工作方法，不是项目依赖，也不是需要在前端项目里安装的 npm 包。

团队使用时按以下规则理解：

| 项目 | 结论 |
|------|------|
| Codex skill | 如果当前 Codex 环境已安装，就按阶段调用；如果未安装，也要按本文档的阶段产物和门禁执行 |
| D2Admin | 不是 Codex skill，只是管理后台静态页面的参考模式 |
| 项目依赖 | 本流程不要求安装 D2Admin、Vue 或其他后台框架 |
| 静态 HTML | 最终仍以可直接打开的 `html/index.html` 为准 |

管理后台页面可以参考 `d2-projects/d2-admin` 的后台信息架构和页面组织方式，但不能迁入 D2Admin 代码。

允许参考：

| 参考项 | 用法 |
|--------|------|
| 后台骨架 | 左侧菜单、顶部栏、面包屑、内容区 |
| 列表页模式 | 筛选表单、工具栏、表格、分页、批量操作 |
| 详情页模式 | 信息分组、状态标签、操作区、审核记录 |
| 弹窗/抽屉模式 | 新增、编辑、审核、驳回、导入、导出确认 |
| Mock 数据 | 用本地数据模拟列表、详情、状态流转 |

禁止：

| 禁止项 | 原因 |
|--------|------|
| 直接迁入 D2Admin/Vue 代码 | 本流程只交付静态 HTML |
| 引入重型前端框架 | 会破坏 HTML 直接打开的目标 |
| 为了参考 D2Admin 改项目架构 | 本阶段只做需求演示 demo |

## 7. 阶段 0：文档读取与页面范围

**本阶段使用技能：** `brainstorming`

### 7.1 目标

确认需求目录完整性，并列出本次静态 HTML 要还原的页面范围。

### 7.2 必读规则

Codex 必须读取：

```text
AGENTS.md
docs/Codex需求文档生成静态Demo工作流.md
用户提供的需求目录
PRD-{编号}_模块公共定义.md
移动端/APP-{编号}_端内定义.md
移动端/模块PRD文档/*.md
移动端/页面规格/*.md
管理后台/ADM-{编号}_端内定义.md
管理后台/模块PRD文档/*.md
管理后台/页面规格/*.md
```

如果存在 `蓝湖UI缺少页面清单.md`，也必须读取；如果不存在，只记录“未提供”，不作为阻塞。

### 7.3 输出

输出到：

```text
docs/静态Demo/{模块名}/00-文档读取与页面范围.md
```

模板：

```markdown
# {模块名} 文档读取与页面范围

## 1. 输入目录

| 项目 | 路径 | 是否存在 | 说明 |
|------|------|----------|------|

## 2. 阅读证据

| 文件 | 阅读方式 | 原文定位 | 提取内容 | 影响页面/字段 | 遗留问题 |
|------|----------|----------|----------|--------------|----------|

原文定位必须尽量写到文件名、章节标题、表格名称、页面 ID 或字段名；不能只写“已阅读”。如果需求文档中存在字段表、页面规则、按钮说明、弹窗说明、状态流转或异常说明，必须逐项记录到阅读证据或后续元素矩阵中。

## 3. 页面范围

| 页面 ID | 端 | 页面名称 | 来源文件 | 是否纳入 HTML | 说明 |
|---------|----|----------|----------|----------------|------|

## 4. 缺口

| 编号 | 缺口 | 等级 | 对 HTML 影响 | 处理方式 |
|------|------|------|-------------|----------|
```

### 7.4 门禁

进入下一阶段前必须满足：

1. 必需目录和必需文件已检查。
2. 小程序页面范围已列出。
3. 管理后台页面范围已列出。
4. 可选蓝湖缺少清单已读取或记录为未提供。
5. 页面范围不清楚时不得直接生成 HTML。
6. 阅读证据必须包含原文定位和影响对象；缺少字段、按钮、状态、控件、详情、分页、统计、反馈或交互规则的原文追溯时，不得进入页面元素清单。

## 8. 阶段 1：页面元素清单

**本阶段使用技能：** `brainstorming`

### 8.1 目标

把需求文档里的自然语言描述拆成 HTML 可还原、可评分的页面元素。

### 8.2 必须提取

| 元素 | 说明 |
|------|------|
| 页面结构 | 标题、区域、分组、列表、卡片、表单 |
| 字段 | 展示字段、表单字段、筛选字段、表格列、详情字段、统计字段 |
| 按钮 | 主按钮、次按钮、行操作、批量操作、底部操作、禁用态、危险操作 |
| 弹窗/抽屉 | 审核、驳回、新增、编辑、导入、导出、确认、详情、二次确认、结果反馈 |
| 状态 | 正常、加载、空态、失败、审核中、已通过、已驳回 |
| 异常态 | 权限不足、数据缺失、上传失败、重复提交、规则不满足 |
| 静态交互 | tab 切换、筛选、重置、分页、查看详情、编辑保存、删除确认、弹窗打开关闭、状态切换、返回刷新 |
| Mock 数据 | 列表数据、详情数据、统计数据、分页数据、表单默认值、异常样例 |
| 列表展示 | 展示形式、表格列、卡片字段、排序、筛选、分页、头部统计、空态 |
| 字段控件 | 输入框、文本域、下拉框、单选、多选、日期时间、上传、开关、只读、禁用、必填校验 |
| 字段权限 | 展示字段、可编辑字段、不可编辑字段、条件展示字段、条件禁用字段 |
| 详情形式 | 页面、弹窗、抽屉、分组信息、审核记录、操作区、返回方式 |
| 操作反馈 | toast、确认框、成功提示、失败提示、加载态、按钮 loading、防重复提交 |
| 交互链路 | 从入口到结果的完整点击路径，包含列表刷新、状态变化、弹窗关闭、页面回退 |
| 视觉细节 | 移动端间距、按钮层级、卡片密度、后台表格密度、状态标签颜色、空态表达 |

### 11.1 目标

### 11.2 必测项

| 类型 | 要求 |
|------|------|

截图规格：

| 类型 | 视口/尺寸 | 必须截图 | 命名建议 |
|------|-----------|----------|----------|
| 小程序页面 | 移动端宽度，建议 390x844 或同等手机比例 | 首页/列表、表单、详情、弹窗、toast、空态、异常态、操作后结果 | `mini-{页面ID}-{状态或操作}.png` |
| 管理后台页面 | 桌面宽度，建议 1440x900 或同等桌面比例 | 列表、筛选结果、分页、头部统计、详情、编辑、确认框、抽屉、空态、异常态、操作后结果 | `admin-{页面ID}-{状态或操作}.png` |
| 关键交互链路 | 与所属页面一致 | 操作前、操作中反馈、操作后状态变化至少各一张 | `{端}-{链路ID}-before/action/after.png` |

截图必须能支撑评分矩阵；如果某页面只截了默认态，没有详情、弹窗、异常态或操作后结果，不得把对应细节项评为满分。

### 11.3 还原度公式

```text
总还原度 =
页面覆盖率 * 15%
+ 字段与文案覆盖率 * 15%
+ 字段控件与可编辑性还原率 * 15%
+ 列表/详情/分页/统计还原率 * 15%
+ 按钮/弹窗/抽屉/反馈还原率 * 15%
+ 状态与异常态覆盖率 * 10%
+ 交互链路连贯性 * 10%
+ 视觉结构与移动端美观度 * 5%
```

通过标准：

| 结果 | 条件 |
|------|------|
| 达标 | 总还原度 >= 95%，页面覆盖率 = 100%，细节还原度 >= 95%，P0/P1 页面无缺失，关键交互链路无断点 |
| 有条件达标 | 总还原度 >= 95%，细节还原度 >= 95%，但存在不影响演示的 UI 缺口或待确认项 |
| 不达标 | 总还原度 < 95%，或细节还原度 < 95%，或任意 P0/P1 页面缺失，或字段控件/详情/分页/反馈/交互链路存在关键缺失 |

评分硬性规则：

1. 页面覆盖率、P0/P1 页面完整性是前置门禁；任意 P0/P1 页面缺失，总分不得超过 80%。
2. 字段控件类型、字段可编辑性、列表字段、详情形式、分页规则、头部统计、操作反馈任一大类缺失，总分不得超过 90%。
3. 关键交互链路只能单点打开、不能完成“操作 -> 反馈 -> 状态变化 -> 返回/刷新”的，总分不得超过 92%。
4. 只有大框架、缺少输入框/下拉框/时间框/只读态/禁用态/提示框等细节的，总分不得超过 88%。
5. 移动端页面明显粗糙、按钮不可触达、底部操作栏缺失、弹窗/toast 不连贯的，视觉结构与移动端美观度不得超过 60%。
6. 所有扣分必须能追溯到 `01-页面元素清单.md`、截图证据或具体需求原文，不允许主观写“基本还原”。

维度内部达标规则：

1. 每个评分维度都必须单独评分，维度内部不得用其他维度高分抵消；任一适用维度低于 95%，本次结果即为不达标。
2. 每个 P0/P1 页面必须逐页计算字段控件、可编辑性、列表/分页/统计、详情形式、操作反馈、交互链路、移动端/视觉等细节维度；任一适用维度低于 95%，该页面不得声明达标。
3. 对低于 95% 的维度，必须记录维度内扣分项、影响页面、原文依据、修复动作、复评截图和复评分数。
4. Codex 必须在可修复范围内自动修改 HTML 细节，重新打开页面，重新操作交互链路，重新截图，重新评分；循环直到该维度 >= 95%、总还原度 >= 95% 且 P0/P1 页面无关键缺失。
5. 如果因为需求原文缺失、工具不可用或截图条件不足导致维度无法修复到 95%，必须记录为阻塞或待确认，不能把该维度、细节还原度或总还原度写为达标。
6. 维度循环结束条件只能是“全部适用维度达标”或“存在明确阻塞并记录为不达标/有条件达标”，不能因为已修改一次就停止。

```text
要求：

10. 使用页面覆盖、字段控件、列表/详情/分页/统计、操作反馈、交互链路和视觉美观度评分矩阵证明是否达到 95%。
11. 如果页面组、总分、细节还原度、任一适用评分维度或任一 P0/P1 页面维度低于 95%，按维度扣分项自动修复 HTML 细节并重新操作、截图、评分，直到达标或记录明确阻塞。
12. 最终输出 5 个标准文档、html/index.html 和 截图证据；大需求可额外输出 页面组 中间文档。
```

## 14. 禁止事项

1. 禁止未读需求文档就直接实现 HTML。
2. 禁止只看文件名，不读页面规格。
3. 禁止把 `蓝湖UI缺少页面清单.md` 当成必需文件。
4. 禁止只做管理后台，不做需求中要求的小程序页面。
5. 禁止只做小程序，不做需求中要求的管理后台页面。
6. 禁止请求真实接口、真实 token、真实账号或真实敏感数据。
7. 禁止把静态 demo 当成真实业务功能交付。
8. 禁止直接引入 D2Admin/Vue 代码。
9. 禁止页面缺失但还原度写 95% 以上。
10. 禁止没有截图证据就声明页面已还原。
11. 禁止没有评分矩阵就声明“高度还原”。
12. 禁止只调整报告分数，不修复页面差异。
13. 禁止 HTML 依赖本地 dev server 才能打开。
14. 禁止大需求一次性加载所有页面规格后直接实现，必须按页面组拆分。
15. 禁止页面组未达 95% 就汇总为总 HTML 并声明达标。
16. 禁止只做页面大框架，不还原字段控件、可编辑性、详情形式、分页、头部统计、提示框和交互结果。
17. 禁止把所有字段都做成普通文本或普通输入框，必须按需求区分输入框、下拉框、日期时间框、上传、开关、只读、禁用和必填校验。
18. 禁止列表页缺少需求要求的展示字段、筛选项、分页、头部统计、空态、详情入口和行操作。
19. 禁止操作按钮点击后没有确认、成功、失败、loading、toast、状态变化或列表刷新等反馈。
20. 禁止详情、编辑、审核、驳回等流程只打开空弹窗或假按钮，必须形成连贯的静态交互链路。
21. 禁止移动端页面只像线框图，必须具备适合演示的间距、层级、按钮触达、底部操作栏、弹窗和状态反馈。
22. 禁止字段控件、详情形式、分页统计、操作反馈或交互链路存在关键缺失时，把总还原度或细节还原度写到 95% 以上。
23. 禁止没有原文定位的字段、按钮、弹窗、分页、统计或交互规则直接进入 HTML 实现。
24. 禁止未完成实现前细节自检清单就生成 HTML。
25. 禁止截图只覆盖默认态；关键详情、弹窗、异常态、操作后结果缺少截图时，不得声明对应细节已还原。

## 15. 后续真实开发衔接

静态 HTML demo 完成后，团队应先用 `html/index.html` 做产品、需求方和技术评审。

评审确认后，真实开发不要继续沿用本流程，而是切换到：

```text
docs/Codex需求到验收标准流程.md
```

真实开发阶段再输出：

```text
docs/技术方案/{yyyy-mm-dd}-{模块名}-tcdesign.md
docs/测试文档/{模块名}-testcase.md
docs/测试文档/{模块名}-testreport.md
docs/验收报告/{yyyy-mm-dd}-{模块名}-acceptance.md
docs/一致性校验/{yyyy-mm-dd}-{模块名}-consistency.md
```

静态 HTML 是需求对齐基线，不替代真实研发验收。
