# 量表全量版需求阻塞问题清单

## 1. 使用目的

本清单用于和甲方沟通“量表与测评全量版”需求边界。

当前甲方提供的管理后台 demo 路径为：

`/Users/外快/homeproject/home`

demo 中确实存在较完整的量表后台页面，包括名称管理、题目管理、维度管理、常模管理、公式管理、解释管理、发布管理、报告配置、岗位管理、婚恋性格匹配等。但这些页面目前主要是前端 mock 和界面演示，缺少关键业务闭环、数据规则和算法接口定义。

因此，如果甲方坚持做全量版，需要先补齐以下问题。否则研发无法准确估期、无法设计数据库、无法确定接口、无法联调推荐算法，也无法形成验收标准。

## 2. 核心结论

全量版不是“多做几个后台页面”，而是要建设一套通用测评平台，至少包含：

1. 动态量表配置。
2. 动态题目配置。
3. 动态维度配置。
4. 动态公式执行。
5. 动态常模计算。
6. 动态报告生成。
7. 动态发布和版本管理。
8. 动态测评结果回写用户画像。
9. 动态测评结果影响推荐算法。
10. 历史报告与历史公式版本回溯。

目前 demo 只证明“页面上有这些菜单”，不证明业务链路已经定义完整。

## 3. 一票否决级阻塞问题

以下问题不确认，无法进入全量版开发。

### 3.1 动态新增量表后，如何影响推荐算法？

demo 中支持量表新增、婚恋匹配方案、公式管理，但没有说明：

1. 新增一份量表后，哪些结果字段会进入推荐算法？
2. 推荐算法如何识别新量表的字段含义？
3. 新量表结果是数值、枚举、标签、向量，还是文本？
4. 这些字段参与召回、粗排、精排，还是只用于展示？
5. 权重由谁配置？
6. 权重修改后是立即生效，还是需要重新训练模型？
7. 如果用户未完成这份新增量表，推荐算法如何处理缺失值？
8. 如果量表停用，历史结果是否继续参与推荐？
9. 如果量表改版，旧版本结果是否继续参与推荐？
10. 推荐结果页是否要展示“因为你们某量表匹配所以推荐”的解释？

阻塞影响：

- 无法设计测评结果表。
- 无法设计推荐特征表。
- 无法设计算法输入接口。
- 无法判断动态新增量表是否真的能动态影响推荐。
- 无法验收“测评影响推荐算法”。

需要甲方补充：

| 需补充内容 | 说明 |
| --- | --- |
| 量表结果字段规范 | 每种量表输出哪些字段 |
| 推荐特征映射规则 | 哪些字段进入推荐算法 |
| 推荐权重配置规则 | 权重在哪里配、谁能配、如何生效 |
| 缺失值处理规则 | 用户没测时如何推荐 |
| 版本兼容规则 | 量表改版后历史结果怎么处理 |
| 推荐解释规则 | 是否展示推荐理由 |

### 3.2 动态公式到底如何执行？

demo 的公式管理页面目前是文本表达式，例如：

`总分 = R + I + A + S + E + C`

但全量版如果允许后台配置公式，就必须明确：

1. 公式使用什么语法？
2. 支持哪些函数？`SUM`、`AVG`、`MAX`、`IF`、`TOP3` 是否支持？
3. 变量如何命名？题目 ID、维度代码、常模字段怎么引用？
4. 公式保存时是否做语法校验？
5. 公式是否支持试算？
6. 公式执行失败时用户提交怎么办？
7. 公式修改后是否影响历史报告？
8. 谁有权限修改公式？
9. 公式是否需要审批后发布？
10. 公式版本如何回滚？

阻塞影响：

- 无法判断公式管理是“展示文本”还是“真正可执行引擎”。
- 无法设计公式解析器。
- 无法设计公式测试和发布流程。
- 公式错误会直接导致用户报告错误，是高风险功能。

需要甲方补充：

| 需补充内容 | 说明 |
| --- | --- |
| 公式语法说明 | 支持的表达式和函数 |
| 变量引用规则 | 题目、维度、常模如何引用 |
| 公式校验规则 | 保存前如何校验 |
| 公式试算规则 | 后台是否必须支持试算 |
| 公式版本规则 | 修改后是否生成新版本 |
| 公式审批规则 | 是否需要审核后上线 |

### 3.3 动态报告配置如何绑定数据？

demo 有报告配置页面，能拖组件、改背景、改主题色。但没有说明：

1. 报告组件如何绑定测评结果字段？
2. 图表数据从哪里来？
3. 文案如何按结果类型动态切换？
4. 不同量表是否共用模板？
5. 同一量表不同结果类型是否用不同模板？
6. 报告模板发布后，历史报告是否跟着变化？
7. 报告生成失败如何处理？
8. 图片、海报、分享卡片如何生成？
9. 移动端展示和 PDF/图片导出是否同一模板？
10. 用户隐藏报告后，后台报告是否仍可查看？

阻塞影响：

- 无法设计报告模板数据结构。
- 无法设计报告渲染服务。
- 无法判断报告配置是“页面装修”还是“数据驱动模板”。
- 无法验收后台配置后移动端是否正确展示。

需要甲方补充：

| 需补充内容 | 说明 |
| --- | --- |
| 报告数据绑定规范 | 组件绑定哪些字段 |
| 模板匹配规则 | 按量表、版本、结果类型如何选模板 |
| 历史报告规则 | 模板修改是否影响旧报告 |
| 失败处理规则 | 报告生成失败如何补偿 |
| 分享海报规则 | 是否需要生成图片 |

### 3.4 量表版本管理没有闭环

全量版一定会涉及版本，否则后台改题目、改公式、改报告会影响历史数据。

但 demo 没说明：

1. 什么操作会生成新版本？
2. 题目改了是否生成版本？
3. 公式改了是否生成版本？
4. 报告模板改了是否生成版本？
5. 旧版本是否允许继续作答？
6. 进行中的用户遇到新版本怎么办？
7. 历史报告按哪个版本展示？
8. 版本是否支持灰度？
9. 版本是否支持回滚？
10. 多版本数据如何统计？

阻塞影响：

- 无法保证历史报告可追溯。
- 用户进行中作答可能与题库版本不一致。
- 运营修改配置可能污染历史结果。

需要甲方补充：

| 需补充内容 | 说明 |
| --- | --- |
| 版本生成规则 | 哪些操作生成新版本 |
| 进行中处理规则 | 用户答到一半遇到新版本怎么办 |
| 历史报告规则 | 历史报告是否固定快照 |
| 灰度/回滚规则 | 是否必须支持 |

### 3.5 常模管理缺少专业口径

demo 有常模管理，但没有定义常模如何使用。

需要明确：

1. 哪些量表需要常模？
2. 常模按哪些人群分层？性别、年龄、学历、学校、地区？
3. 常模数据由谁提供？
4. 常模格式是什么？
5. 常模更新后是否影响历史报告？
6. 常模缺失时如何计算？
7. 常模是否参与推荐算法？
8. 常模是否需要专业审核？

阻塞影响：

- 无法设计常模数据结构。
- 无法实现标准分计算。
- 无法判断常模管理是否必要。

## 4. 高风险但可后置确认的问题

### 4.1 后台动态题目编辑是否允许直接上线？

如果允许运营直接改题：

1. 是否需要审批？
2. 是否需要预览？
3. 是否需要测试账号试答？
4. 是否生成新版本？
5. 是否支持批量导入回滚？

建议口径：

全量版可以支持题目配置，但必须有草稿、预览、发布、版本机制。不能编辑后直接影响线上。

### 4.2 测评结果是否允许用户隐藏？

需要明确：

1. 隐藏后主页是否不展示？
2. 隐藏后推荐算法是否继续使用？
3. 隐藏后合拍度页面是否可展示？
4. 心理健康结果是否永远不公开？

建议口径：

展示隐私和算法使用授权要分开，否则用户会认为隐藏后算法也不能使用。

### 4.3 心理健康量表的合规边界

如果全量版允许动态新增心理健康类量表，需要明确：

1. 是否涉及医疗诊断风险？
2. 是否需要专业机构审核？
3. 高风险结果是否通知平台？
4. 是否引导心理咨询？
5. 是否允许导出原始答案？

建议口径：

心理健康类结果默认不公开、不进推荐、不做医疗诊断，只提供自助建议。

### 4.4 数据导出权限不清晰

全量版涉及用户原始答案、心理健康结果、性格标签和推荐特征。

需要明确：

1. 谁能导出？
2. 是否脱敏？
3. 是否需要二次确认？
4. 是否记录用途？
5. 导出文件保留多久？

## 5. demo 页面与真实需求的差距

| demo 模块 | 当前看到的能力 | 缺少的闭环 |
| --- | --- | --- |
| 名称管理 | 可维护量表名称、编码、说明、状态 | 缺分类、版本、发布状态、配置完整度 |
| 题目管理 | 可看题目、选项、预览、导入导出 | 缺真实题型规则、反向计分、多维度归属、版本发布 |
| 维度管理 | 可维护维度和关联题目 | 缺维度与公式、报告、推荐特征的关系 |
| 常模管理 | 可录入 JSON | 缺常模格式、人群分层、计算规则 |
| 公式管理 | 可录入表达式文本 | 缺可执行语法、校验、试算、版本 |
| 解释管理 | 可配置区间和解释 | 缺区间冲突校验、结果命中规则 |
| 发布管理 | 可选择量表、用户、模块 | 缺版本发布、灰度、进行中用户处理 |
| 报告配置 | 可拖组件和设置样式 | 缺数据绑定、模板匹配、历史快照 |
| 岗位管理 | 可配置岗位和公式 | 缺职业推荐算法口径 |
| 婚恋性格匹配 | 可配方案和公式 | 缺推荐算法接口和在线生效机制 |

## 6. 推荐沟通口径

可以对甲方这样说：

> 我们不是不做全量版，而是全量版已经不是简单后台页面开发，而是一套通用测评平台。  
> 目前 demo 只展示了页面形态，没有给出公式执行、版本发布、报告数据绑定、推荐算法映射等关键闭环。  
> 如果这些不补齐，我们无法准确估期，也无法保证开发出来的东西符合甲方预期。  
> 建议甲方先补齐以上问题；在补齐前，我们只能按固定三套测评方案估期和推进。  
> 如果甲方坚持全量版，需要先确认新增工作量、延期风险和需求补充周期。

## 7. 可推动甲方妥协的替代方案

建议给甲方两个选择：

### 方案 A：固定测评版

本期固定三套测评：

1. 霍兰德职业测评。
2. 荣格认知八维测评。
3. 心理健康测评量表。

优点：

1. 题目、公式、报告固定，风险可控。
2. 可以快速上线。
3. 可以先验证用户是否愿意测、是否愿意看报告、是否提升推荐转化。
4. 后续有真实数据后再平台化。

### 方案 B：全量配置版

甲方需先补齐：

1. 动态公式语法。
2. 量表版本规则。
3. 报告模板数据绑定。
4. 推荐算法字段映射。
5. 常模格式。
6. 权限和审批规则。
7. 数据导出合规规则。

影响：

1. 需求澄清周期变长。
2. 数据库和架构复杂度上升。
3. 联调推荐算法时间增加。
4. 测试范围显著扩大。
5. 上线时间需要重新评估。

## 8. 建议优先要求甲方回答的问题

如果时间有限，优先逼甲方回答以下 10 个问题：

1. 动态新增量表后，如何决定它是否影响推荐算法？
2. 新量表结果字段如何映射成推荐算法特征？
3. 后台公式是纯文本展示，还是必须可执行？
4. 公式使用什么语法，是否支持试算和版本回滚？
5. 报告模板如何绑定结果字段？
6. 修改题目、公式、报告后，历史报告是否变化？
7. 用户作答到一半，后台发布新版本，如何处理？
8. 常模数据由谁提供，格式是什么？
9. 心理健康类量表是否允许导出原始答案？
10. 用户隐藏测评结果后，算法是否还能使用该结果？

这些问题不回答，全量版无法闭环。
