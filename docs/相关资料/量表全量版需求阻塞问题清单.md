# 量表全量版需求阻塞问题清单

## 1. 使用目的

本清单用于和甲方沟通“量表与测评全量版”需求边界。

当前甲方提供的管理后台 demo 路径为：

`/Users/外快/homeproject/home`

demo 中确实存在较完整的量表后台页面，包括名称管理、题目管理、维度管理、常模管理、公式管理、解释管理、发布管理、报告配置、岗位管理、婚恋性格匹配等。但这些页面目前主要是前端 mock 和界面演示，缺少关键业务闭环、数据规则和算法接口定义。

因此，如果甲方坚持做全量版，需要先补齐以下问题。否则研发无法准确估期、无法设计数据库、无法确定接口、无法联调推荐算法，也无法形成验收标准。

## 2. 核心结论

全量版不是“多做几个后台页面”，而是要建设一套通用测评平台，至少包含：

1. 动态量表配置。
2. 动态题目配置。
3. 动态维度配置。
4. 动态公式执行。
5. 动态常模计算。
6. 动态报告生成。
7. 动态发布和版本管理。
8. 动态测评结果回写用户画像。
9. 动态测评结果影响推荐算法。
10. 历史报告与历史公式版本回溯。

目前 demo 只证明“页面上有这些菜单”，不证明业务链路已经定义完整。

补充口径：

1. 本清单不是“建议优化项”，而是“启动前必须补齐的 P0 清单”。
2. 甲方未补齐前，研发不进入详细设计、不进入正式估期、不建表、不定接口、不联调算法。
3. 不接受“先按理解做，后面再补规则”的推进方式，否则后续返工一定发生在数据库、报告、推荐算法和验收阶段。
4. 任何一项核心口径如果只有页面 demo、截图、Excel 片段、口头描述，没有形成统一书面规则，视为“未提供”。

## 3. 一票否决级阻塞问题

以下问题不确认，无法进入全量版开发。

### 3.1 动态新增量表后，如何影响推荐算法？

demo 中支持量表新增、婚恋匹配方案、公式管理，但没有说明：

1. 新增一份量表后，哪些结果字段会进入推荐算法？
2. 推荐算法如何识别新量表的字段含义？
3. 新量表结果是数值、枚举、标签、向量，还是文本？
4. 这些字段参与召回、粗排、精排，还是只用于展示？
5. 权重由谁配置？
6. 权重修改后是立即生效，还是需要重新训练模型？
7. 如果用户未完成这份新增量表，推荐算法如何处理缺失值？
8. 如果量表停用，历史结果是否继续参与推荐？
9. 如果量表改版，旧版本结果是否继续参与推荐？
10. 推荐结果页是否要展示“因为你们某量表匹配所以推荐”的解释？

阻塞影响：

- 无法设计测评结果表。
- 无法设计推荐特征表。
- 无法设计算法输入接口。
- 无法判断动态新增量表是否真的能动态影响推荐。
- 无法验收“测评影响推荐算法”。

需要甲方补充：

| 需补充内容       | 说明                           |
| ---------------- | ------------------------------ |
| 量表结果字段规范 | 每种量表输出哪些字段           |
| 推荐特征映射规则 | 哪些字段进入推荐算法           |
| 推荐权重配置规则 | 权重在哪里配、谁能配、如何生效 |
| 缺失值处理规则   | 用户没测时如何推荐             |
| 版本兼容规则     | 量表改版后历史结果怎么处理     |
| 推荐解释规则     | 是否展示推荐理由               |

### 3.2 动态公式到底如何执行？

demo 的公式管理页面目前是文本表达式，例如：

`总分 = R + I + A + S + E + C`

但全量版如果允许后台配置公式，就必须明确：

1. 公式使用什么语法？
2. 支持哪些函数？`SUM`、`AVG`、`MAX`、`IF`、`TOP3` 是否支持？
3. 变量如何命名？题目 ID、维度代码、常模字段怎么引用？
4. 公式保存时是否做语法校验？
5. 公式是否支持试算？
6. 公式执行失败时用户提交怎么办？
7. 公式修改后是否影响历史报告？
8. 谁有权限修改公式？
9. 公式是否需要审批后发布？
10. 公式版本如何回滚？

阻塞影响：

- 无法判断公式管理是“展示文本”还是“真正可执行引擎”。
- 无法设计公式解析器。
- 无法设计公式测试和发布流程。
- 公式错误会直接导致用户报告错误，是高风险功能。

需要甲方补充：

| 需补充内容   | 说明                     |
| ------------ | ------------------------ |
| 公式语法说明 | 支持的表达式和函数       |
| 变量引用规则 | 题目、维度、常模如何引用 |
| 公式校验规则 | 保存前如何校验           |
| 公式试算规则 | 后台是否必须支持试算     |
| 公式版本规则 | 修改后是否生成新版本     |
| 公式审批规则 | 是否需要审核后上线       |

需要甲方补充：

| 需补充内容       | 说明                             |
| ---------------- | -------------------------------- |
| 报告数据绑定规范 | 组件绑定哪些字段                 |
| 模板匹配规则     | 按量表、版本、结果类型如何选模板 |

全量版一定会涉及版本，否则后台改题目、改公式、改报告会影响历史数据。

但 demo 没说明：

3. 公式改了是否生成版本？
4. 报告模板改了是否生成版本？

7. 历史报告按哪个版本展示？
8. 版本是否支持灰度？
9. 版本是否支持回滚？
10. 多版本数据如何统计？

阻塞影响：

- 无法保证历史报告可追溯。
- 用户进行中作答可能与题库版本不一致。
- 运营修改配置可能污染历史结果。

需要甲方补充：

| 需补充内容     | 说明                         |
| -------------- | ---------------------------- |
| 版本生成规则   | 哪些操作生成新版本           |

### 3.6 量表唯一标识与派生关系未定

结论：

1. 同一份资料如果存在多个名称，例如 `爱情价值观问卷` 与 `LAS-R 爱情态度量表修订版（23题）`，必须明确它们在系统里是“同一量表的不同名称”，还是“不同量表”。
2. 荣格八维与 MBTI 的关系也必须定稿：是“同一量表的派生结果”，还是“两份独立量表”；如果是派生结果，必须明确派生规则、同分规则和版本继承规则。
3. 首批上线哪些量表可以放到运营发布阶段确认，但至少需要一套标准样例量表贯通题库、维度、公式、解释、报告、发布和推荐映射。

阻塞影响：

- 无法处理同一量表多名称导入、复制、升级、回溯。
- 荣格八维结果无法稳定派生 MBTI 并回写推荐特征。
- 验收时无法判断两份名称不同的资料是不是同一套量表。

需要甲方补充：

| 需补充内容           | 说明                                                            |
| -------------------- | --------------------------------------------------------------- |
| 量表别名/展示名规则  | 学术名称、产品名称、后台名称是否允许并存，以及主从关系          |
| 同名/别名判定规则    | `爱情价值观问卷` 与 `LAS-R 爱情态度量表修订版` 在系统中如何归一 |
| 荣格与 MBTI 关系定义 | 是否派生、如何派生、同分时如何判定                              |
| 标准样例量表         | 至少一套完整样例，用于验证配置化链路能否跑通                    |

### 3.7 题库原始资料不具备直接开发条件

现有资料虽然有 docx/xlsx，但距离“可直接开发”还差一个统一、可核对、可导入的主数据包。至少缺以下确定项：

1. 每份量表的正式题库最终版是哪一份文件？
2. 题号是否固定？后续是否允许重排？
3. 每题题型是什么？单选、多选、量尺、排序、填空，是否全部已经确定？
4. 每题是否必答？
5. 每题选项的展示文案、存储值、分值、排序是否固定？
6. 是否存在反向计分题？如果有，反向规则是选项反转还是公式层处理？
7. 一题是否允许归属多个维度？
8. 说明题、练习题、非计分题是否属于正式题库的一部分？
9. 每页题数、是否自动下一题、是否可返回上一题，这些是量表级配置还是发布级配置？
10. 题目文案是否允许在不同场景展示不同版本，例如职业场景、婚恋场景、知音场景？
11. 图片题、插图题、带示例说明的题目是否存在？资源由谁提供？
12. 导入模板字段顺序、唯一键、覆盖策略、报错规则没有定义。

阻塞影响：

- 无法设计题目表、选项表、题目版本表。
- 无法设计 Excel 导入模板和校验逻辑。
- 无法实现作答页题型渲染。
- 无法保证同一道题在前后端、报告、算法中的编号一致。
- 无法保证后续导入不会覆盖线上题库。

需要甲方补充：

| 需补充内容   | 说明                                                     |
| ------------ | -------------------------------------------------------- |
| 题库主数据包 | 每份量表一份最终题库，需可导入，不接受多份候选同时生效   |
| 题目字段清单 | 题号、题型、题干、选项、分值、反向计分、维度归属、必答等 |
| 导入口径     | 以题号、编码还是内部主键作为覆盖依据                     |
| 文案版本口径 | 是否允许同题多场景文案                                   |
| 非计分题规则 | 练习题、说明题、引导题如何存储和发布                     |

### 3.8 结果结构与字段语义没有固化

每种量表最终产出什么字段。

需要明确：

1. 原始分、均分、标准分、百分比、区间等级、主类型之间的区别是什么？
2. 每种量表哪些字段是对用户展示，哪些字段只给算法用，哪些字段只用于后台复核？
3. `resultType` 是否只有一个？如果出现双类型、并列类型、Top3 类型怎么办？
4. `resultTags` 是公式产出、解释管理产出，还是人工配置产出？
5. `confidence` 如何计算？由反应时、漏答率、极端作答、重复作答哪几项组成？
6. 风险等级是否所有量表都有？还是仅心理健康类有？
7. 维度分的范围是否统一归一化，例如 0-100、1-5、T 分、百分位？
8. 报告中显示的分数和算法消费的分数是否同一套值？
9. 同一量表是否同时产出“用户可见结果”和“推荐内部特征结果”两套字段？
10. 历史报告和用户画像中保留的是快照值，还是实时重算值？

阻塞影响：

- 无法设计报告结果表。
- 无法定义用户画像字段。
- 无法定义推荐特征表。
- 无法定义报告模板可绑定的字段集合。
- 无法约束不同量表输出结构的一致性。

需要甲方补充：

| 需补充内容    | 说明                                   |
| ------------- | -------------------------------------- |
| 量表结果协议  | 每份量表输出哪些结构化字段             |
| 字段语义说明  | 每个字段的单位、范围、含义、是否可为空 |
| 展示/算法边界 | 哪些字段给用户看，哪些字段仅内部使用   |
| 低置信度规则  | 如何判断结果不稳定或需重测             |
| 历史快照规则  | 报告、画像、推荐分别保存什么快照       |

### 3.9 婚恋匹配方案与推荐算法口径互相打架

现有资料已经出现多套相似但不完全一致的口径：

1. 推荐资料写的是 `40%CB2 + 60%性格`。
2. 性格内部又拆成依恋 35%、沟通 30%、荣格八维 20%、爱情价值观 15%。
3. 组合匹配资料又给出了双量表、三量表的不同组合权重。
4. PRD-08 后台文档里写了 `recommend.personality.weight.*` 一组配置项。
5. 但没有定义这些权重是只用于冷启动、规则排序、互惠排序，还是最终精排前的 pair feature。
6. 也没有定义“婚恋性格匹配”页面配置的是元数据、静态规则，还是线上实时调用的正式算法方案。
7. 更没有定义当算法模型版本和婚恋匹配方案版本同时变更时，最终以谁为准。

需要明确：

1. 合拍度页面、合拍的人列表、觅缘推荐流是否共用同一套匹配公式？
2. 双人匹配计算是纯规则、规则+模型、还是模型回写后再拼装解释？
3. 权重是固定写死、后台配置，还是仅算法服务内部配置？
4. 匹配理由标签由谁生成？公式、解释管理还是算法服务？
5. 未完成部分心印量表时，是否允许按降级组合计算？
6. 男女是否用同一套公式？如果不同，差异由谁定义？
7. 是否按会员等级、城市、人群包、AB 实验使用不同方案？

阻塞影响：

- 无法定义婚恋匹配表结构。
- 无法定义推荐接口入参出参。
- 无法确定“婚恋性格匹配”后台页面到底配置什么。
- 无法验证推荐理由和合拍度是否可回溯。
- 无法验收推荐效果到底来自哪套规则。

需要甲方补充：

| 需补充内容       | 说明                                   |
| ---------------- | -------------------------------------- |
| 匹配方案总口径   | CB2、性格、互惠、CatBoost 各自角色     |
| 方案生效边界     | 哪些页面、哪些场景使用哪套方案         |
| 权重唯一来源     | 后台配置、算法服务或固定规则，必须唯一 |
| 降级计算规则     | 缺少部分量表时如何算合拍度             |
| 推荐理由生成规则 | 标签来源、展示条件、是否对用户可见     |

### 3.10 解释、建议、图片素材的来源和授权不清

现有资料目录不仅有题库，还有多套解释表、知音情境解释、职业情境解释、恋爱情境解释、男女图片素材等，但没有确认这些内容如何使用。

需要明确：

1. 哪份解释表对应哪个量表、哪个场景、哪个版本？
2. 同一量表是否区分职业场景解释、婚恋场景解释、知音场景解释？
3. 性别差异素材是否一定要分男女展示？若用户未填写性别或性别策略调整怎么办？
4. 报告里的长文案、短标签、推荐理由、海报文案是否来自同一套解释库？
5. 图片、插图、人格海报、分享图是否已有版权授权？
6. 是否允许我们对原始题库、解释、图片做删改、改写、拆分？
7. 解释内容是否经过专业审核？如果没有，谁承担专业正确性责任？
8. 高风险或敏感结论是否需要免责声明、风险提示或人工复核？

阻塞影响：

- 无法设计解释表和建议表。
- 无法决定报告模板的数据绑定方式。
- 无法确认分享海报和报告配图如何生成。
- 无法确认内容是否可合法上线。
- 无法形成验收标准，因为“展示哪套解释”都未确定。

需要甲方补充：

| 需补充内容     | 说明                                   |
| -------------- | -------------------------------------- |
| 解释素材映射表 | 每份解释素材对应哪个量表、哪个场景     |
| 素材授权说明   | 题库、解释、图片、海报是否可商用和改编 |
| 专业审核说明   | 谁确认解释内容正确可上线               |
| 性别/场景策略  | 男女版、知音版、恋爱版如何切换         |
| 文案层级定义   | 长文、短文、标签、理由分别从哪里取     |

### 3.11 作答生命周期、重测、过期、幂等规则不完整

断点续答、过期重测、重复提交幂等等方向。

需要明确：

1. 进度保存频率是每题、每页，还是定时批量保存？
2. 用户切设备、切账号、网络中断后，进度以服务端还是本地为准？
3. 进行中作答遇到题库变更、量表下架、发布下线、前置条件失效时分别怎么处理？
4. 重复点击提交的幂等键是什么？
5. 报告生成失败后，用户看到“生成中”最多持续多久，何时判失败？
6. 用户可以多久重测一次？是否有限次、冷却期、审核条件？
7. 重测后旧报告是否仍可查看、是否仍参与推荐、是否仍展示主页？
8. 量表过期规则按自然日、完成时间、版本时间，还是运营配置时间判断？
9. 作答过程中是否允许跳题、回看、改答案？不同量表是否不同？
10. 异常作答被判低置信度后，是只打标、提示重测，还是不允许生成正式报告？

阻塞影响：

- 无法设计进度表和提交状态机。
- 无法保证幂等和报告唯一性。
- 无法确定过期与重测逻辑。
- 无法处理线上版本切换带来的中断作答。
- 无法界定历史结果在推荐里的有效性。

需要甲方补充：

| 需补充内容     | 说明                                                   |
| -------------- | ------------------------------------------------------ |
| 作答状态机     | 未开始、进行中、已提交、处理中、成功、失败、过期等状态 |
| 进度优先级规则 | 本地缓存与服务端冲突如何处理                           |
| 重测规则       | 冷却期、权限、次数、是否保留旧报告                     |
| 幂等规则       | 重复提交如何识别和返回                                 |
| 异常作答规则   | 低置信度、极短作答、全同选项如何处理                   |

### 3.12 发布链路、前置关系、目标用户规则没有写实

发布管理不只是“选几个量表上线”，而是要决定用户能否看到、何时看到、为什么能看到。

需要明确：

1. 发布对象是量表版本还是量表主档？
2. 多个量表组合发布时，用户列表页按发布单显示，还是按量表显示？
3. 前置条件是“完成量表 A”“完成发布任务 A”，还是“报告达到某结果”？
4. 未解锁量表是直接不展示，还是展示卡片但置灰？
5. 目标用户是全量、标签人群、学校、性别、认证状态、会员等级，还是运营圈选？
6. 同一量表能否同时出现在成家、立业、我的多个入口？
7. 开始时间到了但题库没发布完成，系统如何处理？
8. 发布下线后，历史入口、历史报告、分享链接如何处理？
9. 是否允许灰度到部分人群？灰度失败如何回滚？
10. 导购说明、测评简介、入口文案是跟量表走，还是跟发布单走？

阻塞影响：

- 无法设计发布表、发布明细表、用户命中规则。
- 无法设计移动端列表接口。
- 无法实现前置解锁和未解锁提示。
- 无法控制灰度和回滚。
- 无法定义入口文案和量表主档的边界。

需要甲方补充：

| 需补充内容   | 说明                               |
| ------------ | ---------------------------------- |
| 发布对象定义 | 发布单绑定主档、版本还是配置快照   |
| 用户命中规则 | 目标用户如何计算                   |
| 前置解锁规则 | 前置条件的计算口径                 |
| 灰度规则     | 是否必须支持、按什么维度灰度       |
| 入口绑定规则 | 模块入口、导购文案、量表简介的归属 |

### 3.13 数据授权、敏感信息和算法使用授权没有闭环

量表数据尤其是心理健康、依恋、沟通模式、本人价值观，已经属于高敏感度数据，不能只写一句“用户可隐藏报告”就算完成。

需要明确：

1. 用户是否单独授权“展示给他人”和“参与推荐算法”这两个权限？
2. 用户隐藏报告后，算法是否还能继续使用历史结果？
3. 用户注销后，原始答案、报告快照、推荐特征是否删除、匿名化还是保留审计？
4. 心理健康结果是否允许导出原始答案？允许到什么粒度？
5. 是否存在未成年人用户？如果存在，哪些量表禁止使用？
6. 是否需要弹窗确认“本报告不构成医学/心理诊断”？
7. 高风险心理结果是否要触发平台告警、客服介入、帮助热线？
8. 后台谁能看原始答案、谁能看报告、谁能导出、谁能改公式？
9. 导出是否默认脱敏？脱敏字段有哪些？
10. 推荐理由对用户展示时，是否会暴露对方未公开的测评信息？

阻塞影响：

- 无法设计权限模型。
- 无法设计用户授权记录表。
- 无法确定推荐算法是否可合法使用这些特征。
- 无法确定客服/运营后台可见范围。
- 无法规避心理健康和婚恋敏感标签的上线风险。

需要甲方补充：

| 需补充内容       | 说明                           |
| ---------------- | ------------------------------ |
| 用户授权规则     | 展示授权、算法授权是否分开     |
| 敏感数据访问矩阵 | 哪些角色可看什么数据           |
| 注销和删除规则   | 数据保留周期和匿名化策略       |
| 心理健康处理规则 | 高风险提示、导出限制、免责声明 |
| 推荐暴露边界     | 推荐理由是否可引用未公开结果   |

### 3.14 没有验收样例就无法验收全量版

量表平台最怕“功能都做了，但甲方说结果不对”。如果没有标准答案包和标准报告样例，后续无法验收。

需要明确：

1. 每份量表是否提供至少 3-5 组标准答案样例？
2. 每组样例的预期输出是否包括原始分、维度分、标准分、结果类型、标签、建议、报告文案？
3. 双人匹配是否提供标准样例，例如 A 用户与 B 用户输入后应得多少合拍分、命中什么理由标签？
4. 公式试算是否要求与甲方 Excel 结果逐项一致？
5. 报告页验收是按字段正确、样式近似，还是必须像素级一致？
6. 推荐算法联动验收是验证字段已回写，还是验证推荐效果提升？
7. 历史版本回溯验收如何做？是否需要保留旧题库、旧报告样例？
8. 导入导出验收是否需要甲方提供真实模板和真实错误样本？

阻塞影响：

- 无法做单元测试和联调测试。
- 无法判断公式算错还是甲方期望变了。
- 无法确认报告生成是否正确。
- 无法确认推荐联动是否达标。
- 项目会在验收阶段反复扯皮。

需要甲方补充：

| 需补充内容     | 说明                                 |
| -------------- | ------------------------------------ |
| 标准答案包     | 每份量表至少若干组样例输入输出       |
| 双人匹配样例   | 合拍度和理由标签样例                 |
| 报告验收样例   | 标准报告字段和值                     |
| Excel 对照口径 | 是否以甲方 Excel 为最终基准          |
| 联动验收口径   | 推荐联动验收看字段、看解释还是看效果 |

## 4. 高风险问题

以下问题理论上可后置，但在本项目当前阶段不建议后置。若甲方继续模糊处理，仍然不建议启动全量版开发。

### 4.1 后台动态题目编辑是否允许直接上线？

如果允许运营直接改题：

1. 是否需要审批？
2. 是否需要预览？
3. 是否需要测试账号试答？
4. 是否生成新版本？
5. 是否支持批量导入回滚？

建议口径：

全量版可以支持题目配置，但必须有草稿、预览、发布、版本机制。不能编辑后直接影响线上。

### 4.2 测评结果是否允许用户隐藏？

需要明确：

1. 隐藏后主页是否不展示？
2. 隐藏后推荐算法是否继续使用？
3. 隐藏后合拍度页面是否可展示？
4. 心理健康结果是否永远不公开？

建议口径：

展示隐私和算法使用授权要分开，否则用户会认为隐藏后算法也不能使用。

### 4.3 心理健康量表的合规边界

如果全量版允许动态新增心理健康类量表，需要明确：

1. 是否涉及医疗诊断风险？
2. 是否需要专业机构审核？
3. 高风险结果是否通知平台？
4. 是否引导心理咨询？
5. 是否允许导出原始答案？

建议口径：

心理健康类结果默认不公开、不进推荐、不做医疗诊断，只提供自助建议。

### 4.4 数据导出权限不清晰

全量版涉及用户原始答案、心理健康结果、性格标签和推荐特征。

需要明确：

1. 谁能导出？
2. 是否脱敏？
3. 是否需要二次确认？
4. 是否记录用途？
5. 导出文件保留多久？

## 5. demo 页面与真实需求的差距

| demo 模块    | 当前看到的能力                   | 缺少的闭环                                     |
| ------------ | -------------------------------- | ---------------------------------------------- |
| 名称管理     | 可维护量表名称、编码、说明、状态 | 缺分类、版本、发布状态、配置完整度             |
| 题目管理     | 可看题目、选项、预览、导入导出   | 缺真实题型规则、反向计分、多维度归属、版本发布 |
| 维度管理     | 可维护维度和关联题目             | 缺维度与公式、报告、推荐特征的关系             |
| 常模管理     | 可录入 JSON                      | 缺常模格式、人群分层、计算规则                 |
| 公式管理     | 可录入表达式文本                 | 缺可执行语法、校验、试算、版本                 |
| 解释管理     | 可配置区间和解释                 | 缺区间冲突校验、结果命中规则                   |
| 发布管理     | 可选择量表、用户、模块           | 缺版本发布、灰度、进行中用户处理               |
| 报告配置     | 可拖组件和设置样式               | 缺数据绑定、模板匹配、历史快照                 |
| 岗位管理     | 可配置岗位和公式                 | 缺职业推荐算法口径                             |
| 婚恋性格匹配 | 可配方案和公式                   | 缺推荐算法接口和在线生效机制                   |

影响：

1. 需求澄清周期变长。
2. 数据库和架构复杂度上升。
3. 联调推荐算法时间增加。
4. 测试范围显著扩大。
5. 上线时间需要重新评估。
6. 在上述资料补齐前，不进入开发排期。
