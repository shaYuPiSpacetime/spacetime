## 2. 模块清单与作用

| 模块 | demo 路径 | 作用 |
| --- | --- | --- |
| 名称管理 | `/admin/scale` | 维护量表基础信息，包括量表名称、编码、引导说明、备用名称、创建人、创建时间、启停状态。是其他配置的主数据入口。 |
| 题目管理 | `/admin/scale/question` | 维护某个量表下的题目、题型、必答、选项与选项分值，支持说明页、练习题、移动端/PC 预览、导入导出。当前题型有单选、多选、排序、填空。 |
| 维度管理 | `/admin/scale/dimension` | 维护量表的计分维度，例如霍兰德 R/I/A/S/E/C、荣格八维等；维度可关联题目，是公式和解释的基础。 |
| 常模管理 | `/admin/scale/norm` | 按量表维护常模字段和 JSON 常模数据，例如平均分、标准差、分位数等，用于把原始分转换为标准分或等级。 |
| 公式管理 | `/admin/scale/formula` | 按量表和维度维护计分表达式，例如 `T = 50 + 10Z`、总分公式、职业倾向判断公式。是提交作答后生成结果的计算层。 |
| 解释管理 | `/admin/scale/interpretation` | 按量表、分类、解释类型和临界值范围维护结果解释文案，例如维度解释、类型解释、等级解释。 |
| 发展与建议管理 | `/admin/scale/development` | 维护结果后的行动建议、成长建议、改进建议，通常用于报告里的“下一步建议”。 |
| 发布管理 | `/admin/scale/publish` | 把一个或多个量表组合成发布任务，配置每页题数、起止时间、作答前置逻辑、展示模块、目标用户，并查看用户完成进度和报告。 |
| 报告列表 | `/admin/scale/report/list` | 管理报告模板清单，区分性格、职业、婚恋、知音等类型，支持启停、草稿、编辑。 |
| 报告配置 | `/admin/scale/report/config` | 可视化搭建报告页面，配置页面尺寸、背景、主题色，以及标题、文本、图片、标签、卡片、匹配推荐、图表等组件。 |
| 岗位管理 | `/admin/scale/position` | 面向“立业/职业”场景，维护岗位、岗位代码、关联测评和岗位匹配公式，用测评结果推荐职业/岗位。 |
| 导购说明 | `/admin/scale/guide` | 按模块维护引导文案，当前模块包括事业有成、高山流水、心印测试。 |
| 婚恋性格匹配 | `/admin/scale/dating-match` | 面向“成家/心印”场景，配置婚恋匹配方案、关联测评、适用场景、匹配公式和状态。是量表结果进入推荐算法的最直接入口雏形。 |
| 成长管理 | `/admin/scale/growth` | 当前页面显示“功能开发中”，暂未形成有效配置。 |

## 3. 核心业务关系

量表后台可以按下面的链路理解：

1. `名称管理` 定义量表主档。
2. `题目管理` 维护量表的作答内容和选项分值。
3. `维度管理` 把题目归类到能力、人格、职业倾向或关系维度。
4. `常模管理` 提供标准化所需的参考数据。
5. `公式管理` 根据题目分值、维度和常模计算原始分、标准分、总分、类型和标签。
6. `解释管理`、`发展与建议管理` 把分数和类型转成用户可读的报告文案。
7. `报告配置` 决定报告如何展示，`报告列表` 管理模板状态。
8. `发布管理` 决定哪些量表在什么时间、给哪些用户、在哪些模块可作答。
9. `岗位管理` 消费职业类量表结果，输出岗位/职业方向。
10. `婚恋性格匹配` 消费心印/人格/依恋/价值观类量表结果，输出用户间匹配分或匹配方案。

可以把它抽象为：

```text
量表主档
  -> 题目与选项
  -> 维度归属
  -> 常模与公式
  -> 分数 / 类型 / 标签
  -> 解释文案 / 发展建议 / 报告模板
  -> 发布到移动端模块
  -> 职业推荐或婚恋推荐算法消费
```

## 4. 与移动端功能的对应关系

| 移动端场景 | 后台支撑模块 | 说明 |
| --- | --- | --- |
| 心印测试列表 | 名称管理、发布管理、导购说明 | 决定用户看到哪些测评、测评名称、引导文案和展示模块。 |
| 测评作答页 | 题目管理、发布管理 | 决定题目、题型、选项分值、每页题数、是否有前置测评逻辑。 |
| 测评结果页 | 公式管理、常模管理、解释管理、发展与建议管理、报告配置 | 作答提交后生成分数、类型、标签、解释、建议和报告页面。 |
| 个人主页展示性格标签 | 公式管理、解释管理、报告配置 | 需要把测评结果沉淀成可复用标签字段。 |
| 事业有成/立业报告 | 岗位管理、报告配置、解释/建议 | 职业类量表结果用于岗位或职业方向推荐。 |
| 觅缘推荐流 | 婚恋性格匹配、推荐算法配置 | 心印/婚恋测评结果应进入推荐算法特征体系。当前缺真实链路。 |

## 5. 推荐算法关联现状

已有 PRD 明确写到：

- 心印测试：结果影响觅缘推荐权重，并在多模块复用。
- 立业/职业测评：结果不影响交友推荐，仅作为个人主页补充展示。
- 推荐算法 PRD 的静态属性特征中包含 `MBTI`，并描述了互惠算法、CatBoost 精排、SHAP 解释。
- 后台 PRD-08 已有推荐权重、阈值、规则配置，但没有把“量表结果字段”细化成可配置算法特征。

demo 代码里的关联雏形主要是 `/admin/scale/dating-match`：

- 可配置匹配方案名称和代码，例如 `DM-JUNG-COMP`、`DM-MBTI-CLASSIC`、`DM-FULL-DEPTH`。
- 可关联多份测评，例如荣格八维、MBTI、成人依恋、婚恋价值观、择偶偏好问卷。
- 可配置适用场景：心印合拍、智能推荐、通用。
- 可配置匹配公式，例如 `TypeComp + AttachMatch + ValueAlign`、`DeepScore` 等。

但当前还缺少关键闭环：

1. 测评作答提交后，如何生成稳定的算法特征字段。
2. 每个量表结果字段是否参与推荐、参与哪个推荐场景、权重是多少。
3. 两个用户之间如何从单人测评结果计算“双向匹配分”。
4. 婚恋匹配方案如何被线上推荐服务选择和调用。
5. 推荐排序结果如何回写解释，例如“你们价值观高度一致”“依恋类型互补”。

## 6. 建议补充的数据契约

为了让量表结果真正影响推荐算法，建议把测评结果拆成三层输出。

### 6.1 单人测评结果

作答完成后，每个用户每份量表至少沉淀：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| `userId` | `10001` | 用户 ID。 |
| `scaleId` / `scaleCode` | `Jung`、`MBTI` | 量表标识。 |
| `scaleVersion` | `v1.0.3` | 量表版本，避免公式变化污染历史结果。 |
| `dimensionScores` | `{ "Ti": 62, "Fe": 48 }` | 各维度标准分或归一化分。 |
| `rawScores` | `{ "Ti": 18 }` | 原始分，便于复核和重新计算。 |
| `resultType` | `INTJ`、`安全型`、`AES` | 主类型或结论。 |
| `resultTags` | `["理性决策", "长期主义"]` | 可展示、可搜索、可推荐消费的标签。 |
| `confidence` | `0.86` | 结果置信度，低置信度可降低推荐权重。 |
| `completedAt` | `2026-05-21 12:00:00` | 完成时间。 |
| `visibleOnProfile` | `true` | 是否展示到主页，只影响展示，不应默认影响算法。 |

### 6.2 推荐特征映射

需要在后台补一个“量表结果字段 -> 推荐特征”的映射能力：

| 配置项 | 示例 | 说明 |
| --- | --- | --- |
| 结果来源 | `scaleCode=MBTI,resultType` | 哪个量表的哪个结果字段。 |
| 特征名 | `feature_mbti_type` | 推荐算法读取的字段名。 |
| 特征类型 | 类别 / 数值 / 多标签 / 向量 | 决定算法处理方式。 |
| 生效场景 | 觅缘 / 精选 / 心印合拍 | 职业类结果默认不进交友推荐。 |
| 默认权重 | `0.12` | 冷启动或规则排序时使用。 |
| 缺失值策略 | 不加分 / 走默认值 / 引导补测 | 用户未测时如何处理。 |
| 有效期 | 永久 / 180 天 / 需重测 | 防止旧结果长期影响推荐。 |

### 6.3 双人匹配结果

推荐算法更需要的是“用户 A 与用户 B 的成对特征”：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| `pairFeature.mbtiCompatibility` | `0.78` | MBTI/荣格类型适配度。 |
| `pairFeature.valueAlignment` | `0.83` | 婚恋价值观一致性。 |
| `pairFeature.attachmentFit` | `0.66` | 依恋类型适配度。 |
| `pairFeature.dimensionDistance` | `{ "anxiety": 0.2 }` | 维度差异。 |
| `pairFeature.matchReasonTags` | `["价值观一致", "沟通风格互补"]` | 推荐理由和报告复用。 |
| `pairFeature.planCode` | `DM-MBTI-CLASSIC` | 使用的婚恋匹配方案。 |

## 7. 推荐链路中的建议位置

建议量表结果按场景进入不同阶段：

| 推荐阶段 | 是否使用量表结果 | 建议用法 |
| --- | --- | --- |
| 准入/硬过滤 | 谨慎使用 | 不建议因为人格类型直接过滤用户，除非甲方明确设定强禁配规则。 |
| 冷启动召回 | 强烈建议使用 | 新用户行为少，心印/价值观/依恋结果可以显著提升初始推荐质量。 |
| 粗排/互惠排序 | 强烈建议使用 | 计算双向人格适配、价值观一致、依恋适配等成对特征。 |
| CatBoost 精排 | 建议使用 | 将单人结果和成对匹配分作为特征，由模型学习真实转化贡献。 |
| 推荐解释 | 强烈建议使用 | 用 `matchReasonTags` 生成“为什么推荐 ta”。 |
| 职业/岗位推荐 | 单独使用 | 使用职业类量表结果，不混入交友推荐。 |

## 8. 当前 demo 与需求的差距

1. 缺后端量表实体：当前是前端 mock，未看到量表、题目、选项、维度、常模、公式、报告、发布任务的数据表设计。
2. 缺版本机制：PRD 要求版本号和灰度发布，demo 的名称管理里没有完整版本字段。
3. 缺作答记录与断点续答：demo 有题目配置和预览，但没有用户答题进度、反应时、提交记录。
4. 缺公式执行引擎：公式目前是文本表达式，未看到可执行表达式语法、变量校验、测试计算。
5. 缺结果标签回写：没有明确把测评结果写入用户画像、主页标签、推荐特征。
6. 缺推荐特征映射：`婚恋性格匹配` 有公式配置雏形，但没有与推荐算法服务的接口协议。
7. 缺算法方案发布：没有指定线上启用哪个匹配方案、是否按人群/VIP/实验分流。
8. 缺数据监控：未看到量表完成率、报告生成成功率、测评对推荐转化提升的监控。
9. 题型有差异：旧 PRD 提到量尺题 1-7 分，demo 题型列表暂未看到独立量尺题，需确认是否用单选替代。

## 9. 需要甲方确认的问题

1. 哪些量表属于“心印/婚恋推荐输入”，哪些只做展示或职业推荐？
2. 心印测试是否为进入觅缘推荐的前置条件，还是只作为加权项？
3. 用户未完成测评时，推荐算法是降权、默认值、还是引导先测评？
4. MBTI、荣格八维、依恋、婚恋价值观之间是否有主次权重？
5. 是否允许后台配置“禁配规则”，例如某些依恋组合不推荐？
6. 推荐理由是否需要对用户可见，还是只给运营/客服排查？
7. 测评结果是否允许用户隐藏；隐藏后是只隐藏展示，还是也不参与推荐？
8. 测评结果是否有有效期，过期后是否要求重测？
9. 婚恋匹配方案是否需要按会员等级、城市、人群、AB 实验切换？
10. 职业类量表结果是否绝对不影响交友推荐，还是允许作为主页补充标签轻度参与？

## 10. 建议后续 PRD 增补

建议新增一章“量表结果与推荐算法接口”，至少包含：

1. 量表结果标准输出结构。
2. 结果标签与用户画像字段映射。
3. 推荐算法可消费的特征清单。
4. 单人特征、双人匹配特征的计算口径。
5. 婚恋匹配方案的发布、启停、灰度和回滚规则。
6. 未测、过期、隐藏、重测、多版本结果的处理规则。
7. 推荐解释与报告文案复用规则。
8. 测评对推荐效果的监控指标，例如完成测评用户的右滑率、匹配率、消息转化率、投诉率。
