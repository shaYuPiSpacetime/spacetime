# 开发前甲方必备资料与阻塞项清单

## 1. 使用目的

本清单用于进入开发前向甲方确认：哪些资料、账号、资质、规则、内容和第三方能力必须先提供，否则会影响研发启动、接口联调、提审上线或验收。

项目定位：成家立业是面向大学生的社交与成长平台，包含匹配交友、兴趣社交、职业测评、社区互动、消息私信、会员/成家币等能力。该项目不是普通展示型小程序，涉及社交、婚恋、陌生人互动、UGC、支付、认证、心理/测评数据等高风险能力。

## 2. 阻塞等级定义


| 等级  | 含义                       | 处理原则       |
| --- | ------------------------ | ---------- |
| P0  | 不提供则无法进入正式开发或无法确定技术方案    | 必须在开发启动前确认 |
| P1  | 不提供可先写 mock，但会阻塞联调、测试或提审 | 必须在联调前确认   |
| P2  | 不提供不阻塞编码，但会影响上线、运营或验收    | 必须在上线前确认   |


## 3. P0：不提供无法进入正式开发

### 3.1 最终功能范围与版本边界

甲方必须确认：

1. 首版到底采用“固定测评版”还是“全量量表配置版”。
2. 社区动态、评论、朋友模块是否首版上线。
3. 会员、成家币、付费解锁是否首版上线。
4. 推荐算法首版是规则/冷启动，还是直接接入模型服务。
5. 心理健康测评是否首版上线。
6. 是否支持真实私信聊天，还是先做消息通知和悄悄话。
7. 是否上线推广裂变和校园代理。

阻塞原因：

- 不确认范围，无法排期。
- 不确认范围，无法确定数据库表和接口数量。
- 不确认范围，无法判断微信类目、资质和审核风险。

### 3.2 产品原型与 UI 设计确认

甲方必须提供或确认：

1. 移动端完整页面原型。
2. 管理后台完整页面原型。
3. 核心页面 UI 视觉稿。
4. 页面跳转流程图。
5. 空状态、加载态、异常态设计。
6. 弹窗文案和按钮文案。
7. 图标、品牌色、Logo、启动页、分享图。

阻塞原因：

- 无原型无法确定页面数量。
- 无视觉稿会导致前端返工。
- 无流程图会导致业务状态机反复变更。

### 3.3 小程序主体与账号资料

甲方必须提供：

1. 非个人主体微信小程序账号。
2. 小程序 `AppID`。
3. 小程序管理员信息。
4. 小程序开发者权限邀请。
5. 微信认证状态。
6. 营业执照 / 统一社会信用代码证。
7. 主体负责人身份证件。
8. 项目负责人身份证件。
9. 对外客服信息：客服电话、客服邮箱、投诉联系人。

阻塞原因：

- 没有 `AppID` 无法配置真实开发环境。
- 没有开发者权限无法上传体验版。
- 没有主体认证无法申请部分接口、类目和支付。

### 3.4 类目与资质确认

甲方必须先确认拟申请的微信小程序服务类目：

1. 社交-婚恋。
2. 社交-陌生人交友 / 熟人交友。
3. 如包含公开社区/论坛：社交-社区/论坛。
4. 如包含公开图文笔记分享：社交-笔记。

甲方必须准备或确认：

1. 《增值电信业务经营许可证》，核准服务项目需包含“互联网信息服务业务”。
2. 如涉及社区/论坛开放能力，准备对应类目要求的备案/资质材料。
3. 如微信后台要求前置审批、专项审批、承诺书或补充说明，需甲方负责提供。

阻塞原因：

- 类目不确认，首版功能范围无法确认。
- 交友/婚恋/社区类目审核风险高，不提前准备会直接影响上线时间。
- 如果资质不满足，很多功能即使开发完成也无法提审通过。

### 3.5 固定量表资料

如果采用固定测评版，甲方必须提供三套测评的最终材料：

1. 霍兰德职业测评题库。
2. 荣格认知八维测评题库。
3. 心理健康测评量表题库。
4. 每题选项。
5. 每题分值。
6. 是否反向计分。
7. 维度归属。
8. 计分规则。
9. 结果分段。
10. 报告文案。
11. 免责声明。
12. 题库版权或授权说明。

阻塞原因：

- 没有题库和计分规则，测评模块无法开发真实逻辑。
- 没有结果分段，无法生成报告。
- 心理健康量表没有专业审核和免责声明，会产生合规风险。

### 3.6 全量量表配置版补充资料

如果甲方坚持全量版，除 3.5 外，还必须补齐：

1. 动态公式语法。
2. 公式支持的函数范围。
3. 量表版本规则。
4. 常模数据格式。
5. 报告模板数据绑定规则。
6. 动态量表结果如何映射推荐算法。
7. 题目、公式、报告修改后历史报告如何处理。
8. 量表发布、灰度、回滚规则。
9. 用户作答中遇到量表新版本的处理规则。

阻塞原因：

- 全量版是通用测评平台，不是页面开发。
- 不补齐这些规则，无法设计数据库和服务端架构。

### 3.7 核心字典与基础数据

甲方必须提供或确认：

1. 学校字典。
2. 学历枚举。
3. 专业字典或专业输入规则。
4. 城市/地区范围。
5. 职业分类。
6. 兴趣标签。
7. 性格标签。
8. 择偶偏好标签。
9. 举报原因。
10. 封禁/处罚原因。
11. 通知类型。
12. 成家币流水类型。
13. 会员权益枚举。

阻塞原因：

- 字典不统一，移动端、后台、推荐算法会各写一套。
- 后期统一成本很高，且影响搜索、推荐、标签展示和数据统计。

### 3.8 核心业务规则确认

甲方必须确认：

1. 三重认证是否为成家主链路硬门槛。
2. 未认证用户能看什么、不能做什么。
3. 喜欢、跳过、悄悄话、YO 的配额和消耗规则。
4. VIP 权益。
5. 成家币充值档位和消耗单价。
6. 喜欢我的、访客、精选、理想型是否付费。
7. 黑名单、屏蔽、举报后各模块是否互相不可见。
8. 匹配成功定义。
9. 普通私信开放条件。
10. 女性保护机制是否启用。
11. 用户注销后数据保留和删除规则。

阻塞原因：

- 这些规则直接决定状态机和数据库关系。
- 不确认会导致主链路反复返工。

## 4. P1：可先写 mock，但会阻塞联调

### 4.1 技术基础设施

甲方必须提供或确认：

1. 生产服务器。
2. 测试服务器 / 预发服务器。
3. 数据库实例。
4. Redis 或缓存服务。
5. 对象存储：头像、认证材料、社区图片、报告图片。
6. CDN。
7. HTTPS 证书。
8. 日志平台。
9. 监控告警。
10. 备份与容灾策略。
11. 短信能力。
12. 邮件能力。

阻塞原因：

- 没有测试/预发环境，无法联调。
- 没有对象存储，认证、头像、动态图片都无法走真实链路。
- 没有监控和日志，交友类上线后无法排查事故。

### 4.2 小程序域名与网络配置

甲方必须提供：

1. API 请求域名。
2. WebSocket 域名，如首版做实时聊天。
3. 文件上传域名。
4. 文件下载/CDN 域名。
5. 业务域名，如使用 web-view。
6. 域名 ICP 备案证明。
7. HTTPS 证书。

网络要求：

1. 请求域名必须提前配置在微信后台。
2. 仅支持 `https` 和 `wss`。
3. 不支持 IP、`localhost`。
4. 域名必须完成 ICP 备案。
5. 不建议直接使用裸父域名，应使用明确子域名。
6. 业务域名也必须是 HTTPS 且完成备案。
7. 新备案域名通常需要预留生效时间。
8. TLS 建议 1.2 及以上，证书信任链完整。

阻塞原因：

- 域名不配置，真机无法请求后端。
- 证书不合规，微信端请求失败。

### 4.3 微信支付与商户资料

如首版包含 VIP、成家币、付费解锁，甲方必须提供：

1. 微信支付商户号。
2. 商户平台管理员权限。
3. API v3 密钥。
4. 商户证书。
5. 支付回调域名。
6. 商品名称和价格。
7. 退款规则。
8. 发票规则，如需要。
9. 对账单下载权限。

阻塞原因：

- 没有商户号，支付只能做 mock。
- 没有证书和密钥，无法联调支付、退款和回调。

### 4.4 内容安全能力

甲方必须提供或确认：

1. 使用微信内容安全接口，还是第三方内容审核服务。
2. 文本安全审核范围。
3. 图片安全审核范围。
4. 私信、悄悄话、评论、动态、资料文案是否都审核。
5. 机审失败后的处理方式。
6. 人工审核后台负责人。
7. 举报处理 SLA。
8. 删除、禁言、封禁、申诉规则。

阻塞原因：

- 社交/UGC/交友项目内容安全不是可选项。
- 未接入内容安全能力，提审和上线风险很高。

### 4.5 第三方认证能力

甲方必须确认并提供：

1. 实名认证方案：微信人脸核身、第三方实名，还是人工审核。
2. 学历认证方案：学信网材料、第三方学历接口，还是人工审核。
3. 头像认证方案：人脸比对、人工审核，还是第三方服务。
4. 所需接口账号、密钥、回调地址。
5. 失败兜底规则。
6. 审核通过/驳回文案。

阻塞原因：

- 认证是成家主链路准入条件。
- 没有认证能力，推荐、匹配、聊天的开放状态无法闭环。

### 4.6 推荐算法输入资料

甲方必须确认：

1. 首版推荐算法使用哪些字段。
2. 各字段是否硬过滤或只加权。
3. 测评结果是否进入推荐。
4. VIP 是否加权。
5. 同城、学校、年龄、学历、身高等权重口径。
6. 用户未完成测评时如何推荐。
7. 推荐理由是否展示给用户。
8. 推荐结果是否需要后台排查页。

阻塞原因：

- 推荐主链路需要尽早设计数据采集和特征字段。
- 后期补推荐字段会影响用户表、行为表、测评结果表。

### 4.7 审核测试账号与测试数据

甲方必须准备：

1. 小程序审核测试账号。
2. 已完成认证的测试账号。
3. 未认证测试账号。
4. VIP 测试账号。
5. 普通用户测试账号。
6. 有余额的成家币测试账号。
7. 后台审核账号。
8. 典型测试用户资料。
9. 测试动态、测试评论、测试聊天记录。
10. 审核路径说明。

阻塞原因：

- 没有测试账号，微信审核人员无法走完整路径。
- 没有测试数据，很多付费、认证、推荐、报告功能无法验收。

## 6. 微信交友类小程序特别注意事项

### 6.1 本项目不是普通类目小程序

按当前 PRD 功能形态，“成家立业”明显属于交友/婚恋/社交互动类项目。若按婚恋、陌生人交友、熟人交友形态上线，必须先匹配正确类目与资质，不能按普通企业展示类小程序提审。

### 6.2 首次提审要预留额外审核时间

社交-陌生人交友、熟人交友、婚恋等类目一般存在更严格的首次审核要求。项目排期中必须预留额外审核时间，不能把提审当作普通 1-2 天流程处理。

### 6.3 内容安全不是可选项

只要包含以下能力，就必须接入内容安全和人工处置闭环：

1. 用户资料文本。
2. 用户头像和相册。
3. 社区动态。
4. 评论。
5. 私信。
6. 悄悄话。
7. 话题和帖子。
8. 举报和申诉。

### 6.4 社区能力会抬高审核要求

如果首版包含公开动态、评论、朋友模块、图文笔记分享等 UGC 能力，可能触发社区/论坛/笔记相关类目要求。若甲方资质不齐，建议首版优先上线婚恋主链路，社区公开发帖能力谨慎首发。

## 7. 建议向甲方确认的最小开工清单

如果只抓最关键的，开发启动会前至少让甲方提供或确认以下 20 项：

1. 小程序 `AppID` 和开发者权限。
2. 小程序主体和微信认证状态。
3. 拟申请类目和现有资质文件。
4. 是否具备 ICP 备案。
5. 首版功能范围：社区、支付、私信、测评、推荐是否都上。
6. 固定测评版还是全量量表配置版。
7. 三套固定测评题库、选项、分值、计分规则。
8. 用户协议和隐私政策初稿。
9. 认证方案：实名、学历、头像。
10. 微信支付商户号是否已开通。
11. 生产/测试域名。
12. 测试服务器和数据库。
13. 对象存储方案。
14. 内容安全审核方案。
15. 核心字典：学校、城市、标签、举报原因。
16. VIP 和成家币价格规则。
17. 推荐算法首版口径。
18. 后台角色和权限范围。
19. 审核测试账号准备人。
20. 客服和投诉处理联系人。

以上 20 项不齐，建议只进入原型/技术预研，不进入正式开发排期。
