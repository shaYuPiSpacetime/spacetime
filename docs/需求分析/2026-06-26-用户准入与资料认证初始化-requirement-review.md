## 0. 本版说明

上一版评审主要依据模块公共定义、移动端/后台模块 PRD、关键页面规格与关键词复核完成，可以作为初评，但不足以称为“逐文件深度评审”。本版按新版流程补齐了 PRD-01 需求目录内 29 份文件的全文阅读，并补读项目级/端级全局定义、`AGENTS.md`、`TEAM_STANDARDS.md`，所有冻结判断和风险项均回挂到具体文件或章节。

本次没有修改 PRD 原文，只重写需求合理性评审报告和流程文档。结论不是“完全无风险冻结”，而是“主流程合理、可有条件冻结，条件必须进入技术方案、测试用例和冻结记录”。

## 1. 评审结论

- 结论：有条件冻结。
- 一次性交付风险：中偏高。高风险不来自主流程断链，而来自敏感材料、外部服务、审核历史、人工复核、导出无掩码、移动端异常态缺图和跨模块准入拦截联调。
- P0 阻塞：未发现会导致登录、轻量引导、强引导、三重认证、后台审核、状态回写、核心准入计算无法成立的 P0 断链。
- 是否允许进入 PRD 冻结：允许，但冻结记录必须写入本文 P1/P2 条件项。
- 需要确认项：见第 7 节待确认清单；真正需要产品/设计/法务/安全确认的是 UI 缺图、高敏材料留存/导出、核心准入配置边界和老用户编辑页视觉验收。
- 冻结前不建议再扩范围：不要把人脸核身、婚况认证、海外/港澳台学历审核、独立审核工作台、语音人工审核混入本期。

## 4. 需求合理性判断

| 判断项 | 结论 | 证据 | 说明 |
|--------|------|------|------|
| 准入门槛是否合理 | 合理 | 公共定义第 4、5、6、7 节，APP-13 | 核心曝光、喜欢、私信、社区发布要求实名、头像、学历均 approved 且账号正常，资料完整度不替代三重认证 |
| 实名规则是否清晰 | 合理 | 公共定义 P-01、APP-11、实名审核页 | 首版不做人脸核身，采用姓名+身份证号+已绑定手机号，第三方不可用进入人工兜底 |
| 学历前置是否清晰 | 合理 | 公共定义 M01-JDG-realname-accepted，APP-10、APP-12 | 学历可提交前置是“实名已提审 reviewing 或 approved”，不是必须实名已通过，降低流程阻塞 |
| 审核来源模型是否合理 | 合理但复杂 | 公共定义 M01-ENUM-audit-source，后台三类认证和内容审核页 | 业务只看大状态 approved，auditSource 用于后台展示/复核/审计，模型一致但实现要有审核历史 |
| 字段池是否可落地 | 合理但需配置设计 | 公共定义 M01-DATA-user-input-fields，配置页字段配置 Tab | 字段事实源集中，展示/必填/计分由配置控制，避免移动端和后台各写一套字段 |

## 12. 最终建议

PRD-01 的需求主干是合理的：移动端负责准入采集和认证提交，后台负责审核、导入、配置与审计，公共定义提供统一状态机、字段池、配置、通知和错误码。它可以进入 PRD 冻结，但冻结性质必须是“有条件冻结”。

建议冻结记录同步写入四类条件：第一，移动端缺图异常态按页面规格先实现，后续补图做视觉验收；第二，外部服务缺测试凭证时测试报告必须标跳过；第三，高敏材料留存、原图下载、无掩码导出必须经权限、审计和合规确认；第四，技术方案必须统一审核模型、准入计算、配置版本和事件通知，不得在 APP/ADM 各自落一套口径。
