## 1. 背景与目标

PRD-07 承接成家立业前期增长诉求，拆成两条业务线：

1. 普通用户邀请裂变：用户生成普通用户二维码，新用户扫码进入小程序并完成注册登录、资料完善、三项认证等节点后，邀请人获得成家币奖励。
2. 校园代理推广：后台为校园代理生成二维码编号/小程序路径，按扫码、注册登录、资料完成、三项认证等事件统计，并形成线下奖金结算台账。

核心目标：


| 目标                             | 技术承接                                             |
| ------------------------------ | ------------------------------------------------ |
| 邀请 2 位同学完成三项认证后，邀请人基本可免费体验核心玩法 | 后台配置普通奖励规则；奖励流水先进入审核/待处理状态，成家币真实入账待 PRD-04 接入 |
| 普通邀请与代理邀请不可重复计奖                | 统一推广来源绑定表，单个新用户只允许一个有效来源，代理渠道优先                  |
| 成功邀请口径统一                       | `实名认证 + 头像认证 + 学历认证` 全部完成后，更新为 `verify_success`  |
| 代理可结算                          | 代理奖金事件落入奖金明细，再生成结算单，首版不自动打款                      |
| 异常可冻结与复核                       | 奖励流水支持 `pending/success/frozen/invalid`，后台处理冻结队列 |

## 4. 总体架构与调用链

### 4.1 普通邀请绑定链路

```text
普通用户二维码
  → 小程序打开时携带 inviterId/sourceTraceId
  → POST /miniapp/promotion/invite/share-log 记录点击来源
  → 用户注册并首次登录成功
  → Auth/Register Service 调 PromotionInviteService.bind()
  → 校验登录用户、非自邀、未绑定过有效来源
  → 写入 promotion_invite_relation
  → 更新点击记录 user_id / bind_status
```

### 4.2 普通奖励触发链路

```text
注册登录 / 资料完成 / 三项认证完成 / 首次付费事件
  → 业务 Service 调 PromotionInviteEventService.handleInviteEvent(inviteeId, eventType)
  → 查询 promotion_invite_relation
  → 按 eventType 推进邀请关系状态
  → 查询 promotion_rule
  → 幂等校验：同一 relationId + eventType 不重复生成奖励
  → 写 promotion_reward_log
  → 后续奖励审核/资产/通知模块继续处理发放与通知
```

当前代码已落地统一入口，但注册登录、资料完善、三项认证等主业务模块尚未实际接入调用方。后续对应模块实现或补技术方案时，必须显式引用该入口并说明调用时机、事务边界和失败策略。

### 4.3 代理推广统计与结算链路

```text
代理二维码/小程序路径
  → POST /miniapp/promotion/invite/share-log 记录 qrCode 来源
  → 注册登录后建立 promotion_invite_relation(source_type=agent_qr)
  → 用户后续事件写 promotion_agent_event
  → 后续代理奖金明细可写 promotion_agent_bonus_log
  → 后台/测试数据生成 promotion_agent_settlement
  → 财务确认/发放，更新结算状态，不触发自动打款
```

### 4.4 管理后台链路

```text
frontend/src/pages/promotion/*
  → frontend/src/api/promotion.ts
  → /admin/promotion/**
  → Promotion*Controller
  → Promotion*Service
  → Promotion*ServiceImpl
  → Promotion*Dao
  → Promotion*DaoImpl
  → Promotion*Mapper/XML
  → MySQL
```

## 5. 方案选择


| 方案   | 说明                                          | 优点                        | 缺点                 | 结论   |
| ---- | ------------------------------------------- | ------------------------- | ------------------ | ---- |
| 最小改动 | 用 3 张表承接规则、关系、代理，奖励和结算靠 JSON 扩展             | 交付快                       | 后台查询、冻结、结算、审计会很快失控 | 不选   |
| 平衡方案 | 规则、关系、奖励流水、代理、代理事件、奖金、结算分表；不引入 MQ/外部风控/自动打款 | 满足 PRD-07 本期全量后台能力，符合单体架构 | 表较多，但边界清晰          | 选择   |
| 完整方案 | 增加事件总线、风控引擎、BI 聚合、自动打款、代理工作台                | 扩展性强                      | 超出本期范围与报价边界        | 后续再做 |


本方案采用平衡方案：所有触发在当前 Spring Boot 单体内同步处理，关键写操作加事务；导出、统计使用分页查询和必要索引，不引入消息队列。

## 6. 数据库设计

### 6.1 表清单


| 表名                           | 说明                        |
| ---------------------------- | ------------------------- |
| `promotion_rule`             | 推广规则主表，区分普通邀请与代理规则        |
| `promotion_rule_tier`        | 普通邀请阶梯奖励规则                |
| `promotion_source_trace`     | 点击/扫码/分享来源记录，绑定前暂存来源      |
| `promotion_invite_relation`  | 邀请/代理归属关系，一个被邀请人只允许一个有效关系 |
| `promotion_reward_log`       | 普通用户成家币奖励流水               |
| `promotion_agent`            | 校园代理信息                    |
| `promotion_agent_qr_code`    | 校园代理二维码编号、二维码路径、版本和状态      |
| `promotion_agent_event`      | 代理推广事件明细，用于统计             |
| `promotion_agent_bonus_log`  | 代理奖金事件明细                  |
| `promotion_agent_settlement` | 代理奖金结算单                   |
| `promotion_audit_log`        | 推广模块操作日志                  |

### 7.5 事件处理规则


| 事件                           | 触发时机       | 普通邀请动作                     | 代理动作                                                 |
| ---------------------------- | ---------- | -------------------------- | ---------------------------------------------------- |
| `click`                      | 分享/扫码进入    | 写 `promotion_source_trace` | 写 `promotion_source_trace` + `promotion_agent_event` |
| `register_login_reward`      | 新用户首次注册并登录 | 绑定关系后触发注册登录奖励；当前统一入口要求关系已存在 | 建关系，写 `registered` 代理事件；奖金生成保留扩展 |
| `profile_complete_reward`    | 基础资料完成     | 更新关系状态，发资料完成奖励             | 写 `profile_completed` 代理事件；奖金生成保留扩展 |
| `verify_complete_reward`     | 三项认证完成     | 更新成功邀请状态，发三项认证奖励           | 写 `verify_success` 代理事件；奖金生成保留扩展 |
| `first_vip_reward`           | 首购 VIP     | 当前仅保留枚举/规则扩展点              | 当前仅保留扩展点                                           |
| `first_coin_recharge_reward` | 首充成家币      | 当前仅保留枚举/规则扩展点              | 当前仅保留扩展点                                           |


统一入口约束：

- 已实现入口：`PromotionInviteEventService.handleInviteEvent(Long inviteeId, String eventType)`。
- 当前支持事件：`register_login_reward`、`profile_complete_reward`、`verify_complete_reward`。
- 状态推进口径：注册登录事件要求已存在邀请关系；资料完善事件调用 `markProfileCompleted(inviteeId)`；认证完成事件调用 `markVerifySuccess(inviteeId)`。
- 奖励生成口径：仅普通用户邀请关系生成普通邀请奖励；校园代理二维码来源不生成普通用户邀请奖励。
- 幂等口径：以 `relationId + eventType` 查询 `promotion_reward_log`，已存在则不重复生成。
- 当前调用方状态：该 Service 已实现并有种子测试/单元测试调用，但注册登录、资料完善、认证完成等实际业务 Service 尚未接入。后续模块必须调用该入口，不能直接更新 `promotion_invite_relation` 或直接插入 `promotion_reward_log`。


### 7.6 风控与幂等


| 场景                | 处理                                                        |
| ----------------- | --------------------------------------------------------- |
| 老用户进入邀请链接         | 当前按是否已有邀请关系判断；已有关系时直接返回原关系，不覆盖来源                         |
| 自己邀请自己            | 直接拒绝绑定，抛出“不能邀请自己”                                         |
| 同一被邀请人重复来源        | `promotion_invite_relation.uk_invitee_active` 拦截，首次有效来源优先 |
| 同时命中普通邀请与代理二维码    | 代理渠道优先，关系归属校园代理，不生成普通用户邀请奖励                            |
| 同设备/手机号/实名/支付账号异常 | 当前保留奖励状态 `frozen` 和审核入口，复杂风控规则待用户/认证/支付模块接入后完善        |
| 重复事件回调            | `promotion_reward_log.uk_relation_event` 与 Service 查询保证普通奖励幂等；代理事件表保留唯一键 |
| 后台人工复核            | 记录 `promotion_audit_log`，并更新奖励/关系/结算状态                    |

### 8.2 交互流程

#### 流程 1：配置普通邀请奖励规则

**入口**：推广管理 → 推广规则配置
**权限**：`promotion:rule:list/add/edit`
**组件/API**：`PromotionManagement.tsx`、`frontend/src/api/promotion.ts`

**正常流：**

1. 页面加载规则列表，按规则类型、事件类型、状态筛选。
2. 运营点击新增/编辑，填写奖励事件、奖励金额、生效时间、是否启用。
3. 保存成功后刷新列表；当前页面不开放阶梯 JSON 配置入口，阶梯接口保留给后续扩展或测试。

**异常/边界：**

- 生效时间晚于失效时间 → 后端拦截，提示时间范围错误。
- 奖励金额为负数 → 前端校验，后端二次校验。
- 无编辑权限 → 按钮隐藏。

#### 流程 2：奖励审核

**入口**：推广管理 → 奖励审核
**权限**：`promotion:reward:review`
**组件/API**：`PromotionManagement.tsx`、`frontend/src/api/promotion.ts`

**正常流：**

1. 运营按奖励状态、奖励事件筛选奖励流水。
2. 对 `frozen` 奖励点击“通过”或“拒绝”，使用正式弹窗填写备注；拒绝时备注作为“审核拒绝原因”展示。
3. 后端更新奖励状态并写审计日志；真实成家币入账待 PRD-04 资产模块接入。

**异常/边界：**

- 奖励已被其他管理员处理 → 返回状态已变更，前端刷新。
- 成家币流水写入失败 → 整体事务回滚，奖励仍保持冻结。

#### 流程 3：管理代理与二维码

**入口**：推广管理 → 代理列表
**权限**：`promotion:agent:list/add/edit/code`

**正常流：**

1. 渠道运营新增代理，填写学校、校区、联系人、联系电话、状态和备注。
2. 运营在代理列表点击“二维码”，后端生成 `qrCode` 和小程序路径，前端用弹窗展示二维码。
3. 渠道运营可重新生成代理二维码。
4. 代理暂停/终止后，后台更新状态；是否继续计奖由规则配置控制。

**异常/边界：**

- 联系电话重复可允许，但 qrCode 必须唯一。
- 停用代理二维码后，旧二维码扫码进入小程序仍可展示普通入口，但不再建立新代理关系。

#### 流程 4：代理结算确认/发放

**入口**：推广管理 → 代理结算管理
**权限**：`promotion:settlement:add/confirm/pay`

**正常流：**

1. 结算数据由后台业务/测试数据生成，管理后台不提供页面手工生成入口。
2. 运营按代理名称/联系人/手机号和状态查询结算单。
3. 待确认结算单点击“确认”，使用正式弹窗填写可选备注，状态变为 `confirmed`。
4. 线下打款完成后点击“发放”，填写实发金额与备注，状态变为 `paid`。

**异常/边界：**

- 同代理同周期重复生成 → 后端提示已有结算单。
- 存在已取消/已结算奖金明细 → 不纳入本次汇总。

### 10.1 跨 PRD 联调契约与反向约束

以下依赖不在 PRD-07 单模块内闭环实现，但 PRD-07 的奖励发放、成功邀请统计、代理奖金统计必须依赖它们。后续编写 PRD-03、PRD-04、PRD-06 或其他相关技术方案时，必须在各自方案的“与其他模块联动/风险/实施顺序/测试方案”中显式覆盖本表。

| 依赖 PRD/模块 | PRD-07 需要的能力 | 对方技术方案必须考虑 | PRD-07 当前降级策略 |
| ------------ | ---------------- | ------------------ | ------------------ |
| PRD-04 商业化/资产中心 | 成家币奖励真实入账、资产流水查询、首充成家币事件、首购 VIP 事件 | 提供 `app_user_coin_log` 或等价资产流水表；支持幂等业务单号；区分邀请奖励流水类型；支付成功后通过推广统一入口或其后续扩展触发 `first_vip_reward`、`first_coin_recharge_reward`；资产流水可反查 `promotion_reward_log.id` | 当前仅保留 `promotion_reward_log.coin_log_id` 与事件枚举，未真实写入成家币流水；统一入口暂只实现注册、资料、认证三类事件 |
| PRD-03 消息/通知中心 | 奖励到账、奖励冻结、奖励无效、代理结算状态变更通知 | 提供通知写入服务或表结构；通知 payload 支持跳转邀请记录、奖励流水、资产明细；失败不应回滚核心奖励事务，但需可重试或记录失败 | 当前奖励状态可流转，但通知写入暂不接入 |
| PRD-06 用户资料/认证或对应认证模块 | 注册登录、资料完成、实名/头像/学历三项认证完成事件 | 注册/登录成功后先完成邀请关系绑定，再调用 `PromotionInviteEventService.handleInviteEvent(inviteeId, "register_login_reward")`；资料首次完成后调用 `handleInviteEvent(inviteeId, "profile_complete_reward")`；三项认证全部通过且首次达成时调用 `handleInviteEvent(inviteeId, "verify_complete_reward")`；事件必须幂等，调用失败策略需在对方方案中声明 | 当前小程序绑定接口和统一事件入口已预留；资料/认证完成事件由后续模块接入 |
| 用户基础模块 | 邀请人/被邀请人身份、老用户判断、自邀校验、手机号/实名等风控输入 | 用户表需提供注册时间、首次登录时间、手机号/实名状态等字段或查询接口；推广模块不得直接跨包访问小程序/后台私有实现，应通过 common 服务或事件入口衔接 | 当前只按 `userId` 建立关系，复杂用户画像风控暂缓 |
| 支付/订单模块 | 代理首次付费奖金、普通邀请首购返利开关 | 支付成功回调需提供订单类型、用户 ID、金额、支付成功时间、业务单号；推广事件处理必须放在支付成功事务之后或具备重试补偿 | 当前首购相关规则默认关闭，仅保留事件类型 |
| OSS/小程序码模块 | 代理二维码、普通用户二维码、分享路径 | 小程序码 scene 只放短码，完整来源落 `promotion_source_trace`；二维码/素材 URL 回写 `promotion_agent_qr_code.qr_url/material_url` | 当前可返回小程序 path，二维码/素材生成可后置 |

### 10.2 其他 PRD 技术方案编写要求

当后续技术方案涉及注册登录、用户资料、三项认证、成家币、VIP/支付、通知、OSS、小程序码时，必须增加“PRD-07 推广裂变联动影响”小节，至少回答：

1. 是否会触发 PRD-07 推广事件；触发的 `eventType` 是什么。
2. 是否调用 `PromotionInviteEventService.handleInviteEvent(inviteeId, eventType)`；若暂不调用，必须说明后续补齐入口。
3. 事件是否幂等；幂等键是否包含 `relationId + eventType`、`userId + eventType + bizNo` 或等价字段。
4. 失败处理：推广事件失败是否影响主流程，是否需要重试、补偿或后台人工处理。
5. 数据反查：对方流水或通知是否能关联回 `promotion_reward_log`、`promotion_agent_event` 或 `promotion_agent_bonus_log`。
6. 测试覆盖：是否新增跨模块用例，例如认证完成后发放邀请奖励、支付成功后生成代理奖金、奖励到账后生成通知。

若对方 PRD 技术方案暂不实现联动，必须在方案中写明“暂不接入 PRD-07 的影响”和后续补齐入口，不能只省略不写。

## 13. 风险与回滚


| 风险                   | 影响                   | 应对                           |
| -------------------- | -------------------- | ---------------------------- |
| PRD-04/03/06 基础表未先落地 | 奖励到账、通知、三项认证事件无法完整闭环 | PRD-07 实施前先补基础表，或将联动接口做成可空降级 |
| 其他 PRD 方案未纳入 PRD-07 联动 | 后续认证、支付、通知上线后仍无法触发邀请奖励或代理奖金 | 在 PRD-03/04/06 等技术方案中强制增加“PRD-07 推广裂变联动影响”小节，并按第 10.1/10.2 节验收 |
| 推广事件重复触发             | 重复发币或重复计奖金           | 数据库唯一键 + Service 幂等校验        |
| 风控规则误伤               | 用户奖励冻结过多             | 后台冻结队列支持人工发放，规则可启停           |
| 代理二维码泄露或误用          | 代理统计异常               | 代理二维码支持停用/重生成，历史版本保留        |
| 后续若新增导出能力           | 数据量大导致后台超时        | 另行评估时间范围限制、分页导出或异步导出       |


回滚策略：

1. 关闭对应 `promotion_rule`，停止新奖励生成。
2. 停用代理二维码，停止新代理关系绑定。
3. 对异常奖励批量置 `frozen`，等待人工复核。
4. 前端隐藏推广管理菜单，不影响已存在 RBAC/字典功能。
