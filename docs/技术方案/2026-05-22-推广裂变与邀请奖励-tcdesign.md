## 1. 背景与目标

PRD-07 承接成家立业前期增长诉求，拆成两条业务线：

1. 普通用户邀请裂变：用户通过分享卡片、海报、专属码邀请新用户，被邀请人完成注册登录、资料完善、三项认证等节点后，邀请人获得成家币奖励。
2. 校园代理推广：后台为校园代理生成专属码/二维码/小程序路径，按扫码、注册、登录、资料完成、三项认证、首次付费等事件统计，并形成线下奖金结算台账。

核心目标：


| 目标                             | 技术承接                                             |
| ------------------------------ | ------------------------------------------------ |
| 邀请 2 位同学完成三项认证后，邀请人基本可免费体验核心玩法 | 后台配置普通奖励规则与阶梯规则，奖励到账写入成家币流水                      |
| 普通邀请与代理邀请不可重复计奖                | 统一推广来源绑定表，单个新用户只允许一个有效来源，代理渠道优先                  |
| 成功邀请口径统一                       | `实名认证 + 头像认证 + 学历认证` 全部完成后，更新为 `verify_success`  |
| 代理可结算                          | 代理奖金事件落入奖金明细，再生成结算单，首版不自动打款                      |
| 异常可冻结与复核                       | 奖励流水支持 `pending/success/frozen/invalid`，后台处理冻结队列 |

## 4. 总体架构与调用链

### 4.1 普通邀请绑定链路

```text
小程序分享卡片/海报/专属码
  → 小程序打开时携带 inviterId/inviteCode/sourceTraceId
  → POST /miniapp/promotion/invite/share-log 记录点击来源
  → 用户注册并首次登录成功
  → Auth/Register Service 调 PromotionInviteService.bindInvitee()
  → 校验新用户、非自邀、未绑定过有效来源、风险规则
  → 写入 promotion_invite_relation
  → 更新点击记录 user_id / bind_status
```

### 4.2 普通奖励触发链路

```text
注册登录 / 资料完成 / 三项认证完成 / 首次付费事件
  → 业务 Service 调 PromotionRewardService.handleUserEvent()
  → 查询 promotion_invite_relation
  → 查询 promotion_rule + promotion_rule_tier
  → 风控校验
  → 写 promotion_reward_log
  → 未冻结：写 app_user_coin_log，更新奖励状态 success
  → 写 notification_record 通知
  → 三项认证完成：更新邀请关系 verify_success，刷新成功人数与阶梯奖励
```

### 4.3 代理推广统计与结算链路

```text
代理二维码/小程序路径
  → POST /miniapp/promotion/invite/share-log 记录 agentCode 来源
  → 注册登录后建立 promotion_invite_relation(source_type=agent)
  → 用户后续事件写 promotion_agent_event
  → 按代理规则生成 promotion_agent_bonus_log
  → 后台生成 promotion_agent_settlement
  → 财务确认/发放，更新结算状态，不触发自动打款
```

### 4.4 管理后台链路

```text
frontend/src/pages/promotion/*
  → frontend/src/api/promotion.ts
  → /admin/promotion/**
  → Promotion*Controller
  → Promotion*Service
  → Promotion*ServiceImpl
  → Promotion*Dao
  → Promotion*DaoImpl
  → Promotion*Mapper/XML
  → MySQL
```

## 5. 方案选择


| 方案   | 说明                                          | 优点                        | 缺点                 | 结论   |
| ---- | ------------------------------------------- | ------------------------- | ------------------ | ---- |
| 最小改动 | 用 3 张表承接规则、关系、代理，奖励和结算靠 JSON 扩展             | 交付快                       | 后台查询、冻结、结算、审计会很快失控 | 不选   |
| 平衡方案 | 规则、关系、奖励流水、代理、代理事件、奖金、结算分表；不引入 MQ/外部风控/自动打款 | 满足 PRD-07 本期全量后台能力，符合单体架构 | 表较多，但边界清晰          | 选择   |
| 完整方案 | 增加事件总线、风控引擎、BI 聚合、自动打款、代理工作台                | 扩展性强                      | 超出本期范围与报价边界        | 后续再做 |


本方案采用平衡方案：所有触发在当前 Spring Boot 单体内同步处理，关键写操作加事务；导出、统计使用分页查询和必要索引，不引入消息队列。

## 6. 数据库设计

### 6.1 表清单


| 表名                           | 说明                        |
| ---------------------------- | ------------------------- |
| `promotion_rule`             | 推广规则主表，区分普通邀请与代理规则        |
| `promotion_rule_tier`        | 普通邀请阶梯奖励规则                |
| `promotion_source_trace`     | 点击/扫码/分享来源记录，绑定前暂存来源      |
| `promotion_invite_relation`  | 邀请/代理归属关系，一个被邀请人只允许一个有效关系 |
| `promotion_reward_log`       | 普通用户成家币奖励流水               |
| `promotion_agent`            | 校园代理信息                    |
| `promotion_agent_code`       | 代理专属码、二维码路径、版本和状态         |
| `promotion_agent_event`      | 代理推广事件明细，用于统计             |
| `promotion_agent_bonus_log`  | 代理奖金事件明细                  |
| `promotion_agent_settlement` | 代理奖金结算单                   |
| `promotion_audit_log`        | 推广模块操作日志                  |

### 7.6 风控与幂等


| 场景                | 处理                                                        |
| ----------------- | --------------------------------------------------------- |
| 老用户进入邀请链接         | 只记录点击，不建立新关系，不计奖                                          |
| 自己邀请自己            | 关系置 `invalid` 或直接拒绝绑定，记录原因 `self_invite`                  |
| 同一被邀请人重复来源        | `promotion_invite_relation.uk_invitee_active` 拦截，首次有效来源优先 |
| 同时命中普通邀请与代理码      | 代理渠道优先，普通邀请来源置 `ignored`                                  |
| 同设备/手机号/实名/支付账号异常 | 奖励置 `frozen`，进入后台冻结队列                                     |
| 重复事件回调            | `uk_relation_event`、`uk_agent_user_event` 保证同一关系同一事件只处理一次 |
| 后台人工复核            | 记录 `promotion_audit_log`，并更新奖励/关系/结算状态                    |

### 8.2 交互流程

#### 流程 1：配置普通邀请奖励规则

**入口**：推广管理 → 推广规则配置
**权限**：`promotion:rule:list/add/edit`
**组件/API**：`PromotionRuleManagement.tsx`、`frontend/src/api/promotion.ts`

**正常流：**

1. 页面加载规则列表，按规则类型、事件类型、状态筛选。
2. 运营点击新增/编辑，填写奖励事件、奖励金额、生效时间、是否启用。
3. 若事件为阶梯奖励，编辑 `0-2 / 3-8 / 9-20 / 21-50` 四档奖励。
4. 保存成功后刷新列表。

**异常/边界：**

- 时间范围重叠且事件相同 → 后端拦截，提示规则冲突。
- 奖励金额为负数 → 前端校验，后端二次校验。
- 无编辑权限 → 按钮隐藏。

#### 流程 2：处理冻结奖励

**入口**：推广管理 → 冻结奖励处理页
**权限**：`promotion:reward:review`
**组件/API**：`FrozenRewardManagement.tsx`

**正常流：**

1. 风控人员按冻结原因、邀请人、被邀请人、时间筛选冻结流水。
2. 点击详情查看邀请链路、命中原因、历史奖励。
3. 选择“确认有效并发放”或“确认无效并作废”，填写备注。
4. 后端更新奖励状态，写成家币流水或作废记录，并写审计日志。

**异常/边界：**

- 奖励已被其他管理员处理 → 返回状态已变更，前端刷新。
- 成家币流水写入失败 → 整体事务回滚，奖励仍保持冻结。

#### 流程 3：管理代理与代理码

**入口**：推广管理 → 代理列表
**权限**：`promotion:agent:list/add/edit/code`

**正常流：**

1. 渠道运营新增代理，填写学校、校区、联系人、规则组。
2. 保存后系统生成 `agentCode` 和小程序路径。
3. 渠道运营下载二维码或重新生成代理码。
4. 代理暂停/终止后，后台更新状态；是否继续计奖由规则配置控制。

**异常/边界：**

- 联系电话重复可允许，但 agentCode 必须唯一。
- 停用代理码后，旧码扫码进入小程序仍可展示普通入口，但不再建立新代理关系。

#### 流程 4：生成代理结算单

**入口**：推广管理 → 代理结算管理
**权限**：`promotion:settlement:add/confirm/pay`

**正常流：**

1. 财务选择代理和结算周期。
2. 系统汇总 `pending_settlement` 的奖金明细并生成结算单。
3. 渠道运营确认后状态变为 `confirmed`。
4. 线下打款完成后，财务标记已发放，填写已发金额与备注。

**异常/边界：**

- 同代理同周期重复生成 → 后端提示已有结算单。
- 存在已取消/已结算奖金明细 → 不纳入本次汇总。

### 10.2 其他 PRD 技术方案编写要求

当后续技术方案涉及注册登录、用户资料、三项认证、成家币、VIP/支付、通知、OSS、小程序码时，必须增加“PRD-07 推广裂变联动影响”小节，至少回答：

1. 是否会触发 PRD-07 推广事件；触发的 `eventType` 是什么。
2. 事件是否幂等；幂等键是否包含 `userId + eventType + bizNo` 或等价字段。
3. 失败处理：推广事件失败是否影响主流程，是否需要重试、补偿或后台人工处理。
4. 数据反查：对方流水或通知是否能关联回 `promotion_reward_log`、`promotion_agent_event` 或 `promotion_agent_bonus_log`。
5. 测试覆盖：是否新增跨模块用例，例如认证完成后发放邀请奖励、支付成功后生成代理奖金、奖励到账后生成通知。

若对方 PRD 技术方案暂不实现联动，必须在方案中写明“暂不接入 PRD-07 的影响”和后续补齐入口，不能只省略不写。

## 13. 风险与回滚


| 风险                   | 影响                   | 应对                           |
| -------------------- | -------------------- | ---------------------------- |
| PRD-04/03/06 基础表未先落地 | 奖励到账、通知、三项认证事件无法完整闭环 | PRD-07 实施前先补基础表，或将联动接口做成可空降级 |
| 其他 PRD 方案未纳入 PRD-07 联动 | 后续认证、支付、通知上线后仍无法触发邀请奖励或代理奖金 | 在 PRD-03/04/06 等技术方案中强制增加“PRD-07 推广裂变联动影响”小节，并按第 10.1/10.2 节验收 |
| 推广事件重复触发             | 重复发币或重复计奖金           | 数据库唯一键 + Service 幂等校验        |
| 风控规则误伤               | 用户奖励冻结过多             | 后台冻结队列支持人工发放，规则可启停           |
| 代理码泄露或误用             | 代理统计异常               | 代理码支持停用/重生成，历史版本保留           |
| 导出数据量大               | 后台超时                 | 首版限制时间范围与分页导出，后续再做异步导出       |


回滚策略：

1. 关闭对应 `promotion_rule`，停止新奖励生成。
2. 停用代理码，停止新代理关系绑定。
3. 对异常奖励批量置 `frozen`，等待人工复核。
4. 前端隐藏推广管理菜单，不影响已存在 RBAC/字典功能。
