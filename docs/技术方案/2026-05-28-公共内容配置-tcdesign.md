## 1. 背景与目标

PRD-06 中的“文案与消息中心”和“移动端配置管理”是我的页、设置页、安全中心、搜索页等移动端页面的底座能力。本方案不整块吞掉 PRD-06，优先落地边界清晰、低耦合的平台型配置能力：

1. **公共内容管理**：公告、帮助文档、规则协议说明、安全指南/反诈指南、关于我们原生内容。
2. **应用配置管理**：协议 H5 地址、搜索空状态文案、搜索违规提示文案、基础排序等 key-value 配置。
3. **移动端入口配置**：我的页、设置页、安全中心、搜索结果 Tab 的名称、排序、启停、跳转方式、角标文案。
4. **搜索热词配置**：热门词排序与启停。
5. **搜索屏蔽/违规词配置**：搜索词违规词库、搜索结果屏蔽词，先做配置与服务能力，不实现推荐算法排序。
6. **操作审计**：发布/下线公告、修改入口配置、修改搜索屏蔽规则等动作落审计。

成功标准：

1. 管理后台可维护公共内容、应用配置、移动端入口、搜索热词、搜索屏蔽词。
2. 小程序端可查询已启用的公告/帮助/规则/配置/入口/热词/搜索页展示配置。
3. 搜索屏蔽词不直接暴露给小程序端，仅由后端提供内部匹配能力，供后续搜索结果接口调用。
4. 后续 A/B 同学实现我的页、设置页、安全中心、搜索页时，可直接消费本方案定义的数据与接口。

## 4. 总体架构与调用链

```text
管理后台页面
  → frontend/src/api/content.ts
  → /admin/content/**
  → Admin Content*Controller
  → Admin Content*Service
  → Admin Content*ServiceImpl
  → common Content*Dao
  → common Content*DaoImpl
  → common Content*Mapper
  → MySQL

小程序端公共配置
  → /miniapp/content/** /miniapp/mobile-config/** /miniapp/search/**
  → MiniappContentController / MiniappMobileConfigController / MiniappSearchConfigController
  → Miniapp*Service
  → common DAO
  → MySQL

后续搜索结果接口
  → MiniappSearchService.search(keyword)
  → SearchBlockWordService.match(keyword)
  → 命中违规词则返回违规提示，不暴露词库
```

分层约束：

1. `admin/` 和 `miniapp/` 不互相 import。
2. Entity、Mapper、DAO 放 `common/`。
3. Controller 精确返回 `R<T>`。
4. 后台接口使用 `@RequirePermission`；小程序公共查询接口由 `WebConfig` 放行 token。

### 5.5 实现要点补充

1. **公告/内容生效时间过滤**：小程序查询公告和内容时，DAO 层 WHERE 条件必须包含 `status='ENABLED' AND deleted=0 AND (effective_time IS NULL OR effective_time <= NOW()) AND (expire_time IS NULL OR expire_time > NOW())`，不能只看 status。
2. **入口配置 `config:{key}` 引用解析**：`MiniappMobileConfigService.getEntries()` 返回入口列表时，若 `jumpTarget` 以 `config:` 开头，需查询 `AppConfigDao.selectByKey()` 获取真实 URL 后替换 `jumpTarget` 值返回给小程序端，避免小程序端二次查询。
3. **搜索屏蔽词内部匹配**：`MiniappSearchConfigService.validateKeyword(keyword)` 为内部方法，不暴露 HTTP 接口。后续搜索结果接口调用此方法判断关键词是否违规，命中则返回 `app_config` 中的 `search.violation_text` 文案。

### 7.2 交互流程

#### 流程 1：内容文章管理

**入口**：左侧菜单 → 运营中心 → 内容文章  
**组件/API**：`ContentArticlePage.tsx`、`frontend/src/api/content.ts`

正常流：

1. 顶部 Tab 切换公告、帮助、规则、安全内容、关于我们。
2. 筛选栏支持标题关键词、分类、状态。
3. 表格展示标题、类型、摘要、内容类型、排序、状态、生效时间、创建时间。
4. 新建/编辑弹窗根据 `contentType` 展示 H5 URL 或原生内容文本域。
5. 发布/下线、删除需要二次确认。

异常/边界：

- `contentType=H5` 时校验 URL。
- `contentType=NATIVE` 时校验正文非空。
- 原生内容首版不引入富文本编辑器，前端可用文本域承接简单 HTML/Markdown-like 文本。

#### 流程 2：应用配置管理

**入口**：左侧菜单 → 运营中心 → 应用配置  
**组件/API**：`AppConfigPage.tsx`、`frontend/src/api/content.ts`

正常流：

1. 按 `configGroup` 分 Tab 展示。
2. 每行展示 key、类型、值、是否公开给小程序、状态、备注。
3. 保存时批量提交当前分组。
4. 配置值允许清空，清空后小程序按空值兜底。

异常/边界：

- `configType=URL` 时校验 URL 格式。
- `configType=JSON` 时校验 JSON 合法性。
- `publicVisible=0` 的配置不得出现在小程序公共接口。

#### 流程 3：移动端入口配置

**入口**：左侧菜单 → 移动端配置 → 入口配置  
**组件/API**：`MobileEntryConfigPage.tsx`、`frontend/src/api/content.ts`

正常流：

1. 通过分段控件切换 `MY_PAGE/SETTINGS_PAGE/SECURITY_CENTER/SEARCH_RESULT_TAB`。
2. 表格支持入口名称、排序、启停、跳转类型、跳转目标、角标、是否需要登录。
3. 支持拖拽或排序输入后批量保存顺序。
4. 启停搜索结果 Tab 时，小程序搜索配置接口立即返回最新 Tab。

异常/边界：

- `jumpType=H5` 时必须配置 URL 或 `config:{configKey}` 配置引用。
- `jumpType=NATIVE_ROUTE` 时必须配置小程序路由。
- `entryKey` 创建后不建议修改，避免小程序端映射失效。

#### 流程 4：搜索热词与屏蔽词

**入口**：左侧菜单 → 搜索配置  
**组件/API**：`SearchHotWordPage.tsx`、`SearchBlockWordPage.tsx`

正常流：

1. 热词页维护 word、scene、sort、status。
2. 屏蔽词页维护 word、blockType、matchType、reasonCode、hitMessage、status。
3. 新增/编辑/启停/删除屏蔽词写操作审计。

异常/边界：

- 屏蔽词不提供小程序列表接口。
- 首版不开放正则匹配，避免误伤和性能风险。
- Service 层需校验同类型下启用词不重复。

### 8.3 安全设计

| 项             | 设计                                                                        |
| -------------- | --------------------------------------------------------------------------- |
| 后台鉴权       | `/admin/content/**` 必须登录，并通过 `@RequirePermission` 校验              |
| 小程序公共接口 | 仅返回已启用、已生效、`public_visible=1` 的内容和配置                       |
| 搜索屏蔽词     | 不暴露给小程序端；仅后端内部服务用于搜索关键词校验                          |
| H5 URL         | 后台保存时校验 URL 格式；小程序端跳转前需做白名单或协议校验                 |
| 原生内容       | 后台保存文本/简单 HTML；前端预览和小程序渲染需做 XSS 防护                   |
| 审计           | 发布/下线内容、修改配置、修改入口、修改屏蔽词必须落 `content_operation_log` |
| 敏感配置       | `public_visible=0` 的配置禁止通过 `/miniapp/content/config` 返回            |

## 12. 风险与回滚

| 风险                   | 影响                         | 缓解                                                     |
| ---------------------- | ---------------------------- | -------------------------------------------------------- |
| 后续需要富文本编辑器   | 前端需要引入编辑器和内容清洗 | `content_body` 已用 TEXT，后续升级不改表                 |
| 配置值类型复杂         | KV 表容易变成无约束大杂烩    | 增加 `config_type/public_visible/status`，Service 做校验 |
| 小程序入口 key 变化    | 前端映射失效                 | `entry_key` 创建后不建议修改，只改名称/跳转/状态         |
| 搜索屏蔽词误伤         | 用户搜索体验下降             | 首版不开放正则；支持启停和原因记录；操作写审计           |
| 公共接口误暴露内部配置 | 泄露规则参数或敏感配置       | 小程序只返回 `public_visible=1` 且启用配置               |

回滚方案：

1. 回退代码与前端路由。
2. 删除 `sys_menu` 中 800 段菜单权限和角色绑定。
3. 如尚未产生线上数据，可 DROP 本模块 6 张表。
4. 如已有线上数据，只下线菜单和接口，不物理删除数据。

## 13. 实施顺序

1. **数据库**：新增 `schema-content.sql`，包含 6 张表、配置种子、入口种子、菜单权限。
2. **后端 common 层**：Entity → Enum → Mapper → DAO/DAOImpl。
3. **后端 admin 层**：DTO/VO → Service/ServiceImpl → Controller → 审计写入。
4. **后端 miniapp 层**：公共查询 VO → Service/ServiceImpl → Controller → `WebConfig` 放行。
5. **前端 API**：新增 `frontend/src/api/content.ts`。
6. **前端页面**：内容文章 → 应用配置 → 移动端入口 → 搜索热词 → 搜索屏蔽词 → 操作日志。
7. **联调验证**：执行 L1 cURL、后端单测、前端 build，具备环境时补 L4 Playwright。
